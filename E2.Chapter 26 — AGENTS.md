## 2. Beginner Explanation

`AGENTS.md` ကို အလွယ်ဆုံးပြောရရင်—

```text
AI coding agents အတွက် README.md
```

ပါ။

Human developer တစ်ယောက် project ထဲဝင်လာရင် `README.md` ကိုဖတ်ပြီး—

```text
ဒီ project ကဘာလဲ
ဘယ်လို run မလဲ
ဘယ် folder တွေရှိလဲ
ဘာ rules တွေလဲ
```

သိအောင်လုပ်တယ်။

AI coding agent တစ်ယောက် project ထဲဝင်လာရင် `AGENTS.md` ကိုဖတ်ပြီး—

```text
ဒီ repo layout ဘယ်လိုလဲ
ဘယ် commands သုံးရမလဲ
ဘယ် files မထိရဘူးလဲ
ဘယ် coding conventions လိုက်ရမလဲ
Done ဆိုတာဘာလဲ
Review / verification ဘယ်လိုလုပ်ရမလဲ
```

သိအောင်လုပ်မယ်။

OpenAI Codex docs က `AGENTS.md` ကို Codex က အလုပ်မလုပ်ခင်ဖတ်နိုင်တဲ့ project guidance file အဖြစ်ဖော်ပြထားပြီး global guidance နဲ့ project-specific overrides တွေကို layer လုပ်နိုင်တာကြောင့် repository တစ်ခုဖွင့်တိုင်း consistent expectations နဲ့စနိုင်ကြောင်းရှင်းပြထားပါတယ်။ ([OpenAI Developers][1])


# 4. AGENTS.md ဘာကြောင့်လိုလဲ

Prompt တစ်ခါချင်းစီထဲမှာ rules တွေပြန်ရေးတာက အလုပ်ဖြစ်ပေမယ့် project ကြီးလာရင် မကောင်းပါဘူး။

Prompt တိုင်းထဲမှာ ဒီလိုထပ်ရေးရမယ်ဆိုပါစို့။

```text
.env မဖတ်ပါနဲ့။
package install မလုပ်ခင် approval တောင်းပါ။
git status စစ်ပါ။
Done ဆိုတာ verification ပြီးမှဖြစ်တယ်။
Frontend public UI text ကိုမြန်မာလိုရေးပါ။
```

ဒီ rules တွေကို `AGENTS.md` ထဲထားလိုက်ရင် Agent sessions တွေမှာ consistent ဖြစ်မယ်။

AGENTS.md အသုံးဝင်တဲ့အကြောင်း—

```text
1. Repeated instructions မပြန်ရေးရ
2. Agent sessions အများအပြားမှာ behavior consistent ဖြစ်
3. New repo / nested folder အလိုက် local rules ထည့်နိုင်
4. Review / Done / safety expectations တူညီ
5. Claude, Codex, future tools အတွက် shared instruction source ဖြစ်နိုင်
6. Team workflow ကို AI agents နဲ့ align လုပ်နိုင်
```

OpenAI customization docs က Codex customization layers ထဲမှာ `AGENTS.md` ကို persistent project guidance အဖြစ်ဖော်ပြပြီး memories, skills, MCP, subagents တို့နဲ့ complementary ဖြစ်တယ်လို့ရှင်းပြထားပါတယ်။ `AGENTS.md` သည် durable repo guidance အဖြစ် repo နဲ့အတူလိုက်သွားပြီး agent အလုပ်မစခင် သက်ရောက်နိုင်တယ်လို့လည်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][3])

# 6. Codex က AGENTS.md ကို ဘယ်လိုရှာဖတ်လဲ

Codex docs အရ Codex က instruction chain တစ်ခုတည်ဆောက်တယ်။ Discovery order ကို အခြေခံအနေနဲ့ဒီလိုမှတ်ပါ။

```text
1. Global scope
2. Project scope
3. Current working directory အထိ nested directory files
4. Root မှ current directory အထိ merge
5. Current directory နီးတဲ့ file ကနောက်မှလာတဲ့အတွက် override လုပ်နိုင်
```

Official docs အရ global scope မှာ Codex home directory, default `~/.codex`, ထဲက `AGENTS.override.md` ရှိရင်အရင်ဖတ်ပြီး မရှိရင် `AGENTS.md` ဖတ်တယ်။ Project scope မှာ project root ကနေ current working directory အထိ directory တစ်ခုချင်းစီမှာ `AGENTS.override.md`, `AGENTS.md`, fallback filenames တွေကိုစစ်ပြီး root → current directory order နဲ့ concatenate လုပ်တယ်။ Current directory နီးတဲ့ instructions က combined prompt ထဲနောက်ပိုင်းမှာပါလာတာကြောင့် earlier guidance ကို override လုပ်နိုင်ပါတယ်။ ([OpenAI Developers][1])

Beginner mental model—

```text
~/.codex/AGENTS.md
= personal default rules

repo-root/AGENTS.md
= project/team rules

repo-root/projects/event-management/AGENTS.md
= that folder-specific rules

repo-root/projects/event-management/apps/web/AGENTS.md
= frontend-specific rules
```

# 9. Nested AGENTS.md

Nested `AGENTS.md` က directory-specific rules အတွက်ပါ။

ဥပမာ—

```text
projects/event-management/AGENTS.md
projects/event-management/apps/web/AGENTS.md
projects/event-management/services/api/AGENTS.md
projects/event-management/apps/checkin-pwa/AGENTS.md
```

Why?

```text
Frontend rules နဲ့ backend rules မတူ
PWA offline rules သီးခြားလို
Admin dashboard rules သီးခြားလို
Testing folder rules သီးခြားလို
```

Example—

```text
projects/event-management/apps/web/AGENTS.md
```

```md
# AGENTS.md — Web App

## Frontend Rules

- Use React + Vite patterns.
- Public UI text should be Myanmar unless the task says otherwise.
- Form inputs must have visible labels.
- Do not call secret-key external APIs from the browser.
- Do not edit backend service files from this folder.
```

Codex docs က global, repo-level, subdirectory-level `AGENTS.md` files တွေကိုအသုံးပြုနိုင်ပြီး current directory နီးတဲ့ guidance ကပို specific ဖြစ်လို့ override လုပ်နိုင်တယ်လို့ဖော်ပြထားပါတယ်။ ([OpenAI Developers][2])

# 10. AGENTS.override.md ဆိုတာဘာလဲ

`AGENTS.override.md` သည် temporary override file အဖြစ်သုံးနိုင်တယ်။

Global scope မှာ—

```text
~/.codex/AGENTS.override.md
```

Project scope မှာ—

```text
repo-root/AGENTS.override.md
```

Codex docs အရ same directory level မှာ `AGENTS.override.md` ရှိရင် Codex က `AGENTS.md` အစား override file ကိုသုံးနိုင်ပါတယ်။ Global scope မှာလည်း override file ကို temporary global override အဖြစ်သုံးပြီး remove လုပ်တာနဲ့ base guidance ပြန်သုံးနိုင်ပါတယ်။ ([OpenAI Developers][1])

Use cases—

```text
Temporary experiment
Emergency override
One-off migration mode
Temporary no-edit review mode
```

Beginner rule—

```text
AGENTS.override.md ကို သတိထားသုံးပါ။
Override file ထားပြီးမေ့သွားရင် Agent behavior မထင်မှတ်ဘဲပြောင်းနိုင်တယ်။
```

# 11. AGENTS.md Size Limit / Context Hygiene

Codex docs က AGENTS.md guidance files တွေကို combined size limit ထိရင် truncation ဖြစ်နိုင်ကြောင်းဖော်ပြထားပြီး default `project_doc_max_bytes` သည် 32 KiB ဖြစ်တယ်လို့ဖော်ပြထားပါတယ်။ Large guidance လိုအပ်ရင် limit တိုးတာ သို့မဟုတ် nested directories ထဲခွဲတာကိုစဉ်းစားနိုင်တယ်။ ([OpenAI Developers][1])

Best practice—

```text
AGENTS.md ကို တိုတို၊တိတိကျကျထားပါ။
Long workflows ကို skills / commands / resources docs ထဲထားပါ။
Directory-specific rules ကို nested AGENTS.md ထဲထားပါ။
```

OpenAI customization docs က `AGENTS.md` ကို small ထားရန်, repeated mistakes / recurring PR feedback / directory routing guidance လိုတဲ့အခါ rule ထည့်ရန်, closest directory where it applies ထဲထားရန်အကြံပြုထားပါတယ်။ ([OpenAI Developers][3])

# 15. AGENTS.md နဲ့ Permissions / Hooks မရောပါနဲ့

AGENTS.md ထဲက rule သည် instruction ဖြစ်တယ်။ Hard enforcement မဟုတ်ပါ။

AGENTS.md rule—

```md
- Do not read `.env` files.
```

Hard enforcement—

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

Hook enforcement—

```text
PreToolUse hook checks target file path and blocks .env edits.
```

AGENTS.md ကို “Agent သိထားသင့်တဲ့ working agreement” အဖြစ်သုံးပါ။ Safety-critical rule တွေကို permissions/hooks/CI/pre-commit checks နဲ့ထပ် enforce လုပ်ပါ။

OpenAI customization docs က `AGENTS.md` ကို guidance layer အဖြစ်ဖော်ပြပြီး rules တွေကို pre-commit hooks, linters, type checkers လို enforcement infrastructure နဲ့တွဲသုံးရင် recurring mistakes ကိုပိုကောင်းစွာကာကွယ်နိုင်တယ်လို့ရှင်းပြထားပါတယ်။ ([OpenAI Developers][3])

## Error 2 — AGENTS.md အရမ်းရှည်ခြင်း

Short, accurate AGENTS.md က vague long file ထက်ပိုကောင်းတယ်။ OpenAI docs က short/practical AGENTS.md ကိုအကြံပြုထားပြီး main file ကြီးလာရင် task-specific markdown files သို့မဟုတ် nested guidance သို့ခွဲရန်ပြောထားပါတယ်။ ([OpenAI Developers][2])

## Error 3 — Tool-specific rules အားလုံး AGENTS.md ထဲထည့်ခြင်း

Claude-specific `/hooks`, `/permissions`, Plan Mode notes တွေက `CLAUDE.md` ထဲပိုသင့်တယ်။ Codex-specific sandbox/config notes တွေက `.codex/config.toml` docs / Codex-specific section ထဲပိုသင့်တယ်။

AGENTS.md သည် tool-agnostic ဖြစ်သင့်တယ်။

---

## Error 4 — Product behavior ကို AGENTS.md ထဲသိမ်းခြင်း

Product requirement ကို `SPEC.md` ထဲထားပါ။

မကောင်း—

```md

```

ပိုကောင်း—

```md
SPEC.md:
Registration requires name and phone.

AGENTS.md:
Check SPEC.md before implementing product behavior changes.
```

## Error 5 — Nested AGENTS.md conflict

Root AGENTS.md က—

```text
Public UI text should be Myanmar.
```

Nested AGENTS.md က—

```text
Use English UI text.
```

ဆိုရင် conflict ဖြစ်တယ်။ More specific file က override လုပ်နိုင်ပေမယ့် human confusion ဖြစ်နိုင်တယ်။
