# Chapter 22 — Agent Review Prompts

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့ရေးနိုင်ရပါမယ်။

```text
[ ] Agent Review Prompt ဆိုတာဘာလဲ ရှင်းပြနိုင်မယ်
[ ] “Agent ကို code ရေးခိုင်းတာ” နဲ့ “Agent ကို review ခိုင်းတာ” ကွာခြားချက်သိမယ်
[ ] Review-only prompt ရေးနိုင်မယ်
[ ] Plan review, code review, git diff review, test review, security review, accessibility review တို့ကိုခွဲနိုင်မယ်
[ ] Reviewer Agent ကို code မပြင်အောင် ထိန်းနိုင်မယ်
[ ] Review findings ကို severity အလိုက်ခွဲနိုင်မယ်
[ ] Actionable finding ဆိုတာဘာလဲ နားလည်မယ်
[ ] Agent ရေးထားတဲ့ code ကို Agent တစ်ယောက်တည်းမယုံဘဲ second-pass review လုပ်နိုင်မယ်
[ ] Review prompt ထဲမှာ scope, criteria, evidence, output format ထည့်နိုင်မယ်
[ ] PASS / WARNING / FAIL decision model ကိုသုံးနိုင်မယ်
[ ] Review result ကို human decision အတွက်အသုံးချနိုင်မယ်
```

ဒီအခန်းက Chapter 21 “Good vs Bad Prompts” ရဲ့ဆက်တင်အခန်းပါ။ အရင်အခန်းမှာ Agent ကို ဘယ်လိုကောင်းကောင်းခိုင်းမလဲသင်ခဲ့တယ်။ ဒီအခန်းမှာတော့ Agent ကရေးထားတဲ့ plan/code/diff/test result တွေကို **ပြန်စစ်ခိုင်းနည်း** ကိုသင်မယ်။

Agentic Coding မှာ အရေးကြီးဆုံး rule တစ်ခုက—

```text
Agent ရေးတာကို Agent ကိုယ်တိုင် “ပြီးပြီ” လို့ပြောတာနဲ့မယုံပါနဲ့။
Review လုပ်ပါ။
```

## 2. Beginner Explanation

Agent Review Prompt ဆိုတာ—

```text
Agent ကို code မရေးခိုင်းဘဲ
ရှိပြီးသား plan/code/diff/result ကို စစ်ခိုင်းတဲ့ prompt
```

ပါ။

ဥပမာ Coding Agent က registration form feature တစ်ခုရေးပြီးပြီဆိုပါစို့။ Agent က—

```text
Done. Registration form implemented.
```

လို့ပြောတယ်။ Beginner အနေနဲ့ “ပြီးပြီ” လို့ ယုံလိုက်ရင် အန္တရာယ်ရှိနိုင်တယ်။

ဘာတွေဖြစ်နိုင်လဲ?

```text
Validation တစ်ခုလွတ်နေ
Mobile layout ပျက်နေ
package.json မလိုဘဲပြင်ထား
console.log ထဲ PII ပါနေ
.env file ကိုမတော်တဆဖတ်ထား
test မ run ထား
acceptance criteria တစ်ခုမကိုက်
```

ဒါကြောင့် Reviewer Agent ကိုဒီလိုခိုင်းမယ်။

```text
Code မပြင်ပါနဲ့။
git diff ကို review လုပ်ပါ။

စစ်ရန်:
- Acceptance criteria ကိုက်လား
- Scope ထဲပါလား
- Secret/API key ပါလား
- package.json မလိုဘဲပြင်ထားလား
- Browser manual test လိုလား
- Risk ရှိလား

PASS / WARNING / FAIL နဲ့ပြန်ပေးပါ။
```

ဒီလို review prompt က Agent ကို **Developer role** ကနေ **Reviewer role** သို့ပြောင်းပေးတယ်။

Mental model—

```text
Developer Agent = code ရေးသူ
Reviewer Agent  = code စစ်သူ
Verifier Agent  = acceptance criteria pass/fail စစ်သူ
Auditor Agent   = security/privacy/quality risk စစ်သူ
Human           = final decision maker
```

# 3. Agent Review Prompts ဘာကြောင့်လိုလဲ

Agentic Coding မှာ AI က code မြန်မြန်ရေးနိုင်တယ်။ ဒါပေမယ့် မြန်မြန်ရေးတာနဲ့ မှန်တယ်မဟုတ်ပါ။

Review prompt လိုတဲ့အကြောင်းရင်းများ—

```text
1. Agent hallucination လျော့စေတယ်
2. Scope creep ကိုဖမ်းနိုင်တယ်
3. Bugs ကိုစောစောတွေ့နိုင်တယ်
4. Security/privacy risk ကိုဖမ်းနိုင်တယ်
5. Tests မရှိတာကိုမြင်နိုင်တယ်
6. Git diff ကို human review လုပ်ရလွယ်စေတယ်
7. Commit မလုပ်ခင် safety gate တစ်ခုဖြစ်တယ်
8. Verifier/Auditor workflow အတွက်အခြေခံဖြစ်တယ်
```

Agent ရေးတဲ့ code ကို review မလုပ်ဘဲ commit/push/deploy လုပ်တာက—

```text
စာမဖတ်ဘဲ စာချုပ်လက်မှတ်ထိုးတာနဲ့တူတယ်။
```

Review prompt က—

```text
“ဒီစာချုပ်မှာ ဘာ risk တွေရှိလဲ စစ်ပေးပါ”
```

လိုအလုပ်လုပ်တယ်။

# 4. Review-only Mode

Review prompt ရဲ့အရေးကြီးဆုံးစကားလုံးက—

```text
Code မပြင်ပါနဲ့။
```

ပါ။

Reviewer Agent ကို code ပြင်ခွင့်ပေးလိုက်ရင် “စစ်သူ” နဲ့ “ပြင်သူ” ရောသွားမယ်။ Review output မသန့်တော့ဘူး။

Good review prompt မှာ အမြဲပါသင့်တာ—

```text
Mode:
Review only. Code မပြင်ပါနဲ့။
```

သို့မဟုတ်—

```text
Mode:
Read-only review. No edits.
```

မြန်မာလို—

```text
Code မပြင်ပါနဲ့။
Review findings ပဲပေးပါ။
```

Reviewer Agent ရဲ့တာဝန်—

```text
စစ်မယ်
ဖော်ပြမယ်
severity ခွဲမယ်
evidence ပေးမယ်
recommendation ပေးမယ်
```

Reviewer Agent မလုပ်ရ—

```text
code မပြင်ရ
package မ install ရ
git commit မလုပ်ရ
deployment မလုပ်ရ
secret မဖော်ပြရ
```

# 5. Review Prompt Anatomy

Agent Review Prompt ကောင်းတစ်ခုမှာ အောက်ပါအပိုင်းတွေပါသင့်တယ်။

```text
1. Role
2. Mode
3. Review Target
4. Context
5. Review Scope
6. Review Criteria
7. Evidence Requirement
8. Severity Model
9. Output Format
10. Non-actions
```

## 5.1 Role

Reviewer role ကိုရှင်းပါ။

```text
Role:
သင်က careful code reviewer ဖြစ်တယ်။
```

သို့မဟုတ်—

```text
Role:
သင်က security/privacy auditor ဖြစ်တယ်။
```

## 5.2 Mode

```text
Mode:
Review only. Code မပြင်ပါနဲ့။
```

## 5.3 Review Target

ဘာကို review လုပ်မလဲပြောပါ။

```text
Review target:
- git diff
- implementation plan
- app.js file
- package.json changes
- test results
- SPEC acceptance criteria
```

## 5.4 Context

```text
Context:
ဒီ feature က static frontend-only registration history lab ဖြစ်ပါတယ်။
Backend API မပါရပါ။
Package install မလိုပါ။
```

## 5.5 Review Scope

ဘာတွေစစ်မလဲ။

```text
Review scope:
- task scope
- acceptance criteria
- bugs
- security/privacy
- tests
- maintainability
```

## 5.6 Review Criteria

စစ်ရမယ့် checklist ကိုပေးပါ။

```text
Criteria:
- Allowed files only
- No package install
- No .env access
- Valid form works
- Empty errors show Myanmar text
```

## 5.7 Evidence Requirement

Reviewer က finding တစ်ခုချင်းစီအတွက် evidence ပေးရမယ်။

```text
Finding တစ်ခုချင်းစီမှာ evidence ပါရမယ်။
Evidence မရှိရင် “not enough evidence” လို့ရေးပါ။
```

## 5.8 Severity Model

Severity သတ်မှတ်ပါ။

```text
Severity:
- BLOCKER
- HIGH
- MEDIUM
- LOW
- INFO
```

သို့မဟုတ် beginner-friendly—

```text
Decision:
- PASS
- WARNING
- FAIL
```

## 5.9 Output Format

```text
Output:
| Severity | Finding | Evidence | Recommendation |
```

## 5.10 Non-actions

Reviewer မလုပ်ရမယ့်အရာရေးပါ။

```text
Do not:
- code မပြင်ပါနဲ့
- package install မလုပ်ပါနဲ့
- git command မ run ပါနဲ့, read-only command မဟုတ်ရင်
- secret value မဖော်ပြပါနဲ့
```

# 6. Review Severity Model

Review finding တွေကို severity ခွဲနိုင်ရမယ်။

## 6.1 BLOCKER

Feature ကို merge/commit/deploy မလုပ်သင့်တဲ့ပြဿနာ။

ဥပမာ—

```text
.env secret value commit ထဲပါနေ
registration submit မအလုပ်လုပ်
build fail ဖြစ်နေ
data loss ဖြစ်နိုင်
production deploy command ပါသွား
```

## 6.2 HIGH

အရေးကြီးတဲ့ bug/security/privacy issue, fix လုပ်သင့်။

ဥပမာ—

```text
phone/email PII ကို console.log ထဲ log လုပ်နေ
admin permission မစစ်ထား
duplicate registration မတားထား
package install မလိုဘဲ dependency ထည့်ထား
```

## 6.3 MEDIUM

Quality / maintainability / edge case issue.

ဥပမာ—

```text
spaces-only input မစစ်ထား
error message inconsistent
localStorage corrupted JSON ကို handle မလုပ်ထား
mobile layout minor issue
```

## 6.4 LOW

Minor improvement.

```text
naming ပိုရှင်းစေချင်
comment ထည့်လို့ကောင်း
button text ပိုကောင်းနိုင်
```

## 6.5 INFO

Observation, not necessarily issue.

```text
No automated tests available; manual testing required.
```

# 7. PASS / WARNING / FAIL Model

Beginner အတွက် severity အများကြီးမလိုသေးရင် PASS/WARNING/FAIL သုံးပါ။

```text
PASS
= criteria ကိုက်, critical issue မတွေ့

WARNING
= feature အလုပ်လုပ်နိုင်ပေမယ့် risk/edge case/improvement ရှိ

FAIL
= acceptance criteria မကိုက်, blocker issue ရှိ
```

Example—

```md
## Review Decision

WARNING

Reason:
Feature mostly works, but localStorage corrupted data handling is missing and no automated tests exist.
```

ဒီ decision model က human ဆုံးဖြတ်ဖို့ရိုးရှင်းတယ်။

---

# 8. Actionable Finding ဆိုတာဘာလဲ

Review finding က “မကောင်းဘူး” လို့ပဲပြောရင် အသုံးမဝင်ဘူး။ Actionable ဖြစ်ရမယ်။

မကောင်းတဲ့ finding—

```text
Code messy ဖြစ်နေတယ်။
```

ကောင်းတဲ့ finding—

```text
MEDIUM:
validateRegistrationForm function က formData.name ကို trim မလုပ်ခင် existence မစစ်ထားပါ။
If formData.name is undefined, runtime error ဖြစ်နိုင်ပါတယ်။
Recommendation: String(formData.name ?? "").trim() pattern သုံးပါ။
```

Actionable finding တစ်ခုမှာ—

```text
Severity
Problem
Evidence
Impact
Recommendation
```

ပါသင့်တယ်။

Template—

```md
| Severity | Finding | Evidence | Impact | Recommendation |
|---|---|---|---|---|
```

# 10. Plan Review Prompt

Plan Before Code အဆင့်မှာ Agent plan ကိုစစ်ဖို့သုံးတယ်။

```text
Role:
သင်က careful implementation plan reviewer ဖြစ်တယ်။

Mode:
Review only. Code မပြင်ပါနဲ့။

Review Target:
အောက်ပါ implementation plan ကို review လုပ်ပါ။

Context:
ဒီ feature က static frontend-only registration history lab ဖြစ်ပါတယ်။
Backend/API/database/package install မပါရပါ။

Review Criteria:
- Goal နားလည်ထားတာမှန်လား
- Scope ထဲရှိလား
- Non-goals ချိုးဖောက်ထားလား
- Allowed files ထက်ကျော်လား
- Package install ပါနေလား
- Verification plan လုံလောက်လား
- Risks/assumptions ပါလား
- Rollback plan ပါလား

Output:
| Severity | Finding | Evidence | Recommendation |

Final Decision:
Approve / Revise / Reject

Plan:
[paste plan]
```

Plan review ရဲ့အဓိကရည်ရွယ်ချက်က code မရေးခင် scope/risk ဖမ်းဖို့ပါ။

# 11. Prompt Review Prompt

ကိုယ်ရေးထားတဲ့ prompt ကို Agent ကိုပြန်စစ်ခိုင်းဖို့သုံးတယ်။

```text
Role:
သင်က Agentic Coding prompt coach ဖြစ်တယ်။

Mode:
Review only. Code မရေးပါနဲ့။

Task:
အောက်ပါ prompt ကို review လုပ်ပါ။

Scorecard:
1. Context clear
2. Goal specific
3. Scope defined
4. Non-goals included
5. Allowed files included
6. Do-not-touch list included
7. Acceptance criteria checkable
8. Verification included
9. Output format defined
10. Approval gates included

Output:
- Score table
- Missing items
- Risks
- Improved prompt

Prompt:
[paste prompt]
```

Prompt review က Agent ကိုမခိုင်းခင် prompt ကိုကောင်းအောင်ပြင်ဖို့အသုံးဝင်တယ်။

# 12. Code Review Prompt

Existing code file ကိုစစ်ဖို့သုံးတယ်။

```text
Role:
သင်က careful code reviewer ဖြစ်တယ်။

Mode:
Review only. Code မပြင်ပါနဲ့။

Context:
ဒီ app က static HTML/CSS/JS registration lab ဖြစ်ပါတယ်။

Review Target:
playground/chapter-11-coding-agent/app.js

Review Criteria:
- Logic bugs
- Edge cases
- Input validation
- DOM selector mismatch
- Error handling
- localStorage safety, if used
- PII/secret logging
- Maintainability
- Beginner readability

Output:
| Severity | Finding | Evidence | Recommendation |

Do not:
- Code မပြင်ပါနဲ့
- New code မရေးပါနဲ့
- Package install မလုပ်ပါနဲ့
```

Code review prompt မှာ “review target” ကိုရှင်းရေးပါ။

# 13. Git Diff Review Prompt

Agent ပြင်ပြီးနောက် commit မလုပ်ခင်သုံးရမယ့် အရေးကြီးဆုံး review prompt ပါ။

```text
Role:
သင်က cautious git diff reviewer ဖြစ်တယ်။

Mode:
Review only. Code မပြင်ပါနဲ့။

Review Target:
Current git diff.

Context:
Task scope:
[scope]

Allowed files:
[allowed files]

Do not touch:
[blocked files]

Review Criteria:
- Changed files scope ထဲပါလား
- Secret/API key ပါလား
- .env/node_modules/dist/build ပါသွားလား
- package.json/package-lock.json မလိုဘဲပြောင်းထားလား
- Existing behavior မလိုဘဲပြောင်းသွားလား
- Acceptance criteria ကိုက်လား
- Test/verification evidence ရှိလား
- Commit-ready ဖြစ်လား

Output:
## Summary
## Changed Files
## Findings
| Severity | Finding | Evidence | Recommendation |
## Decision
PASS / WARNING / FAIL
## Suggested Commit Messages

Do not modify files.
```

Git diff review သည် Agentic Coding အတွက် daily habit ဖြစ်ရမယ်။

# 14. Acceptance Criteria Review Prompt

Feature က acceptance criteria ကိုက်/မကိုက်စစ်ဖို့သုံးတယ်။

```text
Role:
သင်က verifier ဖြစ်တယ်။

Mode:
Verification only. Code မပြင်ပါနဲ့။

Feature:
Registration History

Acceptance Criteria:
- Empty name shows “နာမည်ဖြည့်ပါ။”
- Empty phone shows “ဖုန်းနံပါတ်ဖြည့်ပါ။”
- Valid submit generates ticket code
- Attendee appears in history list
- History persists after refresh
- Clear history clears list and localStorage

Evidence:
- Changed files: [list]
- Manual tests: [paste]
- Build/test output: [paste if available]

Output:
| Criteria | Status | Evidence | Notes |
|---|---|---|---|

Final Decision:
PASS / WARNING / FAIL

If FAIL:
- Reproduction steps
- Likely cause
```

# 22. Second Agent Review

Agentic Coding မှာ implementation agent ကိုယ်တိုင် review လုပ်တာထက် **second agent / separate review role** ကပိုကောင်းတယ်။

Why?

```text
Implementation agent က ကိုယ်ရေးထားတဲ့ code ကို over-trust လုပ်နိုင်
Same context bias ရှိနိုင်
Missing criteria ကိုမမြင်နိုင်
Separate reviewer က different perspective ရနိုင်
```

Workflow—

```text
Coding Agent implements
  ↓
Reviewer Agent reviews diff
  ↓
Verifier Agent checks criteria
  ↓
Auditor Agent checks risk
  ↓
Human decides
```

Prompt—

```text
ဒီ code ကို implementation agent မဟုတ်တဲ့ independent reviewer အနေနဲ့စစ်ပါ။
Code မပြင်ပါနဲ့။
```

# 24. Review Decision Rules

Review result ကို ဘယ်လိုဆုံးဖြတ်မလဲ?

```text
PASS:
No blocker/high issues.
Acceptance criteria covered.
Verification evidence exists.

WARNING:
Feature may be usable, but minor/medium risks or missing tests exist.
Human can decide to commit with known risk.

FAIL:
Blocker/high issue exists.
Acceptance criteria not met.
Secret/security issue exists.
Build/test failing.
Scope violation exists.
```

Examples—

```text
PASS:
All criteria pass, only allowed files changed, manual tests documented.

WARNING:
Feature works, but no automated tests and localStorage privacy note missing.

FAIL:
Valid submit does not create attendee record.
```

# 29. Common Errors

## Error 1 — Review prompt ထဲ “Code မပြင်ပါနဲ့” မပါခြင်း

Reviewer Agent က finding တွေ့တာနဲ့ code ပြင်သွားနိုင်တယ်။

ပြင်ရန်—

```text
Mode:
Review only. Code မပြင်ပါနဲ့။
```

---

## Error 2 — Review Criteria မရှင်းခြင်း

မကောင်း—

```text
Review လုပ်ပါ။
```

ကောင်း—

```text
Scope, bugs, secrets, tests, accessibility, package changes စစ်ပါ။
```

---

## Error 3 — Evidence မတောင်းခြင်း

Agent က vague findings ပေးနိုင်တယ်။

ပြင်ရန်—

```text
Finding တစ်ခုချင်းစီ evidence ပါရမယ်။
Evidence မရှိရင် “not enough evidence” လို့ရေးပါ။
```

---

## Error 4 — Severity မခွဲခြင်း

Finding အားလုံးတူတူဖြစ်သွားရင် human decision ခက်တယ်။

ပြင်ရန်—

```text
Severity: BLOCKER / HIGH / MEDIUM / LOW / INFO
```

---

## Error 5 — Reviewer ကို implementation agent နဲ့တူတူထားခြင်း

Same session / same agent က bias ရှိနိုင်တယ်။

ပြင်ရန်—

```text
Independent reviewer role သုံးပါ။
သို့မဟုတ် fresh session / read-only prompt သုံးပါ။
```

---

## Error 6 — Review result ကို final truth လို့ယူဆခြင်း

Reviewer Agent လည်းမှားနိုင်တယ်။

Rule—

```text
Review output = decision support
Human final decision = required
```

---

## Error 7 — Security review ကို skip လုပ်ခြင်း

Feature အလုပ်လုပ်တာနဲ့ security/privacy OK မဟုတ်ပါ။

ပြင်ရန်—

```text

```

# 33. Chapter Summary

ဒီအခန်းမှာ **Agent Review Prompts** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Coding Agent ရေးတာကို Reviewer Agent နဲ့ပြန်စစ်ပါ။
Verifier က acceptance criteria စစ်ပါ။
Auditor က security/privacy/quality risk စစ်ပါ။
Human က final decision ယူပါ။
```

Review prompt ကောင်းတစ်ခုမှာပါသင့်တာ—

```text
Role
Mode: Review only
Review target
Context
Review criteria
Evidence requirement
Severity model
Output format
Non-actions
```

Review types—

```text
Plan Review
Prompt Review
Code Review
Git Diff Review
Acceptance Criteria Review
Test Review
Security / Privacy Review
Accessibility Review
Dependency Review
Documentation Review
Deployment Review
Final Commit Review
```

အရေးကြီးဆုံး rule—

```text
Reviewer Agent ကို code မပြင်ခိုင်းပါနဲ့။
Review findings ကို evidence + severity + recommendation နဲ့တောင်းပါ။
```

Agentic Coding အတွက် takeaway—

```text
Agent ရေးတဲ့ code ကို review မလုပ်ဘဲ commit မလုပ်ပါနဲ့။
Review prompt က Agentic Coding workflow ရဲ့ quality gate ဖြစ်ပါတယ်။
```
