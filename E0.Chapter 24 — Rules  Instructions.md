## 2. Beginner Explanation

Agentic Coding မှာ prompt တစ်ခါရေးတာနဲ့မလုံလောက်ပါဘူး။ Project တစ်ခုကြီးလာတဲ့အခါ Agent ကို session တိုင်း ဒီလိုပြန်ပြောနေရတာ မဖြစ်သင့်ဘူး။

```text
ဒီ project က Event Management Platform ပါ။
PWA-first သွားမယ်။
MVP မှာ email/SMS notification မပါဘူး။
.env မဖတ်ရဘူး။
package install မလုပ်ခင် approval တောင်းရမယ်။
Done ဆိုတာ build/test/verification ပြီးမှဖြစ်တယ်။
```

အဲဒီလို ထပ်ခါတလဲလဲပြောရမယ့်အရာတွေကို project file ထဲမှာရေးထားတာကို **Rules / Instructions** လို့ခေါ်မယ်။

ဥပမာ—

```text
CLAUDE.md
AGENTS.md
COURSE_DECISIONS.md
SPEC.md
.claude/rules/*.md
```

ဒီ files တွေက Agent အတွက် “project handbook” လိုအလုပ်လုပ်တယ်။

Human အတွက် README ရှိသလို Agent အတွက်လည်း instructions file လိုတယ်။ AGENTS.md open format site က AGENTS.md ကို “README for agents” လို dedicated, predictable place အဖြစ်ဖော်ပြထားပြီး build steps, tests, conventions စတဲ့ agent-focused guidance တွေထားဖို့ရည်ရွယ်ထားပါတယ်။ Codex official docs မှာလည်း Codex က task မလုပ်ခင် AGENTS.md files တွေကိုဖတ်ပြီး global guidance နဲ့ project-specific overrides တွေကို instruction chain အဖြစ် layer လုပ်ကြောင်းဖော်ပြထားပါတယ်။ ([Agents][1])

အလွယ်ဆုံး mental model—

```text
Prompt
= ဒီ task တစ်ခုအတွက် အမိန့်စာ

CLAUDE.md / AGENTS.md
= project တစ်ခုလုံးအတွက် အမြဲလိုက်နာရမယ့် working agreement

SPEC.md
= product behavior ရဲ့ single source of truth

COURSE_DECISIONS.md
= course/project decisions history

permissions / hooks
= rules တွေကိုတကယ် enforce လုပ်တဲ့ safety gate
```

# 3. Rules / Instructions ဆိုတာဘာလဲ

Rules / Instructions ဆိုတာ Agent ကို project ထဲမှာ အလုပ်လုပ်တဲ့အခါ လိုက်နာစေချင်တဲ့ စည်းကမ်းများပါ။

ဥပမာ—

```text
- Before editing, run git status.
- Do not read or edit .env files.
- Use Myanmar UI text for public pages.
- Do not install packages without approval.
- Use PWA-first for mobile check-in MVP.
- A feature is Done only after verification.
```

Rules တွေက အဓိကအားဖြင့် Agent ကို—

```text
ဘာလုပ်ရမလဲ
ဘာမလုပ်ရဘူးလဲ
ဘယ် workflow လိုက်ရမလဲ
ဘယ် command run ရမလဲ
ဘယ် files မထိရဘူးလဲ
ဘယ်လို Done လို့သတ်မှတ်မလဲ
```

ဆိုတာကိုပြောတယ်။

Claude Code docs မှာ CLAUDE.md files တွေကို project, personal workflow, organization-wide context အတွက် persistent instructions အဖြစ်ဖော်ပြထားပြီး Claude က session စချိန်ဖတ်တယ်လို့ရှင်းပြထားပါတယ်။ Claude docs က CLAUDE.md ကို “ထပ်ခါတလဲလဲပြန်ရှင်းမယ့်အရာ” တွေ—build commands, conventions, project layout, always-do rules—ရေးရန်အကြံပြုထားပါတယ်။ ([Claude][2])

# 4. Prompt vs Rule vs Memory

ဒီသုံးခုကိုခွဲသိထားရမယ်။

## 4.1 Prompt

Prompt က task တစ်ခုအတွက်ပေးတဲ့ instruction ပါ။

```text
ဒီ file ကိုရှင်းပြပါ။
Registration History feature အတွက် plan ပေးပါ။
git diff ကို review လုပ်ပါ။
```

Prompt က short-term task instruction ဖြစ်တယ်။

## 4.2 Rule / Instruction

Rule က project/session အများအပြားမှာထပ်ခါတလဲလဲလိုက်နာရမယ့် instruction ပါ။

```text
Package install မလုပ်ခင် approval တောင်းပါ။
.env မဖတ်ပါနဲ့။
Feature ကြီးတိုင်း plan first လုပ်ပါ။
```

Rule ကို CLAUDE.md / AGENTS.md / .claude/rules ထဲထည့်နိုင်တယ်။

## 4.3 Memory

Memory က Agent က project/session တစ်လျှောက်သိထားသင့်တဲ့ persistent context ပါ။

Claude Code docs မှာ Claude Code memory system နှစ်မျိုးကိုဖော်ပြထားပါတယ်။ `CLAUDE.md` files တွေက user/team ကရေးတဲ့ persistent instructions ဖြစ်ပြီး, auto memory က Claude က corrections/preferences တွေကနေအလိုအလျောက်မှတ်ထားတဲ့ notes ဖြစ်ပါတယ်။ ဒါပေမယ့် ဒီ memory နှစ်မျိုးလုံးက context အဖြစ်သာ loaded ဖြစ်ပြီး enforced configuration မဟုတ်ကြောင်း docs ကသတိပေးထားပါတယ်။ ([Claude][2])

Mental model—

```text
Prompt = အခု task အတွက်ပြောတာ
Rule = project တစ်ခုလုံးအတွက်ပြောတာ
Memory = session အလွန် project context ကိုသိမ်းထားတာ
Permission/Hook = တကယ် block/enforce လုပ်တာ
```

# 5. Soft Rules vs Hard Enforcement

Rules / Instructions တွေက Agent ကိုလမ်းညွှန်တယ်။ ဒါပေမယ့် hard safety guarantee မဟုတ်ပါဘူး။

ဥပမာ—

```md
# CLAUDE.md

- Do not read .env files.
```

ဒီ rule က Agent ကို “မဖတ်သင့်ဘူး” လို့ပြောတယ်။ ဒါပေမယ့် tool-level block မဟုတ်သေးဘူး။

Hard enforcement လိုချင်ရင်—

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

သို့မဟုတ် PreToolUse hook လိုတယ်။

Claude Code permissions docs မှာ prompt သို့မဟုတ် CLAUDE.md ထဲက instructions တွေက Claude ဘာလုပ်ဖို့ကြိုးစားမလဲကို shape လုပ်ပေးနိုင်ပေမယ့် Claude Code က ဘာခွင့်ပြုမလဲကိုမပြောင်းဘူးလို့ရှင်းပြထားပါတယ်။ Tool access ကို grant/revoke လုပ်ဖို့ `/permissions`, permission rules, permission modes, PreToolUse hooks တွေကိုသုံးရမယ်လို့ docs ကဖော်ပြထားပါတယ်။ ([Claude][3])

ဒီ course ရဲ့ rule—

```text
Instruction = guide
Permission = gate
Hook = deterministic guard
Git diff = review
Human = final decision
```

# 6. ဘယ် Rules တွေကို ဘယ် File ထဲထားမလဲ

Project memory files တွေကို role အလိုက်ခွဲထားရမယ်။

| File                    | Audience               | Purpose                                  |
| ----------------------- | ---------------------- | ---------------------------------------- |
| `README.md`             | Human                  | Project overview, setup, usage           |
| `COURSE_DECISIONS.md`   | Human + Agent          | Long-term decisions and tradeoffs        |
| `SPEC.md`               | Human + Agent          | Product behavior and acceptance criteria |
| `CLAUDE.md`             | Claude Code            | Claude-specific project instructions     |
| `AGENTS.md`             | Codex / generic agents | Tool-agnostic agent instructions         |
| `.claude/settings.json` | Claude Code runtime    | Permissions, hooks, settings             |
| `.claude/rules/*.md`    | Claude Code            | Modular/path-specific instructions       |
| `.env.example`          | Human + Agent          | Environment variable names, no secrets   |

Claude Code docs က project-level CLAUDE.md ကို `./CLAUDE.md` သို့မဟုတ် `./.claude/CLAUDE.md` မှာထားနိုင်ပြီး build/test commands, coding standards, architecture decisions, naming conventions, common workflows စတဲ့ project-level standards တွေထည့်ဖို့အကြံပြုထားပါတယ်။ Claude Code က `AGENTS.md` ကိုတိုက်ရိုက်မဖတ်ကြောင်း, repo မှာ AGENTS.md ရှိပြီး Claude Code လည်းသုံးချင်ရင် `CLAUDE.md` ထဲမှာ `@AGENTS.md` import လုပ်နိုင်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([Claude][2])

# 7. CLAUDE.md ဆိုတာဘာလဲ

`CLAUDE.md` က Claude Code အတွက် project instruction file ပါ။

ထည့်သင့်တဲ့အရာ—

```text
Project purpose
Folder structure
Safety rules
Workflow rules
Build/test commands
Coding conventions
Definition of Done
Do-not-touch paths
Common mistakes Claude should avoid
```

မထည့်သင့်တဲ့အရာ—

```text
Secret values
Long tutorials
Every chapter content
Huge logs
One-time task details
Temporary debugging notes
Large API docs
```

Claude docs က CLAUDE.md files တွေကို full context အနေနဲ့ loaded လုပ်နိုင်သော်လည်း shorter files က adherence ပိုကောင်းစေတယ်လို့ဖော်ပြထားပြီး target under 200 lines per CLAUDE.md file အဖြစ်အကြံပြုထားပါတယ်။ Claude မလိုက်နာဘူးဆိုရင် `/memory` နဲ့ loaded ဖြစ်/မဖြစ်စစ်ရန်, vague instructions ကိုပိုတိကျအောင်ပြင်ရန်, conflicting instructions တွေကိုရှာရန် docs ကအကြံပြုထားပါတယ်။ ([Claude][2])

# 8. AGENTS.md ဆိုတာဘာလဲ

`AGENTS.md` က Codex / generic coding agents အတွက် instructions file ပါ။

ထည့်သင့်တဲ့အရာ—

```text
Repository layout
Setup commands
Run/test commands
Code style
Safety rules
Workflow rules
PR/review expectations
Done criteria
```

Codex official docs မှာ Codex က AGENTS.md files တွေကို task မလုပ်ခင်ဖတ်ပြီး global scope, project scope, directory path အလိုက် instruction chain တည်ဆောက်ကြောင်းဖော်ပြထားပါတယ်။ Files closer to the current working directory appear later in the combined prompt and can override earlier guidance. Codex docs က root AGENTS.md ထဲ repository expectations, nested override files ထဲ specialized rules ထည့်နိုင်ကြောင်းလည်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][4])

AGENTS.md open format site က AGENTS.md ကို README နဲ့ခွဲထားတာက human README ကိုတိုတိုထားနိုင်ပြီး agents အတွက် build steps, tests, conventions စတဲ့ detailed context ကို predictable place တစ်ခုမှာထားနိုင်ဖို့ဖြစ်ကြောင်းရှင်းပြထားပါတယ်။ ([Agents][1])

# 9. CLAUDE.md + AGENTS.md ကိုအတူသုံးနည်း

ဒီ course မှာ tool-agnostic ဖြစ်ဖို့ `AGENTS.md` ကို base instructions အဖြစ်ထားပြီး `CLAUDE.md` ထဲကနေ import လုပ်နိုင်တယ်။

## Option A — Duplicate core rules

```text
CLAUDE.md
AGENTS.md
```

နှစ်ခုလုံးထဲမှာ common safety/workflow rules ထည့်ထားမယ်။ Maintenance ပိုများနိုင်တယ်။

## Option B — CLAUDE.md imports AGENTS.md

`CLAUDE.md`—

```md
@AGENTS.md

## Claude Code Specific Rules

- Use Plan Mode for non-trivial changes.
- Use `/permissions` to verify permission rules when safety behavior is unclear.
- Use `/hooks` to verify project hooks after settings changes.
```

Claude docs က AGENTS.md ရှိတဲ့ repo တွေမှာ Claude Code အတွက် CLAUDE.md က `@AGENTS.md` import လုပ်နိုင်ပြီး Claude-specific content ကိုအောက်မှာဆက်ထည့်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][2])

ဒီ course recommendation—

```text
AGENTS.md = shared agent rules
CLAUDE.md = @AGENTS.md + Claude-specific notes
```

# 10. Rule Scope / Hierarchy

Rules တွေမှာ scope ရှိတယ်။

## 10.1 Global / User-level Rules

သင့် computer မှာ project အားလုံးအတွက်လိုက်နာစေချင်တဲ့ personal preferences.

```text
~/.claude/CLAUDE.md
~/.codex/AGENTS.md
```

ဥပမာ—

```text
Always explain commands before running.
Prefer plan-first workflow.
Ask before package install.
```

## 10.2 Project-level Rules

Repo တစ်ခုလုံးအတွက် team/shared rules.

```text
./CLAUDE.md
./AGENTS.md
./.claude/settings.json
```

ဥပမာ—

```text
This project uses PWA-first mobile strategy.
MVP does not include email/SMS notification.
Do not edit .env files.
```

## 10.3 Path-specific Rules

Folder/file type အလိုက် rules.

```text
.claude/rules/frontend.md
.claude/rules/backend.md
.claude/rules/security.md
```

Claude Code docs က larger projects အတွက် `.claude/rules/` directory ထဲမှာ modular instructions ထားနိုင်ပြီး path-specific rules က matching files နဲ့အလုပ်လုပ်တဲ့အချိန်မှာသာ context ထဲ load လုပ်နိုင်တာကြောင့် context noise လျော့စေနိုင်တယ်လို့ရှင်းပြထားပါတယ်။ ([Claude][2])

## 10.4 Current Prompt

လက်ရှိ user prompt က task-specific instruction ဖြစ်တယ်။ AGENTS.md site က explicit user chat prompt က AGENTS.md instructions ထက် override ဖြစ်နိုင်တယ်လို့ဖော်ပြထားပါတယ်။ ([Agents][1])

# 11. Rule အမျိုးအစားများ

Project instructions ထဲမှာ rules တွေကို category အလိုက်ခွဲရေးပါ။

## 11.1 Safety Rules

```text
- Do not read or edit .env files.
- Do not expose secret values.
- Do not install packages without approval.
- Do not run destructive Git commands.
- Do not deploy without explicit approval.
```

## 11.2 Workflow Rules

```text
- For non-trivial tasks, plan before editing.
- Before editing, check git status.
- After editing, provide changed files summary.
- Before Done, provide verification steps.
- Do not commit without approval.
```

## 11.3 Code Style Rules

```text
- Prefer clear function names.
- Use Myanmar UI text for public user-facing pages.
- Keep validation messages consistent.
- Prefer small helper functions over duplicated logic.
```

## 11.4 Architecture Rules

```text
- Frontend must not call secret-key external APIs directly.
- Backend reads secrets from environment variables.
- PWA-first for mobile check-in MVP.
- Email/SMS notification is Later Scope.
```

## 11.5 Testing / Verification Rules

```text
- Run npm run build when relevant.
- Provide manual browser test steps for static labs.
- Verifier must use PASS/FAIL table.
- Auditor must not edit code.
```

## 11.6 Documentation Rules

```text
- Update SPEC.md when behavior changes.
- Update README.md only for user-facing setup changes.
- Keep CLAUDE.md concise.
- Add reusable prompt templates to resources/prompts/.
```

# 12. Good Rule vs Bad Rule

## Bad Rule

```text
Write good code.
```

ဘာကြောင့်မကောင်းလဲ—

```text
vague
မစစ်နိုင်
Agent အတွက် action မရှင်း
```

## Good Rule

```text
Before declaring a frontend form task done, provide manual test steps for:
- empty required fields
- valid submit
- browser console error check
```

ဘာကြောင့်ကောင်းလဲ—

```text
specific
actionable
checkable
workflow-ready
```

## Bad Rule

```text
Be safe.
```

## Good Rule

```text
Do not read or edit `.env` files. Use `.env.example` only for variable names. Never print secret values.
```

## Bad Rule

```text
Use nice UI.
```

## Good Rule

```text
For public pages, use Myanmar UI text, visible form labels, and mobile-first layout. Do not use placeholder text as the only label.
```

Rule ကောင်းရဲ့ characteristics—

```text

short
actionable
testable
```

# 13. Rule Conflict

Rule conflicts ဖြစ်နိုင်တယ်။

ဥပမာ—

```text
Rule A:
Always run npm test after editing JavaScript.

Rule B:
Do not run npm commands without approval.
```

ဒီ rules နှစ်ခု conflict ဖြစ်တယ်။

ပြင်နည်း—

```text
After editing JavaScript, propose running npm test and ask for approval before running it.
```

နောက်ဥပမာ—

```text
Rule A:
Use English UI text.
Rule B:
Use Myanmar UI text for public pages.
```

ပြင်—

```text
Use Myanmar UI text for public user-facing pages. Use English for developer-only logs and docs unless otherwise specified.
```

Claude docs က conflicting instructions ရှိရင် Claude က arbitrary တစ်ခုကိုရွေးနိုင်ကြောင်းသတိပေးထားပြီး CLAUDE.md, nested CLAUDE.md, .claude/rules files တွေကို periodic review လုပ်ရန်အကြံပြုထားပါတယ်။ ([Claude][2])

# 14. Rule Writing Principles

Rules ရေးတဲ့အခါ ဒီ principles သုံးပါ။

```text
1. Write rules as commands, not wishes.
2. Keep rules short.
3. Make rules checkable.
4. Avoid contradictions.
5. Separate project rules from personal preferences.
6. Put secrets/safety in permissions/hooks too.
7. Put one-time instructions in prompt, not CLAUDE.md.
8. Put long procedures in skills/commands, not top-level memory.
9. Use path-specific rules for specialized folders.
10. Review rules when Agent repeats the same mistake.
```

Claude docs က CLAUDE.md ထဲကို fact/rules/workflows များကိုသာထားရန်, multi-step procedure သို့မဟုတ် codebase တစ်ပိုင်းတည်းမှာသာသက်ဆိုင်တဲ့ instruction ဖြစ်ရင် skill သို့မဟုတ် path-scoped rule သို့ရွှေ့ရန်အကြံပြုထားပါတယ်။ ([Claude][2])

```text
[ ] One-time task instructions

[ ] Very long tutorials

[ ] Conflicting preferences
```

# 20. Rules vs SPEC

Rules နဲ့ SPEC မရောပါနဲ့။

## Rules

```text
Agent ဘယ်လိုအလုပ်လုပ်ရမလဲ
Project convention ဘာလဲ
Safety boundary ဘာလဲ
Workflow ဘာလိုက်ရမလဲ
```

ဥပမာ—

```text
Feature ကြီးတိုင်း plan first.
Package install မလုပ်ခင် approval.
Use Myanmar UI text for public pages.
```

## SPEC

```text
Product ဘာလုပ်ရမလဲ
User behavior ဘာလဲ
Acceptance criteria ဘာလဲ
API contract ဘာလဲ
```

ဥပမာ—

```text
User can register for an event.
Registration requires name and phone.
Successful registration returns ticket code.
```

Rule—

```text
Rules guide the agent.
SPEC defines the product.
```

# 24. Common Errors

## Error 1 — Rules file ထဲ secret ထည့်ခြင်း

မလုပ်ရ—

```text
OPENAI_API_KEY=sk-...
DATABASE_URL=postgres://...
```

Rules file တွေကို Git commit လုပ်မယ့်အတွက် secret မပါရ။

---

## Error 2 — Rule ကို vague ရေးခြင်း

မကောင်း—

```text
Write clean code.
```

ကောင်း—

```text
Prefer small functions with descriptive names. If a function grows beyond one clear responsibility, propose a refactor before editing.
```

---

## Error 3 — CLAUDE.md အရမ်းရှည်ခြင်း

အခန်း notes အားလုံးထည့်တာမလုပ်ပါနဲ့။ Claude docs က longer files consume more context and reduce adherence ဖြစ်နိုင်ကြောင်းသတိပေးထားပါတယ်။ ([Claude][2])

ပြင်ရန်—

```text
Core rules only in CLAUDE.md.
Detailed workflows in skills/commands.
Path-specific rules in .claude/rules/.
```

---

## Error 4 — Contradictory rules ထားခြင်း

မကောင်း—

```text
Always run npm test automatically.
Ask before running npm commands.
```

ပြင်—

```text
After code changes, propose running npm test and ask for approval before running it.
```

---

## Error 5 — Instruction ကို enforcement လို့ထင်ခြင်း

CLAUDE.md ထဲရေးထားတာက context/instruction ဖြစ်တယ်။ Secret blocking, destructive command blocking အတွက် permissions/hooks ထပ်လိုတယ်။

## Error 6 — Personal preference ကို project rules ထဲထည့်ခြင်း

ဥပမာ—

```text
Always use my personal alias `gst` for git status.
```

```text
= Claude Code project instructions

AGENTS.md
= Codex / generic agent instructions

= product behavior and acceptance criteria

COURSE_DECISIONS.md
= project/course decisions

= permissions/hooks/settings enforcement
```
