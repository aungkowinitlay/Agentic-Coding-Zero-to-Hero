## 2. Beginner Explanation

Plugin ကို အလွယ်ဆုံးပြောရရင်—

```text
Agentic Coding tool ထဲထည့်သုံးနိုင်တဲ့ extension package
```

ပါ။

ဥပမာ သင့်မှာ ဒီ reusable assets တွေရှိတယ်ဆိုပါစို့။

```text
diff-review skill
bug-report skill
verifier subagent
auditor subagent
dangerous-command hook
DigitalOcean deployment MCP
TypeScript LSP code intelligence
```

ဒီ assets တွေကို project တစ်ခုချင်းစီမှာ copy-paste လုပ်နေမယ့်အစား plugin တစ်ခုအဖြစ် package လုပ်နိုင်တယ်။

```text
quality-tools-plugin/
├── .claude-plugin/plugin.json
├── skills/
│   ├── diff-review/
│   └── bug-report/
├── agents/
│   ├── verifier.md
│   └── auditor.md
└── hooks/
    └── hooks.json
```

ပြီးရင် team marketplace ထဲတင်မယ်။

```text
company-tools marketplace
  ├── quality-tools-plugin
  ├── frontend-starter-plugin
  ├── backend-starter-plugin
  └── deployer-plugin
```

Team member တစ်ယောက်က Claude Code ထဲမှာ marketplace add လုပ်ပြီး plugin install လုပ်မယ်။

```text
/plugin marketplace add your-org/claude-plugins
/plugin install quality-tools@company-tools
```

ဒီလိုနဲ့ “Agentic workflow” တွေကို project တစ်ခုတည်းမဟုတ်ဘဲ team တစ်ခုလုံး share လုပ်နိုင်လာတယ်။

Claude Code docs အရ plugins တွေက Claude Code ကို skills, agents, hooks, MCP servers, LSP servers, monitors စတဲ့ components တွေနဲ့ extend လုပ်တဲ့ self-contained directory package ဖြစ်ပြီး, marketplaces တွေက အဲဒီ plugins တွေကို discover/install/update လုပ်ဖို့ catalog အဖြစ်အလုပ်လုပ်ပါတယ်။ ([Claude][1])

# 3. Plugin ဆိုတာဘာလဲ

Plugin ဆိုတာ Agentic Coding tool ထဲကို install လုပ်နိုင်တဲ့ reusable package ပါ။

Claude Code plugin တစ်ခုမှာ ပါနိုင်တာ—

```text
Skills
Commands
Subagents / Agents
Hooks
MCP servers
LSP servers
Monitors
Output styles
Themes, experimental
Settings
User configuration
```

Claude Code plugin reference က plugin components တွေထဲမှာ skills, agents, hooks, MCP servers, LSP servers, monitors ပါနိုင်ကြောင်းဖော်ပြထားပြီး plugin တစ်ခုကို Claude Code custom functionality extend လုပ်တဲ့ self-contained directory အဖြစ်သတ်မှတ်ထားပါတယ်။ ([Claude][1])

Codex plugin ဘက်မှာလည်း plugin က reusable workflow distribution unit ဖြစ်ပြီး skills, app integrations, MCP servers ပါနိုင်တယ်။ Codex docs က plugins တွေကို skills, apps, MCP servers ထည့်ပြီး reusable workflows အဖြစ် bundle လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][2])

Mental model—

```text
Skill
= workflow တစ်ခု

Subagent
= specialist worker

Hook
= automatic safety / automation checkpoint

MCP
= external tool/service access

Plugin
= အဲဒီ components တွေကို install/share/update လုပ်နိုင်အောင် package လုပ်ထားတာ

Marketplace
= Plugins တွေကိုရှာ/install/update လုပ်တဲ့ catalog
```

# 4. Marketplace ဆိုတာဘာလဲ

Marketplace ဆိုတာ plugin catalog ပါ။

Claude Code docs က marketplace ကို “someone else has created and shared” plugins တွေကို browse/install လုပ်ဖို့ catalog အဖြစ်ဖော်ပြထားပြီး process ကို two-step အနေနဲ့ရှင်းပြထားပါတယ်။

```text
1. Add marketplace
   → catalog ကို register လုပ်တယ်, plugin မ install သေး

2. Install individual plugins
   → လိုချင်တဲ့ plugin တစ်ခုချင်းစီကိုရွေး install လုပ်တယ်
```

Docs က marketplace add လုပ်တာကို app store ထည့်တာနဲ့နှိုင်းပြီး “store ထည့်တာက collection ကို browse လုပ်ခွင့်ပေးတာ, apps/plugin တစ်ခုချင်းစီကိုတော့ install ရွေးရသေးတာ” လို့ရှင်းပြထားပါတယ်။ ([Claude][3])

Marketplace types—

```text
Official marketplace
Community marketplace
Private team marketplace
Local development marketplace
Remote URL marketplace
```

# 5. Plugin vs Standalone Configuration

အရင် chapters တွေမှာ `.claude/skills/`, `.claude/agents/`, `.claude/hooks/` ထဲမှာ project-specific assets တွေတည်ဆောက်ခဲ့တယ်။ အဲဒါတွေက standalone configuration ဖြစ်တယ်။

Plugin ကတော့ share/install/version/update လုပ်လို့ရတဲ့ package ဖြစ်တယ်။

| အချက်        | Standalone `.claude/` config                           | Plugin                                          |
| ------------ | ------------------------------------------------------ | ----------------------------------------------- |
| Best for     | one project, quick experiment, personal workflow       | sharing, team distribution, versioned release   |
| Location     | `.claude/skills/`, `.claude/agents/`, `.claude/hooks/` | plugin folder with `.claude-plugin/plugin.json` |
| Invocation   | short local names like `/diff-review`                  | namespaced like `/plugin-name:skill-name`       |
| Distribution | copy/paste or repo commit                              | marketplace install/update                      |
| Versioning   | project Git history                                    | plugin version / commit SHA                     |
| Team reuse   | limited                                                | strong                                          |

Claude Code docs က standalone `.claude/` directory ကို personal workflows, project-specific customizations, quick experiments, short skill names အတွက်သုံးသင့်ပြီး plugins ကို teammates/community share, reusable across projects, versioned releases အတွက်သုံးသင့်တယ်လို့ခွဲထားပါတယ်။ ([Claude][4])

Course rule—

```text
Project တစ်ခုတည်းအတွက်
→ .claude/skills, .claude/agents, .claude/hooks

Team/project များစွာမှာသုံးမယ်
→ Plugin

Public/community share မယ်
→ Marketplace
```

# 6. Plugin Components Overview

## 6.1 Skills

Plugin ထဲမှာ `skills/` directory ထည့်နိုင်တယ်။

```text
my-plugin/
├── .claude-plugin/plugin.json
└── skills/
    └── code-review/
        └── SKILL.md
```

Claude docs က plugin skills တွေဟာ Claude က task context အလိုက် automatic invoke လုပ်နိုင်ပြီး plugin install ပြီးနောက် `/plugin-name:skill-name` namespace နဲ့ခေါ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][4])

## 6.2 Agents / Subagents

Plugin ထဲမှာ `agents/` directory ထည့်နိုင်တယ်။

```text
my-plugin/
└── agents/
    └── security-reviewer.md
```

Plugin agents တွေဟာ specialized subagents အဖြစ် Claude က invoke လုပ်နိုင်ပြီး, plugin-scoped name — ဥပမာ `my-plugin:code-reviewer` — နဲ့ @-mention typeahead ထဲပေါ်နိုင်တယ်။ Plugin agents တွေမှာ `name`, `description`, `model`, `effort`, `maxTurns`, `tools`, `disallowedTools`, `skills`, `memory`, `background`, `isolation` fields သုံးနိုင်ပေမယ့် security reasons ကြောင့် plugin-shipped agents မှာ `hooks`, `mcpServers`, `permissionMode` fields မ support လုပ်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([Claude][1])

## 6.3 Hooks

Plugin ထဲမှာ lifecycle hooks ထည့်နိုင်တယ်။

```text
my-plugin/
└── hooks/
    └── hooks.json
```

Plugin hooks တွေက user-defined hooks နဲ့တူတဲ့ lifecycle events — `PreToolUse`, `PostToolUse`, `SessionStart`, `Stop`, `SubagentStart`, `SubagentStop`, `ConfigChange`, `FileChanged` စတာတွေကို respond လုပ်နိုင်ကြောင်း plugin reference မှာဖော်ပြထားပါတယ်။ ([Claude][1])

## 6.4 MCP Servers

Plugin တစ်ခုက MCP servers ထည့်ပြီး Claude/Codex ကို external tools/services တွေနဲ့ချိတ်နိုင်တယ်။

ဥပမာ—

```text
GitHub MCP
DigitalOcean MCP
Database read-only MCP
Slack MCP
Google Drive MCP
```

Codex docs က plugins တွေထဲမှာ MCP servers ပါနိုင်ပြီး, MCP servers တွေက local project အပြင်က systems တွေမှ tools/shared information ကို Agent ဆီပေးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][2])

## 6.5 LSP Servers

Claude Code plugins တွေက LSP servers ထည့်နိုင်တယ်။ LSP plugins က Claude ကို go-to-definition, find references စတဲ့ code intelligence ပိုကောင်းအောင်ပေးနိုင်တယ်။ Claude docs က TypeScript, Python, Rust လို common languages အတွက် pre-built LSP plugins ကို official marketplace မှ install လုပ်ရန်အကြံပြုထားပြီး custom LSP plugin ကို language support မရှိတဲ့အခါသာဖန်တီးသင့်တယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][4])

## 6.6 Monitors

Plugin monitors တွေက logs/files/external status ကို background ကနေ watch လုပ်ပြီး Claude ဆီ event notification ပို့နိုင်တယ်။ Claude docs က `monitors/monitors.json` ထဲ monitor entries ထည့်ပြီး application error log လို source ကို watch လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][4])

# 7. Claude Code Plugin Basic Structure

Minimal plugin—

```text
my-first-plugin/
├── .claude-plugin/
│   └── plugin.json
└── skills/
    └── hello/
        └── SKILL.md
```

`plugin.json` example—

```json
{
  "name": "my-first-plugin",
  "description": "A greeting plugin to learn the basics",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}
```

Claude docs က plugin manifest ကို `.claude-plugin/plugin.json` ထဲထားရပြီး `name`, `description`, `version`, `author` fields တွေကို example အဖြစ်ပြထားပါတယ်။ `name` က plugin unique identifier နဲ့ skill namespace ဖြစ်ပြီး, `version` ထည့်ထားရင် users တွေက field bump လုပ်မှ update ရနိုင်တယ်; version မထည့်ဘဲ git distribution ဖြစ်ရင် commit SHA ကို version အဖြစ်ယူနိုင်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([Claude][4])

Skill invocation—

```text
/my-first-plugin:hello
```

# 8. Plugin Namespacing

Standalone skill—

```text
.claude/skills/diff-review/SKILL.md
```

Invoke—

```text
/diff-review
```

Plugin skill—

```text
quality-tools-plugin/skills/diff-review/SKILL.md
```

Invoke—

```text
/quality-tools-plugin:diff-review
```

ဘာကြောင့် namespace လိုလဲ?

```text
Different plugins ထဲမှာ skill name တူနိုင်
Plugin source သိရလွယ်
Team marketplace ထဲ conflicts လျော့
```

Claude docs က plugin name က skill namespace ဖြစ်ပြီး plugin named `my-first-plugin` ထဲ `hello/` skill က `/my-first-plugin:hello` အဖြစ်ခေါ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][4])

# 9. Plugin Installation Scopes

Claude Code plugin install လုပ်တဲ့အခါ scope ရွေးရတယ်။

```text
User scope
Project scope
Local scope
```

Claude docs က plugin install details screen မှာ installation scope ရွေးနိုင်ပြီး User scope သည် user projects အားလုံးအတွက် default, Project scope သည် repo collaborators အတွက် share ဖြစ်ပြီး `.claude/settings.json` ထဲထည့်နိုင်, Local scope သည် user တစ်ယောက်အတွက် repo တစ်ခုတည်းမှာသာ install ဖြစ်ပြီး collaborators မ share ဖြစ်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][3])

| Scope       | ဘယ်အချိန်သုံးမလဲ                | Git share?        |
| ----------- | ------------------------------- | ----------------- |
| User        | personal workflow, all projects | No                |
| Project     | team shared plugin for repo     | Yes, via settings |
| Local       | one developer’s local repo only | No                |
| Managed/org | enterprise-controlled policy    | Admin controlled  |

Course rule—

```text
Personal experiment → User/Local
Course/team standard workflow → Project
Sensitive/approved org plugin → Managed policy
```

# 10. Official Anthropic Marketplace

Claude Code official marketplace name—

```text
claude-plugins-official
```

Claude docs က official Anthropic marketplace ကို Claude Code start လုပ်တဲ့အခါ automatically available ဖြစ်တယ်လို့ဖော်ပြထားပြီး `/plugin` → Discover tab မှာ browse လုပ်နိုင်ပါတယ်။ Plugin install example—

```text
/plugin install github@claude-plugins-official
```

Marketplace outdated/missing ဖြစ်ရင်—

```text
/plugin marketplace update claude-plugins-official
/plugin marketplace add anthropics/claude-plugins-official
```

လုပ်နိုင်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([Claude][3])

Official marketplace categories docs မှာ code intelligence, external integrations, automatic security review, development workflows, output styles စတာတွေဖော်ပြထားပါတယ်။ ([Claude][3])

Beginner rule—

```text
Official marketplace ဖြစ်တယ်ဆိုတာ trust ပိုမြင့်တယ်။
ဒါပေမယ့် install မလုပ်ခင် plugin details, requested capabilities, external access, hooks/MCP/scripts ပါ/မပါ စစ်ပါ။
```

# 11. Community Marketplace

Claude community marketplace—

```text
anthropics/claude-plugins-community
```

Add—

```text
/plugin marketplace add anthropics/claude-plugins-community
```

Install—

```text
/plugin install <plugin-name>@claude-community
```

Claude docs က community marketplace မှ third-party plugins တွေဟာ Anthropic automated validation and safety screening pass လုပ်ထားပြီး catalog ထဲ plugin တစ်ခုချင်းစီကို commit SHA နဲ့ pin ထားကြောင်းဖော်ပြထားပါတယ်။ Official marketplace နဲ့မတူဘဲ community marketplace ကို manual add လုပ်ရပါတယ်။ ([Claude][3])

Beginner rule—

```text
Community marketplace = useful, but third-party.
Install before review.
Enable after trust.
```

# 12. Private Team Marketplace

Use cases—

```text

DigitalOcean deployer plugin

```

Claude docs က plugin marketplace တစ်ခုဖန်တီးရန်—


# 13. Marketplace File Structure

Marketplace repository example—

```text
my-marketplace/
├── .claude-plugin/
│   └── marketplace.json
└── plugins/
    └── quality-review-plugin/
        ├── .claude-plugin/
        │   └── plugin.json
        └── skills/
            └── quality-review/
                └── SKILL.md
```

`marketplace.json` example—

```json
{
  "name": "company-tools",
  "owner": {
    "name": "DevTools Team",
    "email": "devtools@example.com"
  },
  "plugins": [
    {
      "name": "code-formatter",
      "source": "./plugins/formatter",
      "description": "Automatic code formatting on save",
      "version": "2.1.0"
    }
  ]
}
```

Claude docs က marketplace file ကို repository root ထဲ `.claude-plugin/marketplace.json` အဖြစ်ထားပြီး marketplace name, owner information, plugins list ပါရမယ်လို့ဖော်ပြထားပါတယ်။ Plugin entry တစ်ခုမှာ အနည်းဆုံး `name` နဲ့ `source` လိုပြီး `source` က Claude Code plugin ကိုဘယ်က fetch လုပ်မလဲပြောပါတယ်။ ([Claude][5])

# 14. Plugin Source Pinning

Marketplace plugin source ကို branch, tag, commit SHA နဲ့ pin လုပ်နိုင်တယ်။

Example—

```json
{
  "name": "git-plugin",
  "source": {
    "source": "url",
    "url": "https://gitlab.com/team/plugin.git",
    "ref": "main",
    "sha": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"
  }
}
```

Claude docs က plugin source တွေမှာ `ref` နဲ့ branch/tag သတ်မှတ်နိုင်ပြီး `sha` နဲ့ exact commit pin လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ Git subdirectory source ကိုလည်း `git-subdir` နဲ့ monorepo ထဲ subdirectory plugin ကို partial clone fetch လုပ်နိုင်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([Claude][5])

Course rule—

```text
Team/security-sensitive plugin
→ pin to tag or commit SHA

Fast internal experiment
→ branch ref acceptable

Community plugin
→ prefer pinned / reviewed source
```

# 15. Plugin Caching & File Resolution

Plugin install လုပ်တဲ့အခါ Claude Code က plugin directory ကို cache location ထဲ copy လုပ်နိုင်တယ်။ Marketplace docs က installed plugin directory သည် cache ထဲ copy ဖြစ်တာကြောင့် plugin ထဲက files တွေက plugin directory အပြင်ဘက် paths — ဥပမာ `../shared-utils` — ကို reference လုပ်ရင် copied မဖြစ်နိုင်ကြောင်းသတိပေးထားပါတယ်။ Shared files လိုရင် symlinks စဉ်းစားရန် docs ကဖော်ပြထားပါတယ်။ ([Claude][5])

Plugin design rule—

```text
Plugin must be self-contained.
Do not assume files outside plugin directory exist.
Do not use ../shared-utils style paths.
```

# 17. User Configuration & Secrets

Plugin တစ်ခုက external API/token လိုနိုင်တယ်။ အဲဒီအခါ userConfig ကိုသုံးသင့်တယ်။

Example—

```json
{
  "userConfig": {
    "api_endpoint": {
      "type": "string",
      "title": "API endpoint",
      "description": "Your team's API endpoint"
    },
    "api_token": {
      "type": "string",
      "title": "API token",
      "description": "API authentication token",
      "sensitive": true
    }
  }
}
```

Claude plugin reference က `userConfig` field သည် plugin enable လုပ်ချိန်မှာ user ကို configuration values prompt လုပ်နိုင်ပြီး `sensitive: true` ဖြစ်ရင် input ကို mask လုပ်ကာ `settings.json` ထဲမသိမ်းဘဲ secure storage / system keychain ထဲသိမ်းနိုင်ကြောင်းဖော်ပြထားပါတယ်။ Sensitive values တွေကို plugin subprocesses ဆီ environment variables အဖြစ်ပေးနိုင်ပေမယ့် skill/agent content ထဲ non-sensitive values သာ substitute လုပ်နိုင်ကြောင်း docs ကရှင်းပြထားပါတယ်။ ([Claude][1])

Course rule—

```text
Never hard-code plugin secrets.
Use userConfig sensitive fields.
Do not print token values.
Do not ask user to paste secrets into chat.
```

# 18. Plugin Security

Plugins are powerful. Claude Code docs က plugins နဲ့ marketplaces တွေဟာ highly trusted components ဖြစ်ပြီး user privileges နဲ့ machine ပေါ် arbitrary code execute လုပ်နိုင်တာကြောင့် trusted sources ကနေသာ install/add လုပ်ရန်သတိပေးထားပါတယ်။ Organizations တွေက managed marketplace restrictions နဲ့ allowed marketplaces ကိုကန့်သတ်နိုင်ကြောင်းလည်း docs ကဖော်ပြထားပါတယ်။ ([Claude][3])

Security review checklist—

```text
[ ] Plugin source trusted လား
[ ] Marketplace source trusted လား
[ ] Plugin pinned to version/tag/SHA လား
[ ] plugin.json ဖတ်ပြီး components သိပြီးလား
[ ] skills only လား, hooks/MCP/scripts ပါလား
[ ] Hooks က arbitrary command run မလုပ်ဘူးလား
[ ] MCP servers က external access ဘာတွေယူမလဲ
[ ] userConfig sensitive values သုံးထားလား
[ ] install scope user/project/local မှန်လား
[ ] Plugin default enabled ဖြစ်သင့်/မသင့်စဉ်းစားထားလား
[ ] Team approval လိုလား
```

Plugin install prompt—

```text
Plugin install မလုပ်သေးပါနဲ့။
အရင် security review လုပ်ပါ။

Plugin:
[name/source]

စစ်ရန်:
- source trusted?
- components: skills/agents/hooks/MCP/LSP/monitors?
- scripts or hooks execute commands?
- external network/cloud access?
- secrets/userConfig?
- install scope recommendation?
- risks?
- approve / reject / need more info?
```

# 19. Default Enabled & Manual Enable

Plugin တချို့ကို install ပြီးတာနဲ့ enabled မဖြစ်စေချင်နိုင်တယ်။ Claude plugin reference က `defaultEnabled: false` ကို `plugin.json` ထဲသတ်မှတ်နိုင်ပြီး plugin install ပြီး user က `claude plugin enable <plugin>` သို့ `/plugin` interface ကနေ turn on လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ External service connect/cost/scope တိုးစေတဲ့ plugin တွေမှာ opt-in ဖြစ်သင့်ကြောင်း docs ကညွှန်ပါတယ်။ ([Claude][1])

```text
Skills-only safe plugin

External MCP / cloud / cost plugin
→ defaultEnabled false

Production deployer plugin
→ planning-only by default, manual enable/action approval
```

```text
11. Share to team
```

# 28. Troubleshooting Plugins

Common problems—

```text
/plugin command not recognized
Marketplace not loading
Plugin not found
Plugin install failed
Plugin skills not appearing
Files not found after install
Hooks not firing
MCP server not starting
Command namespace conflict
```

Claude docs troubleshooting section says if `/plugin` is not recognized, check `claude --version`, update Claude Code, and restart terminal/session. # 32. Common Errors

## Error 1 — Plugin source မစစ်ဘဲ install လုပ်ခြင်း

Plugin က arbitrary code run နိုင်တာကြောင့် trusted source မဟုတ်ရင် install မလုပ်သင့်ပါ။ Claude docs က plugins/marketplaces are highly trusted components and can execute arbitrary code with your user privileges လို့သတိပေးထားပါတယ်။ ([Claude][3])

## Error 2 — Marketplace add = plugin install လို့ထင်ခြင်း

Marketplace add က catalog register လုပ်တာပဲ။ Plugin တစ်ခုချင်းစီ install လုပ်ရသေးတယ်။ ([Claude][3])

## Error 3 — Plugin ကို self-contained မထားခြင်း

Installed plugin directory က cache ထဲ copy ဖြစ်နိုင်တာကြောင့် outside path references ပျက်နိုင်တယ်။ ([Claude][5])

## Error 4 — Plugin agents ထဲ unsupported fields ထည့်ခြင်း

Plugin-shipped agents တွေမှာ `hooks`, `mcpServers`, `permissionMode` မ support လုပ်ကြောင်း plugin reference မှာဖော်ပြထားပါတယ်။ ([Claude][1])

## Error 5 — Secrets hard-code လုပ်ခြင်း

Plugin ထဲ token/API key မထည့်ပါနဲ့။ `userConfig` + `sensitive: true` သုံးပါ။ ([Claude][1])

## Error 6 — Hook/MCP ပါတဲ့ plugin ကို skills-only လို့ထင်ခြင်း

Plugin review မှာ components အားလုံးစစ်ပါ။

```text
skills/
agents/
hooks/
mcpServers
lspServers
monitors
scripts
```

## Error 7 — Project scope install ကိုမစဉ်းစားခြင်း

Project scope install က collaborators အတွက် shared settings ဖြစ်နိုင်တယ်။ Plugin ကို team approval မရှိဘဲ project scope မသုံးသင့်ပါ။ ([Claude][3])

## Error 8 — Version bump မလုပ်ခြင်း

Plugin manifest `version` ထည့်ထားရင် updates သွားဖို့ version bump လိုနိုင်တယ်။ ([Claude][4])

```text
Marketplace = plugins တွေကို discover/install/update လုပ်တဲ့ catalog
```

Claude Code plugin တစ်ခုမှာပါနိုင်တာ—

```text
skills
agents
hooks
MCP servers
LSP servers
monitors
settings
userConfig
output styles
```

Marketplace workflow—

```text
Add marketplace
→ Browse plugins
→ Review plugin source/components
→ Install individual plugin
→ Choose scope
→ Test
→ Update/disable/uninstall as needed
```

အရေးကြီးဆုံး safety rule—

```text
Plugins are trusted code.
Install only from trusted sources.
Review hooks, MCP, scripts, secrets, and install scope before enabling.
```

Codex ဘက်မှာ—

```text
Skills = reusable workflow authoring format
Plugins = installable distribution unit for skills/apps/MCP servers
Plugin directory / @ invocation / marketplace sharing concept ရှိ
```

Course strategy—

```text
Single project experiment
→ standalone .claude/skills, .claude/agents, hooks

Repeated team workflow
→ plugin

Team/community distribution
→ marketplace

Sensitive external integration
→ security review + approval gate + least privilege
```

Agentic Coding အတွက် takeaway—

```text
Skills, commands, subagents, hooks တစ်ခုချင်းစီရေးနိုင်တာက local productivity ဖြစ်တယ်။
အဲဒါတွေကို plugin/marketplace အဖြစ် package လုပ်နိုင်တာက team-level Agentic Engineering ဖြစ်တယ်။
```
