## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့ရေးနိုင်ရပါမယ်။

```text
[ ] Prompt ဆိုတာ Agent ကိုပေးတဲ့ အလုပ်အမိန့်စာ ဖြစ်ကြောင်းနားလည်မယ်
[ ] Bad prompt နဲ့ good prompt ကွာခြားချက်ကိုခွဲနိုင်မယ်
[ ] Prompt Anatomy ရဲ့ အခြေခံအစိတ်အပိုင်းတွေကိုသိမယ်
[ ] Context, Goal, Scope, Constraints, Acceptance Criteria, Verification ကိုရေးနိုင်မယ်
[ ] Allowed files နဲ့ Do-not-touch list ကို prompt ထဲထည့်နိုင်မယ်
[ ] Output format ကိုတိတိကျကျသတ်မှတ်နိုင်မယ်
[ ] Approval gate ပါတဲ့ prompt ရေးနိုင်မယ်
[ ] Read-only prompt, plan-first prompt, edit prompt, debug prompt, review prompt ကိုခွဲနိုင်မယ်
[ ] Agent ကို “ဘာလုပ်ရမလဲ” တင်မက “ဘာမလုပ်ရဘူးလဲ” ပါခိုင်းနိုင်မယ်
[ ] Prompt တစ်ခုကို checklist နဲ့ review လုပ်နိုင်မယ်
```

ဒီအခန်းက Agentic Coding course တစ်ခုလုံးအတွက် core skill ပါ။ Tools setup, Git, permissions, hooks, subagents, tests, deployment အကုန်ရှိနေပေမယ့် prompt မကောင်းရင် Agent ကလမ်းလွဲနိုင်တယ်။ Prompt ကောင်းရင် beginner တစ်ယောက်တောင် Agent ကို junior developer တစ်ယောက်လို စနစ်တကျခိုင်းနိုင်လာမယ်။

## 2. Beginner Explanation

Prompt ဆိုတာ AI Agent ကိုပေးတဲ့ အလုပ်အမိန့်စာပါ။

မကောင်းတဲ့ prompt—

```text
Registration လုပ်ပေး။
```

ဒီ prompt က Agent အတွက်မရှင်းဘူး။

Agent မသိနိုင်တာ—

```text
Frontend form လား?
Backend API လား?
Database table လား?
Admin dashboard လား?
PWA check-in app လား?
Validation ပါလား?
Test run ရမလား?
File ဘယ်ဟာတွေပြင်ရမလဲ?
ဘာမလုပ်ရဘူးလဲ?
```

ကောင်းတဲ့ prompt—

```text
Goal:
Registration form မှာ phone field empty ဖြစ်ရင် မြန်မာလို error message ပြချင်ပါတယ်။

Scope:
Frontend validation only.

Allowed file:
- playground/chapter-11-coding-agent/app.js

Do not touch:
- package.json
- .env
- node_modules/**
- backend files

Constraints:
- Package install မလုပ်ပါနဲ့။
- Existing form submit flow မပြောင်းပါနဲ့။
- Code မပြင်ခင် plan ပေးပါ။

Acceptance Criteria:
- Phone empty ဖြစ်ရင် “ဖုန်းနံပါတ်ဖြည့်ပါ။” ပြရမယ်။
- Name validation behavior မပျက်ရဘူး။
- Valid form submit behavior မပျက်ရဘူး။

Verification:
- Empty phone test
- Valid form test
- git diff summary
```

ဒီ prompt က Agent ကို ဘာလုပ်ရမလဲ၊ ဘာမလုပ်ရဘူးလဲ၊ ဘာနဲ့အောင်မြင်တယ်လို့စစ်မလဲ တိတိကျကျပြောထားတယ်။

Claude Code best-practices docs မှာလည်း instructions ပိုတိကျလေ corrections နည်းလေဖြစ်ကြောင်း, specific files, constraints, existing patterns တွေကိုပြောပေးသင့်ကြောင်း, larger features အတွက် spec ကို self-contained ဖြစ်အောင်ရေးပြီး files/interfaces involved, out-of-scope, end-to-end verification step တွေထည့်သင့်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

# 3. Prompt Anatomy ဆိုတာဘာလဲ

Prompt Anatomy ဆိုတာ prompt ကောင်းတစ်ခုကိုဖွဲ့စည်းထားတဲ့ အစိတ်အပိုင်းတွေပါ။

Beginner အတွက် အရေးကြီးဆုံး structure—

```text
1. Role
2. Mode
3. Context
4. Goal
5. Scope
6. Non-goals
7. Inputs / References
8. Allowed files
9. Do-not-touch list
10. Constraints
11. Acceptance Criteria
12. Workflow
13. Verification
14. Output Format
15. Approval Gates
```

အမြဲအားလုံးထည့်ရမယ်မဟုတ်ပါဘူး။ Task သေးရင်တိုတိုရေးလို့ရတယ်။ Task ကြီးရင် ဒီ anatomy ကိုအပြည့်သုံးပါ။

# 4. Prompt Anatomy Overview

## 4.1 Role

Agent ကို ဘယ် role နဲ့အလုပ်လုပ်စေချင်လဲပြောပါ။

```text
သင်က careful frontend developer ဖြစ်တယ်။
```

သို့မဟုတ်—

```text
သင်က verifier agent ဖြစ်တယ်။ Code မပြင်ဘဲ acceptance criteria pass/fail စစ်ပါ။
```

Role က Agent ရဲ့ mindset ကိုချိန်ပေးတယ်။

ဥပမာ—

```text
Role: Beginner-friendly tutor
→ ရှင်းပြတာဦးစားပေး

Role: Senior security auditor
→ risk/security review ဦးစားပေး

Role: Frontend developer
→ UI/component implementation ဦးစားပေး

Role: Verifier
→ pass/fail evidence ဦးစားပေး
```

---

## 4.2 Mode

Agent ကို ဘယ် mode နဲ့လုပ်စေချင်လဲပြောပါ။

```text
Read-only mode
Plan-first mode
Edit mode
Review mode
Debug mode
Verification mode
```

ဥပမာ—

```text
Read-only mode အနေနဲ့လုပ်ပါ။
Code မပြင်ပါနဲ့။
```

သို့မဟုတ်—

```text
Plan first လုပ်ပါ။
Code မပြင်ခင် implementation plan ပေးပါ။
```

Claude Code permission mode docs မှာ Plan Mode သည် files ဖတ်ပြီး shell commands နဲ့ explore လုပ်ကာ source files မပြင်ဘဲ plan တင်ပြနိုင်ကြောင်း, plan approve ပြီးမှ editing စတင်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][2])

---

## 4.3 Context

Project / task အကြောင်း background ပေးပါ။

```text
ဒီ project က Agentic Coding Zero to Hero course workspace ဖြစ်ပါတယ်။
ဒီ lab app က static HTML/CSS/JS practice app ဖြစ်ပြီး backend API မရှိပါ။
```

Context မပါရင် Agent က assumption လုပ်ရမယ်။ Assumption များလေလမ်းလွဲနိုင်လေပါ။

ကောင်းတဲ့ context—

```text
ဒီ file က playground/chapter-11-coding-agent app ရဲ့ frontend-only registration form ဖြစ်ပါတယ်။
Backend API မရှိသေးပါ။
Package install မလိုပါ။
```

မကောင်းတဲ့ context—

```text
ဒီ app ကိုပြင်ပေး။
```

---

## 4.4 Goal

ဘာဖြစ်ချင်လဲ တိတိကျကျရေးပါ။

မကောင်း—

```text
Form ကိုကောင်းအောင်လုပ်ပါ။
```

ကောင်း—

```text
Form submit လုပ်တဲ့အခါ valid name + phone ဖြစ်ရင် attendee list ထဲ record အသစ်ထည့်ပြီး ticket code ပြချင်ပါတယ်။
```

Goal က outcome ကိုပြောရမယ်။ Claude Code prompt library docs မှာလည်း prompt ကောင်းတွေက “steps” ထက် “outcome” ကိုဖော်ပြတတ်ကြောင်း, Agent ကို သူ့ဘာသာ relevant files ရှာခွင့်ပေးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][3])

---

## 4.5 Scope

ဒီ task မှာ ဘယ်အထိလုပ်မလဲသတ်မှတ်ပါ။

```text
Scope:
- Frontend validation only
- Existing app.js file ထဲပဲပြင်
- Backend API မထည့်
- Package install မလုပ်
```

Scope က Agent ကို လိုတာထက်ပိုမလုပ်အောင်ကာကွယ်တယ်။

---

## 4.6 Non-goals

ဘာမလုပ်ရဘူးလဲ ရှင်းရှင်းရေးပါ။

```text
Non-goals:
- Backend API မထည့်ပါ
- Database မထည့်ပါ
- Email/SMS notification မထည့်ပါ
- QR code မထည့်ပါ
- Deployment မလုပ်ပါ
```

Non-goals မပါရင် Agent က “ပိုကောင်းအောင်” လုပ်ရင်း scope ကြီးသွားနိုင်တယ်။

---

## 4.7 Inputs / References

Agent ကို ဘာ files / docs / errors / examples ကိုကြည့်ရမလဲပြောပါ။

```text
References:
- COURSE_DECISIONS.md
- playground/chapter-11-coding-agent/app.js
- Existing validation function
```

သို့မဟုတ်—

```text
Error:
[paste error]

Current code:
[paste relevant code only]
```

Claude Code prompt library docs မှာ prompt ထဲမှာ existing file, test, pattern, error, log, screenshot, plan output စတဲ့ artifact တွေကိုပေးခြင်းက Agent ကို သင့် description အစား source ကိုဖတ်စေပြီး output ပိုတိကျစေကြောင်းဖော်ပြထားပါတယ်။ ([Claude][3])

---

## 4.8 Allowed Files

Agent ဘယ် files/folders ကိုဖတ်/ပြင်ခွင့်ရှိလဲသတ်မှတ်ပါ။

```text
Allowed files:
- playground/chapter-11-coding-agent/index.html
- playground/chapter-11-coding-agent/style.css
- playground/chapter-11-coding-agent/app.js
```

Allowed files မပါရင် Agent က unrelated files တွေပြင်နိုင်တယ်။

---

## 4.9 Do-not-touch List

မထိရမယ့် files/folders တွေကို explicit ရေးပါ။

```text
Do not touch:
- .env
- package.json
- package-lock.json
- node_modules/**
- dist/**
- build/**
- projects/event-management/**
- deployment files
```

Secret/safety အတွက် အရေးကြီးတယ်။

---

## 4.10 Constraints

နည်းပညာ/စည်းကမ်း restrictions တွေကိုရေးပါ။

```text
Constraints:
- Package install မလုပ်ပါနဲ့
- Backend API မထည့်ပါနဲ့
- Existing function name မပြောင်းပါနဲ့
- DOM selectors မပြောင်းပါနဲ့
- UI text မြန်မာလိုဖြစ်ရမယ်
- Accessibility label/input association မပျက်ရဘူး
```

Constraints က Agent ကို “ဘယ်နည်းလမ်းတွေနဲ့မလုပ်ရဘူး” ကိုပြောတယ်။

---

## 4.11 Acceptance Criteria

Feature အောင်မြင်/မအောင်မြင် စစ်လို့ရတဲ့အချက်တွေပါ။

```text
Acceptance Criteria:
- Empty name shows “နာမည်ဖြည့်ပါ။”
- Empty phone shows “ဖုန်းနံပါတ်ဖြည့်ပါ။”
- Valid submit shows success message
- Browser console has no errors
- Only allowed files changed
```

Acceptance Criteria တွေကို checklist နဲ့ရေးပါ။ “ကောင်းရမယ်” လို့မရေးပါနဲ့။ “စစ်လို့ရတဲ့အရာ” ဖြစ်ရမယ်။

---

## 4.12 Workflow

Agent လုပ်ရမယ့်အဆင့်တွေကိုရေးပါ။

```text
Workflow:
1. Read allowed files
2. Explain current behavior
3. Present implementation plan
4. Wait for approval
5. Implement after approval
6. Provide manual test steps
7. Provide git diff summary
```

Workflow က Agent ကို လုပ်ငန်းစဉ်ထိန်းပေးတယ်။

---

## 4.13 Verification

Agent တကယ်စစ်ရမယ့် command/manual checks တွေကိုရေးပါ။

```text
Verification:
- Manual browser test:
  - empty name
  - empty phone
  - valid submit
- Browser console error check
- git diff summary
```

Project မှာ scripts ရှိရင်—

```text
Verification:
- npm run build
- npm test
- git diff summary
```

Claude Code prompt library docs မှာ Agent ကို “run, test, compare, verify” လုပ်နိုင်တဲ့ self-check ပေးထားရင် Agent က တစ်ခါလုပ်ပြီးရပ်မနေဘဲ iterate လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][3])

---

## 4.14 Output Format

Agent response ကို ဘယ်ပုံစံနဲ့လိုချင်လဲပြောပါ။

```text
Output format:

## Plan
## Files to Change
## Risks
## Verification Steps
## Approval Needed
```

သို့မဟုတ် final summary အတွက်—

```text
## Summary
## Changed Files
## Acceptance Criteria Check
## Verification
## Known Risks
## Suggested Commit Message
```

Output format သတ်မှတ်ထားရင် Agent response ကိုဖတ်ရလွယ်တယ်။

Claude Code prompt library docs မှာ answer format, length, audience ကိုသတ်မှတ်ခြင်းက output ကိုသင်အသုံးပြုချင်တဲ့ပုံစံနဲ့ကိုက်စေကြောင်းဖော်ပြထားပါတယ်။ ([Claude][3])

---

## 4.15 Approval Gates

Agent ဘယ်အရာတွေမလုပ်ခင် approval တောင်းရမလဲရေးပါ။

```text
Approval Gates:
- Code မပြင်ခင် plan approval တောင်းပါ
- Package install မလုပ်ခင် approval တောင်းပါ
- Git commit မလုပ်ခင် approval တောင်းပါ
- Deployment action မလုပ်ပါ
```

Agentic Coding မှာ approval gate မရှိရင် Agent က အဆင့်တွေကျော်သွားနိုင်တယ်။

[project/task background]

[desired outcome]

[what to do]

Non-goals:
[what not to do]

Inputs / References:
[files, errors, docs, examples]

Allowed:
[allowed files/folders/actions]

Do not touch:
[blocked files/folders/actions]

[technical/business/safety rules]

[checkable success criteria]

[how response should be structured]

ဒီ formula အားလုံးကို အမြဲ copy-paste မလုပ်ရပါ။ Task ကြီး/ရှုပ်/အန္တရာယ်ရှိလေ ဒီ formula ကိုပိုအပြည့်သုံးပါ။

# 6. Prompt Length ဘယ်လောက်ဖြစ်သင့်လဲ

Prompt ကောင်းတာက အရှည်ဆုံး prompt မဟုတ်ပါဘူး။ **လိုအပ်တဲ့ context ကို မပိုမလိုပေးထားတဲ့ prompt** ဖြစ်တယ်။

## Small task

```text
playground/chapter-11-coding-agent/app.js ထဲက success message ကိုမြန်မာလိုပြင်ပါ။
တခြား file မပြင်ပါနဲ့။
ပြင်ပြီး diff summary ပေးပါ။
```

## Medium task

```text
Plan first + allowed files + acceptance criteria + verification ပါရမယ်။
```

## Large task

```text
Feature Brief / SPEC section / non-goals / verification / approval gates ပါရမယ်။
```

Claude Code best-practices docs မှာ planning သည် scope မရှင်း, multiple files ပြင်, unfamiliar code ဖြစ်တဲ့အခါအသုံးဝင်ပြီး typo fix, log line ထည့်ခြင်း, variable rename လို diff ကိုတစ်ကြောင်းနဲ့ဖော်ပြနိုင်တဲ့ small tasks တွေမှာ plan overhead မလိုနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

# 32. Chapter Summary

ဒီအခန်းမှာ **Prompt Anatomy** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Prompt = Agent ကိုပေးတဲ့ task specification
```

Prompt ကောင်းတစ်ခုမှာ ပါနိုင်တဲ့အပိုင်းတွေ—

```text
Role
Mode
Context
Goal
Scope
Non-goals
Inputs / References
Allowed files
Do-not-touch list
Constraints
Acceptance Criteria
Workflow
Verification
Output Format
Approval Gates
```

အရေးကြီးဆုံး rule—

```text
Agent ကို “ဘာလုပ်ရမလဲ” တင်မက “ဘာမလုပ်ရဘူးလဲ” ပါပြောပါ။
```

မကောင်းတဲ့ prompt—

```text
Registration history လုပ်ပေး။
```

ကောင်းတဲ့ prompt—

```text
Plan first. Static frontend only. Backend/API/package install မပါ။
Allowed files ၃ ဖိုင်ပဲပြင်ပါ။
Acceptance criteria အတိုင်းစစ်ပါ။
Approval မရသေးဘဲ code မပြင်ပါနဲ့။
```

Agentic Coding အတွက် takeaway—

```text
Prompt ကောင်းရင် Agent ကောင်းကောင်းအလုပ်လုပ်နိုင်တယ်။
Prompt မကောင်းရင် powerful Agent တောင် scope ကျော်၊ wrong file edit, weak verification, security risk ဖြစ်နိုင်တယ်။
```
