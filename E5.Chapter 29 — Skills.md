## 2. Beginner Explanation

Skill ဆိုတာ AI Agent ကို “အလုပ်တစ်မျိုးကို ထပ်ခါတလဲလဲ မှန်မှန်လုပ်နိုင်အောင် သင်ပေးထားတဲ့ reusable instruction package” ပါ။

ဥပမာ သင် Agent ကို မကြာခဏ ဒီလိုခိုင်းနေရတယ်ဆိုပါစို့။

```text
git diff ကို review လုပ်ပါ။
Scope ထဲပါလားစစ်ပါ။
Secret/API key ပါလားစစ်ပါ။
package.json မလိုဘဲပြင်ထားလားစစ်ပါ။
Acceptance criteria pass ဖြစ်လားစစ်ပါ။
PASS / WARNING / FAIL နဲ့ပြန်ပေးပါ။
```

ဒီ prompt ကို session တိုင်း copy-paste လုပ်နေရမယ့်အစား `diff-review` skill တစ်ခုရေးထားနိုင်တယ်။

ပြီးရင်နောက်ပိုင်း—

```text
/diff-review
```

သို့မဟုတ်—

```text
Current changes ကို review လုပ်ပါ။
```

လို့မေးလိုက်ရုံနဲ့ Agent က အဲဒီ skill ထဲက instructions ကိုသုံးပြီး review workflow ကိုလိုက်နိုင်မယ်။

Claude Code docs က Skill ကို `SKILL.md` file ပါတဲ့ directory တစ်ခုအဖြစ်ဖော်ပြထားပြီး Claude က skill ကို task နဲ့သက်ဆိုင်တယ်လို့ဆုံးဖြတ်တဲ့အခါ load လုပ်နိုင်သလို user က `/skill-name` နဲ့တိုက်ရိုက် invoke လုပ်နိုင်ကြောင်းရှင်းပြထားပါတယ်။ Skill body ဟာ အသုံးပြုတဲ့အချိန်မှ context ထဲဝင်တာကြောင့် CLAUDE.md ထဲ procedure အရှည်ကြီးထည့်ထားတာထက် context control ပိုကောင်းပါတယ်။ ([Claude][1])

Mental model—

```text
Prompt
= တစ်ကြိမ်သုံး instruction

CLAUDE.md / AGENTS.md
= project-wide general rules

Skill
= ထပ်ခါတလဲလဲသုံးမယ့် procedure / checklist / workflow

Subagent
= specialized worker role

MCP
= external tools/services access

Hook
= deterministic safety/automation checkpoint
```

# 3. Skill ဆိုတာဘာလဲ

Skill ဆိုတာ folder တစ်ခုပါ။ အဲဒီ folder ထဲမှာ အနည်းဆုံး `SKILL.md` file ပါရမယ်။ `SKILL.md` ထဲမှာ YAML frontmatter နဲ့ instructions ပါတယ်။ Claude Code docs က skill တစ်ခုမှာ `SKILL.md` သည် required entrypoint ဖြစ်ပြီး templates, examples, scripts, references စတဲ့ supporting files တွေကို optional အဖြစ်ထည့်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

Skill folder example—

```text
.claude/skills/diff-review/
├── SKILL.md
├── templates/
│   └── review-output.md
├── examples/
│   └── good-review.md
└── scripts/
    └── summarize-diff.sh
```

အနည်းဆုံး version—

```text
.claude/skills/diff-review/
└── SKILL.md
```

Skill ကို ဒီလိုမှတ်ပါ။

```text
Skill = reusable workflow
SKILL.md = workflow instructions
description = Agent ကို ဘယ်အချိန် skill သုံးရမလဲပြောတဲ့ trigger
supporting files = references/templates/scripts/examples
```

# 4. Skill ဘာကြောင့်လိုလဲ

Prompt တစ်ခါတည်းနဲ့အလုပ်ပြီးရင် skill မလိုပါဘူး။ ဒါပေမယ့် တူညီတဲ့ procedure ကို အကြိမ်ကြိမ်ပြန်သုံးနေရင် skill လိုလာတယ်။

Skill လိုလာတဲ့ signs—

```text
[ ] Prompt တစ်ခုကို copy-paste မကြာခဏလုပ်နေရတယ်
[ ] CLAUDE.md ထဲ procedure အရှည်ကြီးတိုးလာတယ်
[ ] Agent ကို review format တူတူအမြဲပြောနေရတယ်
[ ] Testing checklist တူတူပြန်ပြောနေရတယ်
[ ] Deployment runbook တစ်ခုကိုအကြိမ်ကြိမ်သုံးမယ်
[ ] Frontend starter pattern တစ်ခုထပ်ခါတလဲလဲလိုတယ်
[ ] Backend endpoint starter workflow တစ်ခုလိုတယ်
```

Claude Code docs က “same instructions, checklist, or multi-step procedure” ကို chat ထဲထပ်ခါတလဲလဲ paste လုပ်နေရင် သို့မဟုတ် CLAUDE.md section တစ်ခုက fact/rule မဟုတ်တော့ဘဲ procedure တစ်ခုလိုကြီးလာရင် skill ဖန်တီးသင့်တယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][1])

# 5. Progressive Disclosure

Skill ရဲ့အရေးကြီးဆုံး idea က **progressive disclosure** ပါ။

Progressive disclosure ဆိုတာ—

```text
Agent က skill အားလုံးရဲ့ full instructions ကိုအစကတည်းကမဖတ်ဘဲ
name / description လောက်ပဲသိထားတယ်။
Task နဲ့ကိုက်တဲ့ skill ဖြစ်မှ SKILL.md အပြည့်ဖတ်တယ်။
SKILL.md ထဲက referenced files တွေကိုလည်းလိုအပ်မှဖတ်တယ်။
```

Codex docs က Skills တွေဟာ context efficiency အတွက် progressive disclosure သုံးကြောင်း၊ Codex က skill name, description, file path ကိုပဲအရင်သိထားပြီး skill သုံးဖို့ဆုံးဖြတ်မှ full `SKILL.md` instructions ကို load လုပ်ကြောင်းရှင်းပြထားပါတယ်။ ([OpenAI Developers][2])

Claude platform docs ကလည်း Skill architecture မှာ metadata, instructions, resources ဆိုပြီး staged loading ဖြစ်နိုင်ကြောင်းဖော်ပြထားပြီး unused skill resources တွေဟာ context window ထဲနေရာမယူကြောင်းရှင်းပြထားပါတယ်။ ([Claude Platform][3])

Mental model—

```text
CLAUDE.md
= session စချိန် context ထဲဝင်

Skill description
= startup catalog ထဲဝင်

SKILL.md body
= skill trigger ဖြစ်မှ context ထဲဝင်

references/scripts/assets
= SKILL.md ကလိုအပ်လို့ဖတ်/သုံးမှ context/tool result ထဲဝင်
```

# 6. Skill vs CLAUDE.md

| အချက်    | CLAUDE.md                            | Skill                                   |
| -------- | ------------------------------------ | --------------------------------------- |
| Purpose  | project-wide persistent instructions | reusable task-specific workflow         |
| Loaded   | session startup မှာ loaded           | skill triggered ဖြစ်မှ loaded           |
| Best for | facts, rules, conventions            | step-by-step procedures                 |
| Length   | concise ဖြစ်သင့်                     | concise SKILL.md + optional references  |
| Example  | “Do not read .env”                   | “Run code review checklist”             |
| Risk     | ကြီးလွန်းရင် context bloat           | trigger မကောင်းရင် skill မသုံး/မှားသုံး |

Rule—

```text
Always-needed rule → CLAUDE.md / AGENTS.md
Repeated procedure → Skill
Hard safety → Permission / Hook
Product behavior → SPEC.md
```

# 7. Skill vs Command

Claude Code docs အရ custom commands have been merged into skills; `.claude/commands/deploy.md` နဲ့ `.claude/skills/deploy/SKILL.md` နှစ်ခုလုံးက `/deploy` command အဖြစ်အလုပ်လုပ်နိုင်ပြီး existing commands files တွေက ဆက်အလုပ်လုပ်ပါတယ်။ Skill က supporting files, invocation control, subagent execution, dynamic context စတဲ့ additional features တွေပါလာနိုင်တာကြောင့် new reusable workflow တွေအတွက် skill ကိုပိုအကြံပြုနိုင်ပါတယ်။ ([Claude][1])

Course strategy—

```text
Chapter 29 Skills
= reusable workflow package

Chapter 30 Commands, if used
= slash-command style invocation / legacy/simple command patterns

Practical rule
= New complex workflow ကို Skill အဖြစ်ရေးပါ။
  Simple one-file command prompt ကို command/skill နှစ်မျိုးလုံးဖြစ်နိုင်တယ်။
```

# 8. Skill vs Subagent

| အချက်        | Skill                                 | Subagent                                        |
| ------------ | ------------------------------------- | ----------------------------------------------- |
| What         | reusable procedure/instructions       | specialized role / worker                       |
| Example      | diff-review skill                     | verifier agent                                  |
| Main purpose | workflow reuse                        | context isolation + role specialization         |
| Can combine? | Yes                                   | Skill can run inside subagent in some tools     |
| Best for     | checklist, runbook, standard workflow | review, audit, exploration, implementation role |

Claude Code docs မှာ skill တစ်ခုကို forked Explore agent လို subagent context ထဲမှာ run စေနိုင်တဲ့ example ဖော်ပြထားပြီး results ကို main conversation ပြန်ပေးနိုင်ကြောင်းရှင်းပြထားပါတယ်။ ([Claude][1])

Course rule—

```text
Skill = ဘယ်လိုလုပ်မလဲ
Subagent = ဘယ်သူလုပ်မလဲ
```

ဥပမာ—

```text
Skill:
diff-review workflow

Subagent:
verifier agent

Together:
verifier agent uses diff-review skill
```

# 9. Skill vs MCP

| အချက်                   | Skill                                  | MCP                                    |
| ----------------------- | -------------------------------------- | -------------------------------------- |
| What                    | instructions/resources/scripts package | external tool/server protocol          |
| Example                 | “review git diff” workflow             | GitHub issue tool, database query tool |
| Needs external service? | မလိုနိုင်                              | များသောအားဖြင့်လိုနိုင်                |
| Best for                | process knowledge                      | live tool/data access                  |
| Safety                  | skill instructions/scripts risk        | external tool permission/data risk     |

Rule—

```text
Skill = Agent ကို workflow သင်ပေး
MCP = Agent ကို external capability ပေး
```

ဥပမာ—

```text
Skill:
“Before deploying, run this checklist.”

MCP:
DigitalOcean app deploy tool.
```

Deployment skill က MCP tool ကို ဘယ်လိုအသုံးပြုရမလဲညွှန်နိုင်တယ်။ MCP tool က actual cloud action လုပ်နိုင်တယ်။ ဒီနှစ်ခုကိုမရောပါနဲ့။

# 10. Skill Folder Locations

## Claude Code

Claude Code skills ကို personal, project, plugin locations မှာထားနိုင်တယ်။ Personal skill က `~/.claude/skills/<skill-name>/SKILL.md` ဖြစ်ပြီး project skill က `.claude/skills/<skill-name>/SKILL.md` ဖြစ်ပါတယ်။ Nested `.claude/skills/` folders တွေလည်း monorepo package-specific skills အဖြစ် discover ဖြစ်နိုင်ပြီး skill names conflict ဖြစ်ရင် namespace/precedence rules ရှိပါတယ်။ ([Claude][1])

Common locations—

```text
~/.claude/skills/
= personal skills, all projects

.claude/skills/
= current project skills

projects/event-management/.claude/skills/
= project area-specific skills

apps/web/.claude/skills/
= nested frontend-specific skills
```

## Codex

Codex skills ကို repository, user, admin, system locations မှာဖတ်နိုင်ပြီး repository mode မှာ current working directory ကနေ repo root အထိ `.agents/skills` folders တွေကို scan လုပ်နိုင်ပါတယ်။ User skill folder က `$HOME/.agents/skills`, admin location က `/etc/codex/skills`, bundled system skills တွေလည်းရှိနိုင်ပါတယ်။ ([OpenAI Developers][2])

Common locations—

```text
.agents/skills/
= repo skills for Codex

~/.agents/skills/
= personal Codex skills

/etc/codex/skills/
= shared machine/admin skills
```

Course workspace မှာ cross-tool learning အတွက် folder နှစ်မျိုးသုံးနိုင်တယ်။

```text
.claude/skills/
= Claude Code skills

.agents/skills/
= Codex/generic skills
```

# 11. Skill Anatomy

Skill တစ်ခုက အနည်းဆုံးဒီလိုဖြစ်မယ်။

```text
skill-name/
└── SKILL.md
```

`SKILL.md` structure—

```md
---
name: skill-name
description: What this skill does and exactly when to use it.
---

# Skill Name

## Purpose

[What this workflow does]

## When to Use

[Trigger situations]

## Do Not Use When

[Boundaries]

## Inputs Needed

[What context/data is required]

## Workflow

1. [step]
2. [step]
3. [step]

## Output Format

[expected response format]

## Safety Rules

[constraints]
```

Claude platform docs က every Skill requires `SKILL.md` with YAML frontmatter; `name` and `description` fields are required in the platform skill structure, while Claude Code docs notes frontmatter fields differ slightly and `description` is recommended for automatic selection. Codex docs explicitly say `SKILL.md` must include `name` and `description`. ([Claude Platform][3])

Course rule—

```text
Cross-tool compatibility အတွက် name + description ကိုအမြဲထည့်ပါ။
```

# 12. Skill Description ရေးနည်း

Skill selection အတွက် description ကအလွန်အရေးကြီးတယ်။ Codex docs က implicit matching က skill `description` ပေါ်မူတည်လို့ concise, clear scope and boundaries ပါတဲ့ description ရေးရန်၊ key use case and trigger words ကိုအစမှာထားရန်အကြံပြုထားပါတယ်။ ([OpenAI Developers][2])

Bad description—

```yaml
description: Helps with code.
```

ဘာကြောင့်မကောင်းလဲ—

```text
vague
trigger မရှင်း
skill မသုံးသင့်တဲ့အချိန်မသိ
အခြား skills တွေနဲ့ရောနိုင်
```

Good description—

```yaml
description: Review current git diff before commit. Use when the user asks what changed, requests a commit readiness check, or wants risk/secret/package/scope review of uncommitted changes.
```

ကောင်းတဲ့ description မှာ—

```text
[ ] action ပါ
[ ] trigger phrases ပါ
[ ] scope ပါ
[ ] output expectation hint ပါ
[ ] too broad မဖြစ်
```

# 13. Skill Body ရေးနည်း

Skill body ကို concise, imperative, reusable ဖြစ်အောင်ရေးပါ။

မကောင်း—

```md
You are a very smart and careful assistant and you should think deeply about many things...
```

ကောင်း—

```md
## Workflow

1. Inspect the current git diff.
2. Summarize changed files.
3. Check for secrets, package changes, generated files, and scope drift.
4. Return PASS / WARNING / FAIL with evidence.
```

Claude Code docs က skill body ကို concise ထားရန်, once a skill loads, its content stays in context across turns, so every line becomes recurring token cost ဖြစ်ကြောင်းသတိပေးထားပါတယ်။ Complex skills တွေမှာ supporting files ထည့်ပြီး main skill focused ဖြစ်အောင်ထားနိုင်ပါတယ်။ ([Claude][1])

# 14. Supporting Files

Skill folder ထဲမှာ optional supporting files ထည့်နိုင်တယ်။

```text
references/
= long docs, checklists, API rules

templates/
= output templates, task card templates

examples/
= good/bad examples

scripts/
= deterministic helper scripts

assets/
= reusable non-code resources
```

Example—

```text
.claude/skills/bug-report/
├── SKILL.md
```

# 16. When to Create a Skill

Skill ဖန်တီးသင့်တဲ့အချိန်—

```text
[ ] Same prompt/workflow ကို ၃ ကြိမ်ထက်ပိုသုံးမယ်
[ ] Procedure သည် multi-step ဖြစ်တယ်
[ ] Output format consistent လိုတယ်
[ ] Review checklist အမြဲတူတယ်
[ ] Safety/verification workflow အမြဲလိုတယ်
[ ] CLAUDE.md / AGENTS.md ထဲ procedure အရှည်ကြီးဖြစ်လာတယ်
[ ] Team/project အတွက် reusable workflow လိုတယ်
```

Skill မဖန်တီးသင့်သေးတဲ့အချိန်—

```text
[ ] One-off task
[ ] Requirement မသေချာသေး
[ ] Workflow မတည်ငြိမ်သေး
[ ] Skill body က prompt တစ်ကြောင်းလောက်ပဲ
[ ] Hard enforcement လိုတာကို skill နဲ့လုပ်ချင်နေ
```

Rule—

```text
Repeated procedure → Skill
One-time task → Prompt
Hard block → Permission/Hook
External action → MCP/Tool + approval
```

# 17. Skill Safety

Skill က powerful ဖြစ်နိုင်တယ်။ Skill ထဲမှာ scripts, references, command recipes ပါနိုင်တာကြောင့် safety စဉ်းစားရမယ်။

Skill safety checklist—

```text
[ ] Secret values မပါ
[ ] Dangerous command မပါ
[ ] Package install / deploy / database action မလုပ်ခင် approval gate ပါ
[ ] Scripts deterministic ဖြစ်
[ ] Scripts safe input validation ရှိ
[ ] Skill description too broad မဟုတ်
[ ] Skill does not override security rules
[ ] Skill output asks for human approval before commit/push/deploy
[ ] If skill should be manual only, disable implicit invocation
```

Claude Code docs မှာ skill access ကို permission rules နဲ့ထိန်းနိုင်ပြီး `Skill(name)` exact match သို့မဟုတ် `Skill(name *)` prefix match နဲ့ allow/deny လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ `disable-model-invocation: true` သည် Claude ကို skill ကို automatic load မလုပ်စေဖို့သုံးနိုင်ပြီး user menu visibility ကို `user-invocable` ထိန်းပေမယ့် tool access ကိုတားတာမဟုတ်ကြောင်း docs ကရှင်းပြထားပါတယ်။ ([Claude][1])

# 18. Skill Frontmatter Fields

Cross-tool safe frontmatter—

```yaml
---
name: diff-review
description: Review current git diff before commit. Use when the user asks what changed, wants commit readiness, or requests risk review of uncommitted changes.
---
```

Claude Code-specific fields တွေထည့်နိုင်တယ်—

```yaml

```

Claude Code docs က `description` ကို recommended field အဖြစ်ဖော်ပြပြီး Claude က skill ကိုဘယ်အချိန်သုံးမလဲဆုံးဖြတ်ဖို့အသုံးပြုကြောင်းဖော်ပြထားပါတယ်။ `disable-model-invocation` ကို `true` ထားရင် Claude automatic loading မလုပ်စေဘဲ manual `/name` invocation အတွက်ပိုသင့်တဲ့ workflows မှာအသုံးပြုနိုင်ပါတယ်။ ([Claude][1])

Codex skill frontmatter အတွက် `name` နဲ့ `description` required ဖြစ်ကြောင်း docs ကဖော်ပြထားပါတယ်။ ([OpenAI Developers][2])

Course rule—

```text
Always include:
- name
- description

Optional:
- disable-model-invocation, if manual-only workflow
- allowed-tools, if Claude-specific and safe
```

```md
name: diff-review
description: Review current git diff before commit. Use when the user asks what changed, wants commit readiness, or requests risk review of uncommitted changes.
---

# Diff Review Skill

## Purpose

Review uncommitted changes before commit.

## Inputs Needed

- Current task or expected scope, if available
- Acceptance criteria, if available
- Current git diff or permission to inspect it

## Workflow

1. Identify changed files.
2. Check whether changes match the expected task scope.
3. Check for secrets, tokens, `.env`, generated files, dependency folders, or unexpected package changes.
4. Check whether acceptance criteria appear covered.
5. Check whether verification evidence exists.
6. Return PASS, WARNING, or FAIL.

## Review Criteria

- Scope respected
- No secrets
- No `.env`
- No `node_modules/`, `dist/`, or `build/`
- No unexpected `package.json` or lock file changes
- Verification present
- Commit message can describe one coherent change

## Output Format

## Decision

PASS / WARNING / FAIL

## Changed Files

- path: summary

## Findings

| Severity | Finding | Evidence | Recommendation |
|---|---|---|---|

## Verification Gaps

- gap

## Suggested Commit Message

message
```


# 31. Common Errors

## Error 1 — Skill description vague ဖြစ်ခြင်း

Bad—

```yaml
description: Helps with review.
```

Good—

```yaml
description: Review current git diff before commit. Use when the user asks what changed, wants commit readiness, or requests risk review of uncommitted changes.
```

---

## Error 2 — Skill body အရမ်းရှည်ခြင်း

Fix—

```text
Main steps only in SKILL.md.
Long examples in examples/.
Templates in templates/.
Long docs in references/.
```

---

## Error 3 — Skill ကို hard enforcement လို့ထင်ခြင်း

Skill သည် instruction layer ဖြစ်တယ်။ Dangerous command block ချင်ရင် permission/hook သုံးပါ။

---

## Error 4 — Skill တစ်ခုထဲ အလုပ်အများကြီးထည့်ခြင်း

Bad—

```text
full-stack-devops-everything skill
```

Good—

```text
diff-review
bug-report
manual-verify
staging-deploy-plan
```

---

## Error 5 — Skill trigger များလွန်းခြင်း

Description too broad ဖြစ်ရင် unrelated task တွေမှာ trigger ဖြစ်နိုင်တယ်။

Fix—

```text
Use when / Do not use when boundaries ထည့်ပါ။
```

---

## Error 6 — Supporting files ကို SKILL.md ထဲမှာ mention မလုပ်ခြင်း

Skill body ထဲမှာ supporting file purpose ကိုရေးပါ။

```md
If you need the output template, read `templates/review-output.md`.
```

---

## Error 7 — Project-specific skill ကို personal folder ထဲထားခြင်း

Project-specific workflow ဖြစ်ရင် repo `.claude/skills/` သို့ `.agents/skills/` ထဲထားပါ။ Personal cross-project workflow ဖြစ်ရင် home folder ထဲထားပါ.

# 35. Chapter Summary

ဒီအခန်းမှာ **Skills** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Skill = Agent ကို repeated workflow တစ်ခုကို မှန်မှန်လုပ်စေတဲ့ reusable instruction package
```

Skill တစ်ခုမှာ—

```text
SKILL.md
= main instructions + metadata

references/
= long docs

templates/
= reusable output templates

examples/
= good/bad examples

scripts/
= deterministic helper scripts
```

ပါနိုင်တယ်။

Skills ရဲ့အရေးကြီးဆုံးအားသာချက်—

```text
Progressive disclosure:
Skill name/description ပဲအရင် context ထဲဝင်ပြီး
full instructions/supporting files ကိုလိုမှ load လုပ်တယ်။
```

ဘယ်အချိန် skill ဖန်တီးမလဲ—

```text
Same prompt/checklist/procedure ကိုထပ်ခါတလဲလဲသုံးနေရင်
CLAUDE.md / AGENTS.md ထဲ procedure အရှည်ကြီးဖြစ်လာရင်
Output format/verification workflow ကို consistent ဖြစ်စေချင်ရင်
```

အရေးကြီးဆုံး safety takeaway—

```text
Skill သည် guidance/reusable workflow ဖြစ်သည်။
Hard enforcement မဟုတ်ပါ။
Dangerous command, secret access, deployment action တို့ကို permissions/hooks/approval gates နဲ့ထပ်ကာကွယ်ရမယ်။
```

Agentic Coding အတွက် takeaway—

```text
Prompt တစ်ခါချင်းစီက task ကိုထိန်းတယ်။
CLAUDE.md / AGENTS.md က project-wide rules ကိုထိန်းတယ်။
Skills က repeated workflows ကို reusable ဖြစ်အောင်ထိန်းတယ်။
```
