# Chapter 15 — Safe Agent Permissions

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့သုံးနိုင်ရပါမယ်။

```text
[ ] Agent permission ဆိုတာဘာလဲ ရှင်းပြနိုင်မယ်
[ ] Allow / Ask / Deny permission mindset ကိုနားလည်မယ်
[ ] Agent ကို ဘာတွေ auto-allow လုပ်သင့်၊ ဘာတွေ approval တောင်းသင့်၊ ဘာတွေ deny လုပ်သင့်လဲ ခွဲနိုင်မယ်
[ ] Claude Code `/permissions` UI ကို conceptually နားလည်မယ်
[ ] Claude Code permission modes — default/manual, plan, acceptEdits, auto, dontAsk, bypassPermissions — ကို အခြေခံခွဲနိုင်မယ်
[ ] Codex sandbox + approval policy concept ကိုနားလည်မယ်
[ ] `.claude/settings.json` ထဲမှာ beginner-safe permission rules ရေးနိုင်မယ်
[ ] Risky commands တွေကို မျက်စိမှိတ်မ run ရကြောင်းနားလည်မယ်
[ ] Agent ကို package install, file delete, Git reset, deployment action မလုပ်ခင် approval တောင်းခိုင်းနိုင်မယ်
[ ] Permission rule နဲ့ prompt instruction မတူကြောင်းနားလည်မယ်
[ ] Hooks က next chapter မှာ deterministic enforcement layer ဖြစ်လာမယ်ဆိုတာနားလည်မယ်
```

ဒီအခန်းက Agentic Coding လုပ်တဲ့အခါ **အလုံခြုံဆုံးအခြေခံစည်းကမ်း** ဖြစ်ပါတယ်။ Agent တစ်ယောက်က file ဖတ်နိုင်၊ file ပြင်နိုင်၊ terminal command run နိုင်၊ package install လုပ်နိုင်လာတာနဲ့ power ကြီးလာပါတယ်။ Power ကြီးလာလေ **permission boundary** ပိုအရေးကြီးလာပါတယ်။

## 2. Beginner Explanation

Agent ကို coding project ထဲမှာ အလုပ်ခိုင်းတာကို အိမ်ထဲကို technician တစ်ယောက်ခေါ်လာတာနဲ့နှိုင်းယှဉ်ပါ။

Technician ကို—

```text
မီးခလုတ်စစ်ခွင့်ပေးမယ်
အိမ်ထဲဝင်ခွင့်ပေးမယ်
သိုလှောင်ခန်းမဖွင့်ခွင့်
မီးကြိုးဖြတ်မယ်ဆိုရင် အရင်မေးရမယ်
အိမ်တစ်လုံးလုံးဖျက်မယ်ဆိုရင် လုံးဝမလုပ်ရ
```

လို boundaries သတ်မှတ်ရမယ်။

AI Coding Agent ကိုလည်းဒီလိုသတ်မှတ်ရမယ်။

```text
README.md ဖတ်ခွင့် — Allow
git status run ခွင့် — Allow
npm install လုပ်ခွင့် — Ask
package.json ပြင်ခွင့် — Ask
.env ဖတ်ခွင့် — Deny
rm -rf run ခွင့် — Deny
production deploy — Ask / Deny until deployment chapter
```

အရေးကြီးဆုံး mindset—

```text
Agent ကို “ယုံကြည်” လို့မလုံလောက်ပါ။
Agent ကို “boundary ထဲမှာအလုပ်လုပ်စေ” ရမယ်။
```

Claude Code permissions docs မှာ permissions ကို `/permissions` UI နဲ့ကြည့်/စီမံနိုင်ပြီး allow rules က manual approval မလိုဘဲ tool ကိုခွင့်ပြု၊ ask rules က confirmation တောင်း၊ deny rules က tool use ကိုတားကြောင်းဖော်ပြထားပါတယ်။ Rules တွေကို deny → ask → allow အစဉ်လိုက် evaluate လုပ်ပြီး prompt instruction သို့မဟုတ် `CLAUDE.md` က model ကိုလုပ်စေချင်တာကိုလမ်းညွှန်နိုင်ပေမယ့် tool access ကိုတကယ် enforce လုပ်တာက permission rules, permission modes, PreToolUse hooks တို့ဖြစ်ကြောင်း official docs ကရှင်းပြထားပါတယ်။ ([Claude][1])

# 3. Permission ဆိုတာဘာလဲ

Permission ဆိုတာ Agent တစ်ယောက်က tool/action တစ်ခုကို လုပ်ခွင့်ရှိ/မရှိ သတ်မှတ်တဲ့စည်းကမ်းပါ။

Agentic Coding မှာ permission က အောက်ပါအရာတွေကိုထိန်းနိုင်တယ်။

```text
File read
File edit/write
Shell command / Bash command
Package install
Git operation
Network access
MCP tool access
Deployment action
Database command
```

Permission မရှိဘဲ Agent ကို free rein ပေးလိုက်ရင်—

```text
file မှားပြင်နိုင်
.env ဖတ်မိနိုင်
package မလိုဘဲထည့်နိုင်
git reset လုပ်မိနိုင်
database data ဖျက်မိနိုင်
production deploy လုပ်မိနိုင်
```

ဒါကြောင့် beginner အတွက် default approach က—

```text
Read more freely
Edit carefully
Run commands with explanation
Install/deploy/delete only with explicit approval
```

ဖြစ်သင့်တယ်။

# 4. Allow / Ask / Deny Mental Model

Permission ကို အလွယ်ဆုံး ၃ မျိုးခွဲပါ။

```text
Allow = အလိုအလျောက်ခွင့်ပြု
Ask   = လုပ်မယ်ဆိုရင် အရင်မေး
Deny  = လုံးဝမလုပ်ရ
```

## 4.1 Allow

Low-risk read-only သို့မဟုတ် safe commands တွေအတွက်သုံးမယ်။

ဥပမာ—

```text
pwd
ls
git status
git diff
cat README.md
npm run build, project အခြေအနေအရ
```

သို့သော် `npm run build` သည် project အလိုက် generated files ထုတ်နိုင်တာရှိလို့ beginner အဆင့်မှာ “Ask” ထားတာလည်းလုံခြုံတယ်။

## 4.2 Ask

Project state ပြောင်းနိုင်တဲ့ actions တွေအတွက်သုံးမယ်။

ဥပမာ—

```text
file edit
package install
git commit
git restore
git push
npm install
npm update
database migration
deployment command
```

Ask ဆိုတာ “မလုပ်ရ” မဟုတ်ပါဘူး။ “နားလည်ပြီးမှလုပ်မယ်” ဆိုတဲ့ gate ပါ။

## 4.3 Deny

Secret exposure, destructive action, course scope မကျော်အောင်တားရန်သုံးမယ်။

ဥပမာ—

```text
.env ဖတ်ခြင်း
secrets folder ဖတ်ခြင်း
node_modules ထဲဝင်ပြင်ခြင်း
rm -rf
git reset --hard
git clean -fd
curl ... | bash
production database drop
```

Deny ကို strict ထားသင့်တဲ့အရာတွေမှာ မလျော့ပါနဲ့။

# 5. Permission Rule နဲ့ Prompt Instruction မတူပါ

ဒီအချက်က အလွန်အရေးကြီးပါတယ်။

Prompt instruction—

```text
.env မဖတ်ပါနဲ့။
```

ဒါက Agent ကို “မလုပ်သင့်ဘူး” လို့ပြောတာပါ။

Permission rule—

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}
```

ဒါက tool level မှာ “လုပ်ခွင့်မရှိ” လို့တားတာပါ။

`CLAUDE.md` ထဲက rule တွေက Agent behavior ကိုလမ်းညွှန်ပေးတယ်။ ဒါပေမယ့် actual tool access ကို enforce လုပ်ချင်ရင် permission rules, permission modes, hooks တွေလိုတယ်။ Claude Code docs မှာလည်း permissions rules က Claude Code မှ enforce လုပ်တာဖြစ်ပြီး instructions က Claude ကိုဘာကြိုးစားမလဲ shape လုပ်ပေးသော်လည်း Claude Code ကဘာခွင့်ပြုမလဲကိုမပြောင်းကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

Mental model—

```text
CLAUDE.md / prompt = soft rule
permissions = hard gate
hooks = deterministic guardrail
```

Hooks ကို နောက် Chapter 17 မှာအပြည့်သင်မယ်။

# 6. Claude Code Permission Modes

Claude Code မှာ tool calls ကို ဘယ်လို approve လုပ်မလဲထိန်းတဲ့ permission modes ရှိတယ်။ Beginner အတွက် mode တွေကို အောက်ပါအတိုင်းနားလည်ပါ။

```text
default / manual
Plan Mode
acceptEdits
auto
dontAsk
bypassPermissions
```

Claude Code docs မှာ `default` mode က tool တစ်ခုကိုပထမဆုံးသုံးရာမှာ permission prompt ပြတာ, `acceptEdits` က working directory ထဲက file edits နဲ့ common filesystem commands တချို့ကို auto-accept လုပ်တာ, `plan` mode က source files မပြင်ဘဲ files ဖတ်/read-only shell commands နဲ့ explore လုပ်တာ, `auto` က research-preview အဖြစ် background safety checks နဲ့ tool calls auto-approve လုပ်တာ, `dontAsk` က pre-approved tools မဟုတ်ရင် auto-deny လုပ်တာ, `bypassPermissions` က prompts များကို skip လုပ်တာဖြစ်ပြီး isolated container/VM မဟုတ်ရင်မသုံးသင့်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

## 6.1 Recommended Beginner Mode

Beginner အတွက် default approach—

```text
Use default/manual mode.
Use plan mode for non-trivial tasks.
Avoid bypassPermissions.
Avoid broad auto approvals.
```

## 6.2 Plan Mode

Plan Mode က code မပြင်ခင် project ကိုဖတ်ပြီး plan တင်ဖို့အကောင်းဆုံးပါ။

သုံးသင့်တဲ့ prompt—

```text
Plan Mode အနေနဲ့လုပ်ပါ။
Code မပြင်ပါနဲ့။
Project files ကိုဖတ်ပြီး implementation plan ပေးပါ။
ကျွန်တော် approve လုပ်မှ edit လုပ်ပါ။
```

## 6.3 acceptEdits

`acceptEdits` က file edits တွေကို friction လျှော့ပေးနိုင်တယ်။ ဒါပေမယ့် beginner အတွက် အစပိုင်းမှာ manual review ပိုကောင်းတယ်။

Use only when—

```text
project clean ဖြစ်တယ်
task scope သေးတယ်
Git checkpoint ရှိတယ်
agent behavior ကိုယုံကြည်နိုင်တယ်
```

## 6.4 bypassPermissions

ဒီ mode ကို beginner အနေနဲ့ မသုံးပါနဲ့။ Docs ကလည်း filesystem/root/home directory removal circuit breaker တချို့ရှိသော်လည်း bypass mode သည် prompts ကို skip လုပ်နိုင်ပြီး containers/VMs လို isolated environments မှာသာသုံးရန်အကြံပြုထားပါတယ်။ ([Claude][1])

Course rule—

```text
Never use bypassPermissions on your main computer workspace.
```

# 7. Codex Sandbox + Approval Concept

Codex ဘက်မှာ permission mindset ကို sandbox + approval policy နဲ့နားလည်နိုင်တယ်။ Codex docs အရ default လုပ်ပုံမှာ network access ကိုပိတ်ထားပြီး local Codex သည် OS-enforced sandbox ကိုသုံးကာ active workspace ထဲမှာ write permission ကန့်သတ်ထားတတ်ပြီး approval policy က sandbox ထဲကနေထွက်ခြင်း, network သုံးခြင်း, trusted set အပြင် command run ခြင်းတို့အတွက် မေးရမလားဆုံးဖြတ်ပါတယ်။ ([OpenAI Developers][2])

Codex sandbox docs မှာ default permissions အနေနဲ့ `workspace-write` sandbox, `on-request` approval policy ပါပြီး Codex က current workspace ထဲရှိ files ဖတ်/ပြင်နိုင်, routine local commands run နိုင်, internet သို့မဟုတ် workspace boundary အပြင်သွားမယ်ဆိုရင်မေးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ `read-only`, `workspace-write`, `danger-full-access` modes တွေရှိပြီး `danger-full-access` သည် sandbox restrictions ဖယ်ရှားတာဖြစ်လို့ အလွန်သတိထားရမယ်။ ([OpenAI Developers][3])

Beginner translation—

```text
read-only
= ဖတ်/စစ်ရန်, edit မလုပ်ရန်

workspace-write
= workspace ထဲမှာ file edit + routine local commands, default local work

danger-full-access
= sandbox boundary မရှိသလောက်; beginner မသုံးသင့်
```

Course rule—

```text
Claude Code ဖြစ်ဖြစ် Codex ဖြစ်ဖြစ်:
read-only / plan-first / workspace-limited mode နဲ့စပါ။
danger/full bypass style mode မသုံးပါနဲ့။
```

# 8. Permission Categories

Agent actions တွေကို category အလိုက်ခွဲမယ်။

## 8.1 Read-only Actions

များသောအားဖြင့် low risk ဖြစ်တယ်။ ဒါပေမယ့် secret files မပါရ။

```text
pwd
ls
git status
git diff
cat README.md
cat package.json
read SPEC.md
read CLAUDE.md
read AGENTS.md
```

Allow or Ask?

```text
Allow:
README.md, SPEC.md, package.json, non-secret source files

Deny:
.env, secrets/**, private key files
```

## 8.2 File Edit Actions

Project state ပြောင်းနိုင်တယ်။

```text
Edit app.js
Write new file
Modify README
Modify package.json
Modify .gitignore
```

Default—

```text
Ask
```

Small documentation edit လို low-risk task မျိုးမှာ allow လုပ်နိုင်ပေမယ့် beginner course အတွက် file edits ကို plan + approval နဲ့သွားမယ်။

## 8.3 Shell Commands

Commands တွေကို risk level နဲ့ခွဲပါ။

Low-risk—

```text
pwd
ls
git status
git diff
node -v
npm -v
```

Medium-risk—

```text
npm run build
npm test
npm run dev
git add
git commit
```

High-risk—

```text
npm install
npm update
git push
git restore
git rm
database migration
deployment commands
```

Deny / never without serious review—

```text
rm -rf
git reset --hard
git clean -fd
curl ... | bash
sudo rm -rf
drop database
```

## 8.4 Network Actions

Network access က supply-chain/security risk ပါလာနိုင်တယ်။

Examples—

```text
curl
wget
npm install
package download
web fetch
MCP remote server
cloud API call
```

Default—

```text
Ask
```

## 8.5 Cloud / Deployment Actions

```text
DigitalOcean deploy
GitHub Actions secret update
database backup/restore
domain/DNS change
production deploy
scale app
delete app
```

Default—

```text
Ask or Deny until deployment chapter
```

# 9. Beginner Permission Matrix

| Action                  | Beginner Policy            | Reason                     |
| ----------------------- | -------------------------- | -------------------------- |
| `pwd`, `ls`             | Allow                      | Folder state စစ်ရန် safe   |
| `git status`            | Allow                      | Project state ဖတ်ရုံ       |
| `git diff`              | Allow                      | Changes review             |
| Read `README.md`        | Allow                      | Public project docs        |
| Read `package.json`     | Allow                      | Scripts/deps သိရန်         |
| Read `.env`             | Deny                       | Secrets ပါနိုင်            |
| Edit normal source file | Ask                        | Project state ပြောင်း      |
| Edit `package.json`     | Ask                        | dependency/scripts risk    |
| `npm run build`         | Ask/Allow project-specific | output changes ဖြစ်နိုင်   |
| `npm test`              | Ask/Allow project-specific | safe-ish but command run   |
| `npm install`           | Ask                        | dependencies/files ပြောင်း |
| `git commit`            | Ask                        | history save point         |
| `git push`              | Ask                        | remote/public sharing      |
| `git reset --hard`      | Deny                       | changes ဆုံးရှုံးနိုင်     |
| `git clean -fd`         | Deny                       | untracked files ဖျက်       |
| `rm -rf`                | Deny                       | destructive                |
| Deployment              | Ask/Deny                   | cloud/resource/cost risk   |

# 10. `.claude/settings.json` Beginner Template

Chapter 14 မှာ `.claude/settings.json` starter ထည့်ခဲ့တယ်။ Chapter 15 မှာ permissions ကိုပိုရှင်းအောင် template ပြင်မယ်။

> မှတ်ချက် — permission rule syntax/tool names တွေက tool version အလိုက်ပြောင်းနိုင်လို့ `/permissions` UI နဲ့ validate လုပ်ပါ။ Claude docs အရ rule syntax က `Tool` သို့မဟုတ် `Tool(specifier)` format ဖြစ်ပြီး `Bash(npm run build)`, `Read(./.env)`, `WebFetch(domain:example.com)` လို fine-grained specifiers သုံးနိုင်ပါတယ်။ Bash rules တွေမှာ wildcard patterns သုံးနိုင်ပေမယ့် broad deny သည် narrower allow ကို override လုပ်နိုင်ကြောင်းသတိထားရမယ်။ ([Claude][1])

```json
{
```

# 18. MCP Permission Preview

MCP ကို Part 16 မှာသင်မယ်။ အခုအဆင့်မှာ MCP tools တွေကို “external system access” အဖြစ်မြင်ထားပါ။

MCP tool တစ်ခုက—

```text
GitHub issue ဖတ်နိုင်
Database query လုပ်နိုင်
DigitalOcean app deploy လုပ်နိုင်
Ticket status update လုပ်နိုင်
```

ဖြစ်နိုင်တယ်။

Beginner policy—

```text
MCP read-only tools = Ask first until trusted
MCP write tools = Ask / deny until specific chapter
MCP cloud/deploy tools = Deny until deployment chapter
MCP destructive tools = Deny
```

Claude Code permission docs မှာ MCP tool names တွေကို `mcp__server__tool` format နဲ့ match လုပ်နိုင်ပြီး MCP tools အားလုံးကို deny လုပ်ချင်ရင် `mcp__*` pattern ကို deny rule အနေနဲ့သုံးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

Example—

```json
{
  "permissions": {
    "deny": [
      "mcp__*"
    ]
  }
}
```

ဒီကိုအစပိုင်းမှာသုံးပြီး နောက်ပိုင်း MCP chapter ရောက်မှ scoped allow/ask rules ပြန်ထည့်နိုင်တယ်။

```text
Manual approval for edits

No MCP write tools
```

```text

Ask for frontend file edits
```

```text
Ask for backend edits
Ask for database commands
Ask for migrations
Ask for auth/RBAC changes
Deny production database changes
```

```text
Read-only MCP first
Ask for staging deploy
Deny production deploy until manual approval
Ask for DigitalOcean actions
Deny delete/scale/destructive cloud commands unless explicit deployment chapter lab
```

```text
Least privilege
Audit logs
Verifier/Auditor read-only
Deployer approval gates
Rollback plan required
```

## Error 2 — Allow rules အရမ်းကျယ်ခြင်း

မကောင်း—

```json
{
  "permissions": {
    "allow": [
      "Bash(*)"
    ]
  }
}
```

ပြဿနာ—

```text
Agent command အများကြီး auto-run လုပ်နိုင်တယ်။
```

ပြင်နည်း—

```text
Allow only exact low-risk commands.
Ask for npm/git state-changing commands.
Deny destructive commands.
```

---

## Error 3 — Deny broad rule နဲ့ allow exception ထင်ခြင်း

Claude Code docs အရ deny → ask → allow အစဉ်လိုက် evaluate လုပ်ပြီး broad deny rule က narrower allow ကို override လုပ်နိုင်ပါတယ်။ ဥပမာ `Bash(aws *)` deny ထားရင် `Bash(aws s3 ls)` allow ထားလည်း block ဖြစ်နိုင်တယ်။ ([Claude][1])

Beginner lesson—

```text
Deny rule ကို broad ထားမယ်ဆိုရင် exceptions လုပ်လို့မရနိုင်တာသတိထားပါ။
```

---

## Error 4 — `bypassPermissions` သုံးခြင်း

Beginner မသုံးသင့်ပါ။ Main machine/workspace မှာ အန္တရာယ်များပါတယ်။

ပြင်နည်း—

```text
default/manual + plan mode သုံးပါ။
```

---

## Error 5 — `.env.example` ကို deny ထဲထည့်မိခြင်း

`.env.example` ကို Agent ဖတ်နိုင်ရင် variable names သိနိုင်တယ်။ Actual secret မပါရင် safe ဖြစ်တတ်တယ်။

Deny—

```text
.env
.env.local
.env.*.local
```

Allow/read—

```text
.env.example
```

---

## Error 6 — `node_modules` ကို Agent ဖတ်/ပြင်ခိုင်းခြင်း

`node_modules` က dependency output folder ဖြစ်တယ်။ Agent မဖတ်/မပြင်သင့်ပါ။

ပြင်နည်း—

```text
Read(./node_modules/**) deny
node_modules/ in .gitignore
```

---

## Error 7 — Ask rules မထားဘဲ package install ခွင့်ပြုခြင်း

`npm install` က `package.json`, `package-lock.json`, `node_modules` ကိုပြောင်းနိုင်တယ်။

ပြင်နည်း—

```text
Bash(npm install*) → Ask
```

## 24.4 Permission policy ကြောင့် agent stuck ဖြစ်ရင်

တစ်ခါတလေ deny/ask rules များလွန်းလို့ Agent အလုပ်မလုပ်နိုင်တာဖြစ်နိုင်တယ်။

လုပ်ရန်—

```text
1. Agent ဘာလုပ်ဖို့လိုလဲရှင်းပါ

3. Broad allow မလုပ်ပါနဲ့
4. Specific ask rule ထည့်ပါ
```

Prompt—

```text
Permission policy ကြောင့် task မပြီးနိုင်ပါ။

လုပ်ရန်:
- ဘာ permission လိုနေလဲရှင်းပါ
- allow/ask/deny ဘယ် policy သင့်လဲအကြံပြုပါ
- exact rule ပေးပါ
- broad wildcard မသုံးပါနဲ့
```

# 27. Chapter Summary

ဒီအခန်းမှာ **Safe Agent Permissions** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Agent permission = Agent ဘာလုပ်ခွင့်ရှိလဲ သတ်မှတ်တဲ့ boundary
```

အရေးကြီးဆုံး policy—

```text
Allow = low-risk read-only
Ask = project state ပြောင်းနိုင်သော action
Deny = secret/destructive/high-risk action
```

အရေးကြီးဆုံးသတိ—

```text
Prompt instruction / CLAUDE.md = soft guidance
permissions = hard gate
hooks = deterministic enforcement layer
```

Beginner policy—

```text
Read-only first
Plan before edit
Ask before file changes
Ask before package install
Deny secrets/destructive commands
No deployment without approval
No bypass/danger-full-access on main workspace
```

Agentic Coding အတွက် takeaway—

```text
Agent ကို powerful လုပ်ချင်ရင် permission boundary တည်ဆောက်ပါ။
Boundary မရှိတဲ့ agent က productivity tool မဟုတ်ဘဲ risk ဖြစ်လာနိုင်တယ်။
```
