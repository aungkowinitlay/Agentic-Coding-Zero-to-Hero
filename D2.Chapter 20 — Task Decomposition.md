# Chapter 20 — Task Decomposition

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့အသုံးချနိုင်ရပါမယ်။

```text
[ ] Task Decomposition ဆိုတာဘာလဲ ရှင်းပြနိုင်မယ်
[ ] Feature ကြီးတစ်ခုကို task သေးသေးလေးတွေခွဲနိုင်မယ်
[ ] Feature, task, step, checklist ကွာခြားချက်ကိုနားလည်မယ်
[ ] MVP scope နဲ့ Later scope ကိုခွဲနိုင်မယ်
[ ] Vertical slice နဲ့ horizontal slice ကိုခွဲနိုင်မယ်
[ ] Frontend / Backend / Database / Test / Verification task တွေခွဲနိုင်မယ်
[ ] Dependency order ကိုစဉ်းစားနိုင်မယ်
[ ] Agent တစ်ယောက်ကို task ကြီးလွန်းတာ မခိုင်းမိအောင်ထိန်းနိုင်မယ်
[ ] Task card တစ်ခုရေးနိုင်မယ်
[ ] Agent ကို task decomposition ခိုင်းမယ့် prompt ရေးနိုင်မယ်
[ ] Task တစ်ခုချင်းစီအတွက် acceptance criteria နဲ့ verification ထည့်နိုင်မယ်
[ ] “small enough to verify” ဆိုတဲ့ mindset ကိုနားလည်မယ်
```

Chapter 18 မှာ Prompt Anatomy ကိုသင်ခဲ့တယ်။ Chapter 19 မှာ Plan Before Code ကိုသင်ခဲ့တယ်။ ဒီ Chapter 20 မှာတော့ feature ကြီးတစ်ခုကို Agent မလမ်းလွဲအောင် **task သေးသေးလေးတွေအဖြစ် ခွဲနည်း** ကိုသင်မယ်။

## 2. Beginner Explanation

Task Decomposition ဆိုတာ—

```text
အလုပ်ကြီးတစ်ခုကို နားလည်လွယ်၊ လုပ်လွယ်၊ စစ်လွယ်တဲ့ အလုပ်သေးသေးလေးတွေအဖြစ် ခွဲခြင်း
```

ပါ။

ဥပမာ—

```text
Event Management Platform ဆောက်ပါ။
```

ဆိုတာ task မဟုတ်ပါဘူး။ အရမ်းကြီးတယ်။ Agent ကိုဒီလိုခိုင်းရင် scope ပွသွားမယ်။

ပိုကောင်းတာက—

```text
1. Event list page ဆောက်
2. Event detail page ဆောက်
3. Registration form ဆောက်
4. Registration validation ထည့်
5. Ticket confirmation page ဆောက်
6. Backend events API ဆောက်
7. Registration API ဆောက်
8. Admin attendee list ဆောက်
9. PWA check-in screen ဆောက်
10. Tests / verification လုပ်
```

ဆိုပြီးခွဲတာပါ။

အလွယ်ဆုံး analogy နဲ့ပြောရရင်—

```text
“အိမ်ဆောက်ပါ” ဆိုတာ feature ကြီး
“အုတ်မြစ်ချပါ” ဆိုတာ task
“ဘယ်နေရာမှာတူးမလဲတိုင်းပါ” ဆိုတာ step
“အုတ်မြစ် level မှန်လားစစ်ပါ” ဆိုတာ verification
```

Agentic Coding မှာ task decomposition မရှိရင် Agent က—

```text
အလုပ်ကိုကြီးလွန်းအောင်ယူမယ်
file အများကြီးပြင်မယ်
test မစစ်နိုင်တော့ဘူး
scope ကျော်မယ်
bug ဖြစ်ရင် root cause ရှာရခက်မယ်
```

Task decomposition ရှိရင်—

```text
Agent ကို task တစ်ခုချင်းစီခိုင်းနိုင်မယ်
Git commit တစ်ခုချင်းစီသိမ်းနိုင်မယ်
Bug ဖြစ်ရင် rollback လုပ်ရလွယ်မယ်
Verifier/Auditor စစ်ရလွယ်မယ်
```

# 3. Feature, Task, Step, Checklist ကွာခြားချက်

Beginner တွေ အရင်ဆုံး ခွဲသိထားရမယ်။

## 3.1 Feature

Feature ဆိုတာ user မြင်နိုင်တဲ့ capability တစ်ခုပါ။

ဥပမာ—

```text
Event Registration
Registration History
Admin Attendee Management
PWA Check-in
Event Search / Filter
```

Feature က user outcome ကိုပြောတယ်။

```text
User can register for an event.
Staff can check in attendee.
Admin can view attendee list.
```

---

## 3.2 Task

Task ဆိုတာ feature ကိုအကောင်အထည်ဖော်ဖို့ လုပ်ရမယ့် အလုပ်ခွဲတစ်ခုပါ။

Feature—

```text
Event Registration
```

Tasks—

```text
- Registration form UI ဆောက်
- Form validation ထည့်
- Registration API ဆောက်
- Attendee database table ဆောက်
- Ticket code generation ထည့်
- Success page ဆောက်
- API integration test ရေး
```

Task က Agent ကိုခိုင်းလို့ရတဲ့ unit ဖြစ်သင့်တယ်။

---

## 3.3 Step

Step ဆိုတာ task တစ်ခုအတွင်းလုပ်ရမယ့် အဆင့်အသေးပါ။

Task—

```text
Registration form UI ဆောက်
```

Steps—

```text
1. RegisterPage file ဖန်တီး
2. Name input ထည့်
3. Phone input ထည့်
4. Email input ထည့်
5. Submit button ထည့်
6. Error message area ထည့်
7. Mobile layout စစ်
```

Step က implementation အဆင့်ကိုပြောတယ်။

---

## 3.4 Checklist

Checklist ဆိုတာ task ပြီး/မပြီးစစ်တဲ့ စာရင်းပါ။

Task—

```text
Registration form UI ဆောက်
```

Checklist—

```text
[ ] Name input ပါတယ်
[ ] Phone input ပါတယ်
[ ] Required validation ပါတယ်
[ ] Error message မြန်မာလိုပြတယ်
[ ] Mobile responsive ဖြစ်တယ်
[ ] Browser console error မရှိ
```

ဒီ course မှာ feature တစ်ခုကို—

```text
Feature → Tasks → Steps → Checklist → Verification
```

ဆိုပြီးခွဲသွားမယ်။

---

# 4. Task Decomposition ဘာကြောင့်အရေးကြီးလဲ

## 4.1 Agent ကို scope ထိန်းပေးတယ်

မကောင်းတဲ့ prompt—

```text
Event registration system အကုန်လုပ်ပါ။
```

Agent က frontend, backend, database, email, auth, admin, tests အကုန်ထည့်ချင်နိုင်တယ်။

ကောင်းတဲ့ task—

```text
Task 1:
Registration form UI ကို mock submit နဲ့ဆောက်ပါ။
Backend API မချိတ်သေးပါ။
```

Scope ကျဉ်းသွားတယ်။

---

## 4.2 Git commit လုပ်ရလွယ်တယ်

Task တစ်ခု = commit တစ်ခု ဖြစ်နိုင်တယ်။

```text
commit 1: add registration form UI
commit 2: add registration validation
commit 3: add registration success page
commit 4: add registration API
commit 5: connect registration form to API
```

Bug ဖြစ်ရင် ဘယ် commit ကစဖြစ်တာလဲရှာရလွယ်တယ်။

---

## 4.3 Verification လုပ်ရလွယ်တယ်

Task ကြီးလွန်းရင် verification မရှင်းတော့ဘူး။

မကောင်း—

```text
Event platform အကုန်စစ်ပါ။
```

ကောင်း—

```text
Registration validation task ကိုစစ်ပါ:
- Empty name error
- Empty phone error
- Valid submit success
```

---

## 4.4 Agent output ကို review လုပ်ရလွယ်တယ်

Task သေးရင် diff သေးတယ်။ Diff သေးရင် review လုပ်ရလွယ်တယ်။

```text
Small task → small diff → easier review → safer commit
```

---

## 4.5 Multi-agent workflow အတွက်အခြေခံဖြစ်တယ်

Level 5 မှာ Planner, Frontend Developer, Backend Developer, Verifier, Auditor စတဲ့ agents တွေခွဲမယ်။ Task decomposition မကောင်းရင် subagent orchestration မကောင်းနိုင်ဘူး။

---

# 5. Good Task ဆိုတာဘာလဲ

Good task တစ်ခုမှာ အောက်ပါအချက်တွေရှိသင့်တယ်။

```text
[ ] Goal တစ်ခုတည်းရှိ
[ ] Scope ရှင်း
[ ] Non-goals ရှင်း
[ ] Files / layer ခန့်မှန်းနိုင်
[ ] Acceptance criteria ရှိ
[ ] Verification ရှိ
[ ] Git diff review လုပ်နိုင်လောက်အောင်သေး
[ ] Agent တစ်ယောက်က plan → implement → verify လုပ်နိုင်လောက်အောင်ရှင်း
```

## Good task example

```text
Task:
Registration form validation messages ကို မြန်မာလိုပြင်ပါ။

Scope:
Frontend only. app.js file ထဲ validation messages ပဲပြင်ပါ။

Acceptance Criteria:
- Empty name shows “နာမည်ဖြည့်ပါ။”
- Empty phone shows “ဖုန်းနံပါတ်ဖြည့်ပါ။”
- Valid submit behavior မပျက်ရဘူး။

Verification:
Manual browser test ၃ ခု။
```

## Bad task example

```text
Registration system အကုန်လုပ်ပါ။
```

ဘာကြောင့်မကောင်းလဲ—

```text
- Scope ကြီးလွန်း
- Frontend/backend/database အကုန်ရော
- Acceptance criteria မရှိ
- Verification မရှိ
- Git diff review ခက်
```

---

# 6. Task Size Rule

Task တစ်ခုက အောက်ပါအတိုင်းဖြစ်သင့်တယ်။

```text
One task should be small enough to:
- explain in 1 paragraph
- implement in a few related files
- verify with clear checks
- review in git diff
- commit with one meaningful commit message
```

Task ကြီးလွန်းတဲ့လက္ခဏာ—

```text
[ ] “and” များလွန်း
[ ] frontend + backend + database + deployment အကုန်ပါ
[ ] package install ပါနိုင်
[ ] acceptance criteria ၁၅ ခုထက်များ
[ ] file ၁၀ ခုကျော်ပြင်နိုင်
[ ] verification plan မရှင်း
[ ] Agent က clarify questions အများကြီးမေးရ
```

Task သေးလွန်းတဲ့လက္ခဏာ—

```text
[ ] line တစ်ကြောင်းစီကို commit တစ်ခုလုပ်နေ
[ ] feature flow မမြင်ရ
[ ] context switching များ
[ ] Agent overhead များ
```

Balance လိုတယ်။

# 7. Decomposition Dimensions

Feature ကြီးတစ်ခုကို ခွဲတဲ့နည်း အမျိုးမျိုးရှိတယ်။

```text
1. By user journey
2. By layer
3. By file/folder
4. By risk
5. By dependency order
6. By agent role
7. By verification type
```

---

## 7.1 By User Journey

User action flow အလိုက်ခွဲတာပါ။

Feature—

```text
Event Registration
```

User journey—

```text
1. User sees event detail
2. User opens registration form
3. User enters name/phone/email
4. User submits form
5. User sees validation errors or success
6. User receives ticket code
7. Admin sees attendee
```

Tasks—

```text
- Event detail registration CTA
- Registration form UI
- Form validation
- Ticket confirmation UI
- Admin attendee visibility
```

ဒီနည်းက product thinking အတွက်ကောင်းတယ်။

---

## 7.2 By Layer

Frontend, backend, database, test အလိုက်ခွဲတာပါ။

Feature—

```text
Event Registration
```

Layer tasks—

```text
Frontend:
- Registration form UI
- Validation messages
- Success page

Backend:
- POST /api/events/:id/register
- Duplicate registration prevention
- Ticket code generation

Database:
- attendees table
- unique constraint

Tests:
- validation tests
- API integration tests
- E2E registration flow
```

ဒီနည်းက full-stack project အတွက်ကောင်းတယ်။

---

## 7.3 By File / Folder

Project structure အလိုက်ခွဲတာပါ။

```text
apps/web/
services/api/
packages/shared/
apps/admin/
apps/checkin-pwa/
```

Example—

```text
Task 1:
packages/shared ထဲ registration validator ဆောက်

Task 2:
services/api ထဲ registration endpoint ဆောက်

Task 3:
apps/web ထဲ registration form ချိတ်
```

ဒီနည်းက monorepo project တွေမှာအသုံးဝင်တယ်။

---

## 7.4 By Risk

အန္တရာယ်/မသေချာမှုများတဲ့အပိုင်းကို အရင်သီးခြားခွဲတာပါ။

Risky parts—

```text
- Authentication
- Authorization
- Payment
- Database migration
- Secret handling
- Offline sync conflict
- External API integration
```

ဥပမာ PWA Offline Check-in feature မှာ—

```text
Task 1:
Offline queue data model design only

Task 2:
Local queue save/load implementation

Task 3:
Online sync simulation

Task 4:
Conflict handling plan

Task 5:
Verifier/Auditor review
```

Risky part ကို task သီးခြားခွဲရင် review လုပ်ရလွယ်တယ်။

---

## 7.5 By Dependency Order

ဘယ် task က ဘယ် task ကိုမှီခိုလဲစဉ်းစားတာပါ။

ဥပမာ—

```text
Registration form UI
→ Validation
→ Backend API
→ API integration
→ Success page
→ E2E test
```

Backend API မရှိသေးခင် frontend ကို mock data နဲ့စနိုင်တယ်။ ဒါပေမယ့် real integration မလုပ်ခင် API contract လိုတယ်။

Dependency map—

```text
SPEC
 ↓
API Contract
 ↓
Backend Endpoint
 ↓
Frontend Integration
 ↓
E2E Test
```

# 8. Vertical Slice vs Horizontal Slice

Task decomposition မှာ အရေးကြီးတဲ့ concept ၂ ခုရှိတယ်။

```text
Vertical Slice
Horizontal Slice
```

---

## 8.1 Horizontal Slice

Layer တစ်ခုလုံးကို အရင်ဆောက်တာပါ။

ဥပမာ—

```text
Frontend UI အားလုံးအရင်
Backend API အားလုံးနောက်မှ
Database နောက်မှ
```

Advantages—

```text
- Layer တစ်ခုချင်းစီ focus လုပ်ရလွယ်
- Beginner အတွက် UI-only စနိုင်
```

Disadvantages—

```text
- End-to-end flow နောက်ကျမှအလုပ်လုပ်
- Integration issue နောက်ကျမှတွေ့
```

---

## 8.2 Vertical Slice

Feature flow တစ်ခုကို frontend → backend → database → test အထိသေးသေးလေး end-to-end ဆောက်တာပါ။

ဥပမာ—

```text
Event list:
Backend GET /api/events
Frontend event list
Basic test
```

Advantages—

```text
- End-to-end flow အစောပိုင်းစမ်းနိုင်
- Integration issue အစောပိုင်းတွေ့
- Production thinking ပိုကောင်း
```

Disadvantages—

```text
- Beginner အတွက် layer အများကြီးတပြိုင်နက်စဉ်းစားရ
- Scope ထိန်းရခက်နိုင်
```

---

## 8.3 Course Strategy

ဒီ course မှာ beginner stage မှာ horizontal slice နဲ့စမယ်။

```text
Frontend mock UI
→ Backend API
→ Integration
```

Full-stack stage မှာ vertical slice သို့ပြောင်းမယ်။

```text
Registration flow end-to-end
→ Admin visibility
→ PWA check-in
```

Rule—

```text
Beginner:
horizontal slice for learning

Builder:
vertical slice for product validation
```

# 9. MVP vs Later Scope

Task decomposition မှာ MVP နဲ့ Later ကိုမခွဲရင် feature ကြီးသွားမယ်။

Feature—

```text
Event Registration
```

MVP—

```text
- Registration form
- Required validation
- Ticket code
- Admin attendee list
```

Later—

```text
- Email confirmation
- SMS/Viber notification
- Payment
- QR scanner
- Reminder notification
```

Agent prompt မှာ အမြဲထည့်ပါ။

```text
MVP only.
Later scope items ကို implementation ထဲမထည့်ပါနဲ့။
လိုအပ်ရင် Later Scope section ထဲ note ပဲထည့်ပါ။
```

# 10. Example — Event Registration Decomposition

Feature ကြီး—

```text
Event Registration
```

## 10.1 Product decomposition

```text
1. User sees event detail
2. User opens registration form
3. User enters attendee info
4. System validates input
5. System creates registration
```

# 17. Task Splitting Patterns

## Pattern 1 — Docs first

```text
SPEC → Task Cards → Implementation
```

Use when—

```text
Feature behavior not fully defined
```

## Pattern 2 — UI mock first

```text
Mock UI → Validation → Backend integration
```

Use when—

```text
Beginner frontend learning
Backend not ready
```

## Pattern 3 — API contract first

```text
API contract → Backend → Frontend integration
```

Use when—

```text
Frontend/backend teams or agents work separately
```

## Pattern 4 — Test first

```text
Acceptance criteria → Tests → Implementation
```

Use when—

```text
Critical logic
Regression bugs
Validation/business rules
```

## Pattern 5 — Risk spike first

```text
Small prototype → decision → implementation
```

Use when—

```text
Offline sync, payment, auth, deployment uncertainty
```

Risk spike ဆိုတာ production feature မဆောက်ခင် “ဒီနည်းလမ်းအလုပ်လုပ်နိုင်လား” ကိုစမ်းတဲ့ small research/prototype task ပါ။

# 27. Chapter Summary

ဒီအခန်းမှာ **Task Decomposition** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Task Decomposition = အလုပ်ကြီးကို စစ်လို့ရ၊ commit လုပ်လို့ရ၊ Agent ကိုခိုင်းလို့ရတဲ့ အလုပ်သေးသေးလေးတွေအဖြစ် ခွဲခြင်း
```

အရေးကြီးဆုံး hierarchy—

```text
Feature
→ Task
→ Step
→ Checklist
→ Verification
```

Task ခွဲတဲ့နည်းများ—

```text
By user journey
By layer
By file/folder
By risk
By dependency order
By agent role
By verification type
```

Good task တစ်ခုမှာပါရမယ့်အရာ—

```text
Goal
Scope
Non-goals
Acceptance Criteria
Verification
Dependencies
Agent Role
```

Agentic Coding အတွက် takeaway—

```text
Agent ကို task ကြီးကြီးတစ်ခုပေးပြီး magic ဖြစ်မယ်လို့မမျှော်လင့်ပါနဲ့။
Task ကိုသေးအောင်ခွဲပါ။
Task တစ်ခုချင်းစီကိုစစ်လို့ရအောင်ရေးပါ။
ပြီးမှ Agent ကို plan → implement → verify workflow နဲ့ခိုင်းပါ။
```
