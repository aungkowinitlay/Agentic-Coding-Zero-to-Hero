## 2. Beginner Explanation

Subagent ကို အလွယ်ဆုံးနားလည်မယ်ဆိုရင်—

```text
Main Agent အောက်မှာ အလုပ်ခွဲယူလုပ်တဲ့ specialist assistant
```

ပါ။

ဥပမာ သင့် project မှာ Registration History feature တစ်ခုရေးပြီးပြီဆိုပါစို့။ Main Agent တစ်ယောက်တည်းကို—

```text
Code ရေးပါ။
Bug စစ်ပါ။
Security စစ်ပါ။
Tests စစ်ပါ။
Commit message ပေးပါ။
```

လို့အကုန်ခိုင်းနိုင်တယ်။ ဒါပေမယ့် အဲဒီလိုဆိုရင် role တွေရောသွားနိုင်တယ်။

ပိုကောင်းတဲ့ workflow—

```text
Main Agent
  ↓
Frontend Developer Subagent
  → UI / JS implementation

Verifier Subagent
  → Acceptance criteria PASS / FAIL စစ်

Auditor Subagent
  → Security / privacy / PII risk စစ်

Test Engineer Subagent
  → Test plan / test failures စစ်
```

ဒီလို role ခွဲရင်—

```text
Developer = ရေးသူ
Verifier = criteria စစ်သူ
Auditor = risk စစ်သူ
Human = final decision maker
```

ဆိုပြီးတာဝန်တွေရှင်းသွားတယ်။

Mental model—

```text
Skill
= ဘယ်လိုလုပ်မလဲဆိုတဲ့ reusable workflow

Command
= workflow ကို /command နဲ့ခေါ်တာ

Subagent
= အဲဒီ workflow ကိုလုပ်မယ့် specialist worker

Hook
= automatic safety checkpoint

MCP
= external tool/service access
```

# 3. Subagent ဆိုတာဘာလဲ

Subagent ဆိုတာ main conversation / main agent က spawn သို့မဟုတ် invoke လုပ်နိုင်တဲ့ specialized agent instance ပါ။

Claude Code docs က subagents တွေကို main agent က focused subtasks အတွက် spawn လုပ်နိုင်ပြီး, context isolation, parallelization, specialized instructions, tool restrictions စတဲ့အကျိုးကျေးဇူးများအတွက်သုံးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ Subagent တစ်ခုဟာ fresh conversation တစ်ခုအဖြစ် run ဖြစ်ပြီး intermediate tool calls/results တွေကို subagent context ထဲမှာပဲထားကာ main conversation ဆီ final summary ကိုသာပြန်ပေးနိုင်ပါတယ်။ ([Claude][2])

Subagent တစ်ခုမှာ အများအားဖြင့်—

```text
name
description
role / system prompt
allowed tools
disallowed tools
model
permission mode
skills
MCP access
output contract
```

တို့ရှိနိုင်တယ်။

အလွယ်မှတ်ရန်—

```text
Subagent = Role + Instructions + Tools + Context boundary
```

# 4. Main Agent vs Subagent

| အချက်                 | Main Agent                       | Subagent                                  |
| --------------------- | -------------------------------- | ----------------------------------------- |
| User တိုက်ရိုက်ပြောတာ | ဟုတ်                             | များသောအားဖြင့် main agent က invoke       |
| Context               | Main conversation history အပြည့် | Own fresh context window                  |
| Role                  | Coordinator / general worker     | Specialist                                |
| Tool access           | Session-wide permissions         | Role-specific tools သတ်မှတ်နိုင်          |
| Best for              | User goal coordinate             | Focused task / review / exploration       |
| Output                | User-facing final answer         | Summary / report back to parent           |
| Risk                  | Context bloat / role mixing      | Delegation overhead / cost / coordination |

Claude Code docs အရ subagent context window က fresh ဖြစ်ပေမယ့် empty မဟုတ်ဘဲ subagent ရဲ့ system prompt, Agent tool prompt, project CLAUDE.md, tool definitions, preloaded skills စတာတွေပါနိုင်ပြီး parent conversation history သို့မဟုတ် parent tool results မပါဝင်နိုင်တာကြောင့် subagent ကိုလိုအပ်တဲ့ file paths, error messages, decisions တွေကို prompt ထဲ explicit ထည့်ပေးရပါတယ်။ ([Claude][2])

Course rule—

```text
Subagent ကို “သိထားမယ်ထင်တာ” မလုပ်ပါနဲ့။
လိုအပ်တဲ့ context ကို handoff prompt ထဲ ထည့်ပေးပါ။
```

# 5. Subagent ဘာကြောင့်လိုလဲ

## 5.1 Context Isolation

Main session ထဲမှာ file/log/search output အများကြီးဝင်လာရင် context bloat ဖြစ်တယ်။ Subagent ကို read-only exploration ခိုင်းရင် အဲဒီ file reads/search results တွေ subagent context ထဲမှာပဲနေပြီး main conversation ဆီ summary ပဲပြန်လာနိုင်တယ်။

```text
Bad:
Main Agent reads 40 files → context bloat

Good:
Explorer Subagent reads 40 files → main receives 10-line summary
```

## 5.2 Role Specialization

Reviewer, Auditor, Test Engineer, Planner စတာတွေကို system prompt သီးခြားထားနိုင်တယ်။

```text
Verifier:
Code မပြင်ဘူး။ Acceptance criteria PASS/FAIL စစ်တယ်။

Auditor:
Code မပြင်ဘူး။ Security/privacy risk စစ်တယ်။

Frontend Developer:
Frontend files ပြင်နိုင်တယ်။ Backend/database မထိဘူး။
```

## 5.3 Tool Restriction

Subagent တစ်ယောက်ကို Read-only tools ပဲပေးနိုင်တယ်။ Claude docs မှာ subagents ကို `tools` field နဲ့ tool allowlist အဖြစ်ကန့်သတ်နိုင်ပြီး `Read`, `Grep`, `Glob` သာပေးထားတဲ့ doc reviewer သို့ architecture analyst က code ကိုစစ်နိုင်သော်လည်း file modify မလုပ်နိုင်ကြောင်းဥပမာပြထားပါတယ်။ ([Claude][2])

## 5.4 Parallel Review

Security, test coverage, accessibility, maintainability review တွေကို parallel လုပ်နိုင်တယ်။

```text
Spawn:
- security-auditor
- test-reviewer
- accessibility-reviewer
- maintainability-reviewer

Then consolidate results.
```

Codex docs က subagent workflows တွေကို parallel specialized agents spawn လုပ်ပြီး results တွေကို consolidated response တစ်ခုအဖြစ်ပြန်စုနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][3])

## 5.5 Quality Gate Separation

Implementation agent ကိုယ်တိုင် review လုပ်တာထက် independent reviewer / verifier / auditor ကိုခွဲတာပိုကောင်းတယ်။

```text
Developer writes
Verifier checks
Auditor reviews risk
Human approves
```

# 6. Subagent မသုံးသင့်တဲ့အချိန်

Subagent သည် powerful ဖြစ်ပေမယ့် task တိုင်းအတွက်မလိုပါ။

မသုံးသင့်တဲ့အချိန်—

```text
[ ] Typo တစ်လုံးပြင်
[ ] README line တစ်ကြောင်းပြင်
[ ] CSS class text တစ်ခုပြင်
[ ] Task context အလွန်သေး
[ ] Cost/token budget အရေးကြီး
[ ] Workflow မရှင်းသေး
[ ] Subagent output ကို review မလုပ်နိုင်
[ ] Same file ကို agents အများကြီးပြင်ဖို့ဖြစ်နေ
```

Codex docs က subagent workflows တွေသည် comparable single-agent runs ထက် token ပိုသုံးနိုင်ကြောင်း, Codex သည် subagents ကို user က explicit ask လုပ်မှ spawn လုပ်ကြောင်းဖော်ပြထားပါတယ်။ ([OpenAI Developers][3])

Course rule—

```text
Small task → main agent / skill / command
Large exploration → subagent
Independent review → subagent
Full-stack multi-role workflow → subagents
```

# 7. Subagent vs Skill vs Command vs Hook vs MCP

| Concept  | Purpose                   | Example              | Store                            |
| -------- | ------------------------- | -------------------- | -------------------------------- |
| Prompt   | One-time task instruction | “Plan this feature”  | chat                             |
| Skill    | Reusable workflow         | diff-review skill    | `.claude/skills/`                |
| Command  | Slash invocation          | `/diff-review`       | built-in or skill                |
| Subagent | Specialist worker role    | verifier, auditor    | `.claude/agents/`                |
| Hook     | Automatic lifecycle guard | block `rm -rf`       | `.claude/settings.json` + script |
| MCP      | External tool access      | GitHub, DB, DO tools | MCP config                       |

Rule—

```text
Skill = how
Subagent = who
Command = quick invoke
Hook = automatic guard
MCP = external capability
```

ဥပမာ—

```text
Verifier Subagent uses verify-criteria Skill.
User invokes /verify-feature Command.
PreToolUse Hook blocks unsafe command.
MCP provides GitHub PR data.
```

# 8. Claude Code Built-in Subagents

Claude Code မှာ built-in subagents တွေရှိနိုင်ပါတယ်။ Docs က Explore, Plan, General-purpose စတဲ့ built-in agents တွေကိုဖော်ပြထားပြီး Explore သည် read-only codebase exploration အတွက်, Plan သည် plan mode ထဲမှာ codebase research အတွက်, general-purpose သည် complex multi-step operations / modifications အတွက်သုံးနိုင်ကြောင်းရှင်းပြထားပါတယ်။ Built-in subagents တွေဟာ parent permissions ကို inherit လုပ်သော်လည်း additional tool restrictions ရှိနိုင်ပါတယ်။ ([Claude][1])

Beginner mental model—

```text
Explore
= read-only code search / architecture exploration

Plan
= planning research in separate context

General-purpose
= complex multi-step worker
```

သတိ—

```text
Built-in agents ရှိတယ်ဆိုတာနဲ့ အလိုအလျောက်အမြဲသုံးသင့်တယ်မဟုတ်ပါ။
High-stakes review / audit အတွက် custom role-specific subagent ရေးတာပိုကောင်းနိုင်တယ်။
```

# 9. Claude Code Custom Subagent File Structure

Claude Code custom subagent ကို Markdown file တစ်ခုအဖြစ်ရေးနိုင်တယ်။

Location—

```text
.claude/agents/verifier.md
```

Basic structure—

```md
---
name: verifier
description: Verify implemented features against acceptance criteria. Use after implementation when the user asks whether a feature is done or requests PASS/FAIL verification.
tools: Read, Grep, Glob
model: sonnet
---

You are the Verifier.

Responsibilities:
- Read acceptance criteria and relevant implementation evidence.
- Check each criterion as PASS, WARNING, or FAIL.
- Do not edit code.
- Do not install packages.
- Do not commit.

Output:
| Criteria | Status | Evidence | Notes |
```

Claude Code docs အရ subagent files တွေသည် YAML frontmatter + Markdown body ပုံစံဖြစ်ပြီး `name` နှင့် `description` fields required ဖြစ်ပါတယ်။ `tools`, `disallowedTools`, `model`, `permissionMode`, `skills`, `mcpServers`, `hooks`, `memory`, `background`, `effort`, `isolation`, `color` စတဲ့ optional frontmatter fields တွေလည်းသုံးနိုင်ပါတယ်။ ([Claude][1])

# 10. Subagent Locations & Scope

Claude Code subagents ကို scope အလိုက်နေရာအမျိုးမျိုးထားနိုင်တယ်။

```text
.claude/agents/
= current project-specific subagents

~/.claude/agents/
= user-level personal subagents, all projects

plugin agents/
= plugin-provided subagents

--agents
= current session only, CLI-defined
```

Claude docs အရ project subagents ကို `.claude/agents/` ထဲထားပြီး version control ထဲ commit လုပ်နိုင်တာကြောင့် team/project shared agents အဖြစ်အသုံးဝင်ပါတယ်။ User-level subagents ကို `~/.claude/agents/` ထဲထားရင် projects အားလုံးမှာသုံးနိုင်ပါတယ်။ Same name conflict ဖြစ်ရင် scope priority ရှိပြီး project-level nested directories တွေမှာ current working directory နီးတဲ့ definition က active ဖြစ်နိုင်ပါတယ်။ ([Claude][1])

Course recommendation—

```text
Course/project-specific agents
→ .claude/agents/

Personal reusable agents
→ ~/.claude/agents/

Temporary experiment
→ prompt or --agents, not committed

Codex custom agents
→ .codex/agents/ or ~/.codex/agents/
```

# 11. Subagent Frontmatter Fields

Beginner အတွက် အရေးကြီးဆုံး fields—

```yaml
---
name: verifier
description: Verify implemented features against acceptance criteria.
tools: Read, Grep, Glob
model: sonnet
permissionMode: default
---
```

## 11.1 `name`

Subagent identifier.

```yaml
name: security-auditor
```

Rules—

```text
lowercase
hyphen-separated
unique
specific
```

## 11.2 `description`

Claude ဘယ်အချိန် subagent ကိုသုံးရမလဲဆုံးဖြတ်ဖို့ အရေးကြီးဆုံး field.

Bad—

```yaml
description: Helps with code.
```

Good—

```yaml
description: Review code and git diffs for security, privacy, secret exposure, PII logging, and dependency risk. Use after implementation or before commit/deploy.
```

Claude docs က subagent delegation matching ကို `description` field ပေါ်မူတည်နိုင်တာကြောင့် clear, specific description ရေးဖို့အကြံပြုထားပါတယ်။ ([Claude][2])

## 11.3 `tools`

Subagent သုံးနိုင်တဲ့ tools.

```yaml
tools: Read, Grep, Glob
```

Read-only reviewer အတွက်—

```yaml
tools: Read, Grep, Glob
```

Test runner အတွက်—

```yaml
tools: Read, Grep, Glob, Bash
```

Developer အတွက်—

```yaml
tools: Read, Grep, Glob, Edit, Write, Bash
```

Beginner rule—

```text
Reviewer / Verifier / Auditor = Read-only
Developer = Scoped edit tools, when needed
Deployer = planning-only until deployment chapter
```

## 11.4 `model`

Subagent model.

```yaml
model: sonnet
```

Docs အရ model alias တွေဖြစ်တဲ့ `sonnet`, `opus`, `haiku`, `fable`, `inherit` သို့ full model ID သုံးနိုင်ပြီး omitted ဖြစ်ရင် main conversation model ကို inherit လုပ်နိုင်ပါတယ်။ ([Claude][1])

Course rule—

```text
Review/audit high-stakes → stronger model, if available
Routine exploration → cheaper/faster model, if appropriate
Default beginner → inherit or sonnet
```

## 11.5 `permissionMode`

Permission behavior.

```yaml
permissionMode: default
```

High-risk agent တွေအတွက် plan/default/manual style ထားပါ။ `bypassPermissions` ကို beginner မသုံးသင့်ပါ။

## 11.6 `skills`

Subagent startup မှာ preload လုပ်မယ့် skills.

```yaml
skills:
  - diff-review
  - context-pack
```

Docs အရ `skills` field ထဲထည့်ထားတဲ့ skills တွေကို subagent startup context ထဲ preload လုပ်နိုင်ပြီး unlisted skills တွေကိုလည်း Skill tool မှတစ်ဆင့် invoke လုပ်နိုင်ပါတယ်။ ([Claude][1])

## 11.7 `disallowedTools`

Inherited tools ထဲကဖယ်ချင်တာ.

```yaml
disallowedTools: Write, Edit
```

## 11.8 `mcpServers`

Subagent-specific MCP access. # 15. Automatic vs Explicit Invocation

Subagent description နဲ့ task ကိုက်ရင် Claude က automatic delegate လုပ်နိုင်တယ်။

User says—


Specific subagent ကိုသုံးစေချင်ရင် name နဲ့ပြောပါ။


# 24. Codex Custom Agents

Codex subagents work differently from Claude Code file-based Markdown agents. Codex custom agents are standalone TOML files under `~/.codex/agents/` for personal agents or `.codex/agents/` for project-scoped agents. Official Codex docs state that every custom agent file must define `name`, `description`, and `developer_instructions`; optional fields can include model settings, sandbox mode, MCP servers, skills config, and other config keys. Codex also ships built-in agents such as `default`, `worker`, and `explorer`. ([OpenAI Developers][3])

Example Codex custom agent—

```text
.codex/agents/verifier.toml
```

```toml
name = "verifier"
description = "Verify implemented features against acceptance criteria. Use after implementation and before commit."
developer_instructions = """
You are the Verifier.

Responsibilities:
- Check acceptance criteria.
- Report PASS, WARNING, or FAIL.
- Do not edit code.
- Do not install packages.
- Do not commit, push, or deploy.

Output:
| Criteria | Status | Evidence | Notes |
"""
sandbox_mode = "read-only"
```

Codex docs က subagents inherit current sandbox policy / approvals and live runtime overrides from the parent session when spawned, so parent session `/permissions` changes or sandbox choices can still affect child agents. ([OpenAI Developers][3])

Course rule—

```text
Claude Code subagent
→ .claude/agents/*.md

Codex custom agent
→ .codex/agents/*.toml

Shared concept
→ role, description, tool/sandbox limits, output contract
```

# 27. Subagent Safety Rules

```text
Rule 1 — Use least privilege tools.
Rule 2 — Review agents must be read-only.
Rule 3 — Auditor must not silently fix issues.
Rule 4 — Developer agents must receive allowed files.
Rule 5 — Do not let multiple agents edit the same file in parallel.
Rule 6 — Do not allow deployer to deploy by default.
Rule 7 — Do not let subagents read .env.
Rule 8 — Use explicit invocation for important reviews.
Rule 9 — Subagent output is evidence, not final truth.
Rule 10 — Human approval still required for commit/push/deploy.
```

Subagents can reduce risk by restricting tools, but they also add coordination overhead and can consume extra tokens/cost. Codex docs explicitly warn that subagent workflows consume more tokens than comparable single-agent runs. ([OpenAI Developers][3])

```md
You are the Auditor.

## Responsibilities

- Review for security and privacy risks.
- Check for secret exposure.
- Check for PII storage/logging risks.
- Check dependency and package-change risk.
```

# 32. Common Errors

## Error 1 — Subagent description vague ဖြစ်ခြင်း

Bad—

```yaml
description: Helps with code.
```

Good—

```yaml
description: Verify implemented features against acceptance criteria. Use after implementation, before commit, or when the user asks whether a feature is done.
```

## Error 2 — Reviewer agent ကို Edit/Write tools ပေးခြင်း

Verifier/Auditor agents ကို read-only ထားပါ။

```yaml
tools: Read, Grep, Glob
```

## Error 3 — Handoff context မပေးခြင်း

Subagent က parent conversation history မရနိုင်တာကြောင့် needed context ကို prompt ထဲထည့်ရမယ်။

Bad—

```text
Use verifier.
```

Good—

```text
Use verifier.
Feature, criteria, files, evidence, constraints ထည့်ပါ။
```

## Error 4 — Parallel agents ကို same file edit ခိုင်းခြင်း

Parallel write operations က conflict ဖြစ်နိုင်တယ်။

Rule—

```text
Parallel = read-only review / exploration
Sequential = write/edit implementation
```

## Error 5 — Subagent output format မသတ်မှတ်ခြင်း

Output format မပါရင် result မျိုးစုံဖြစ်နိုင်တယ်။

## Error 6 — Subagent ကို hard security boundary လို့ယူဆခြင်း

Subagent tool restriction ကအကူအညီဖြစ်ပေမယ့် permissions/hooks/human review တို့လိုဆက်လိုတယ်။

## Error 7 — Too many subagents

Subagents များလွန်းရင် cost/latency/coordination overhead တက်နိုင်တယ်။ Codex docs က subagent workflows token ပိုသုံးနိုင်ကြောင်းသတိပေးထားပါတယ်။ ([OpenAI Developers][3])

## 33.1 Subagent file မ load ဖြစ်ရင်

```text
စစ်ရန်:
- File path: .claude/agents/name.md
- Frontmatter valid YAML

- description present

```

Claude docs က `/doctor` သည် same-scope duplicate agent names ကို report လုပ်နိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

## 33.2 Subagent writes when read-only expected

Fix—

```yaml
tools: Read, Grep, Glob
```

or—

```yaml
disallowedTools: Edit, Write, Bash
```

## 33.3 Subagent uses wrong role

Fix description and system prompt—

```text
Use for acceptance criteria verification only.
Do not perform implementation.
```

## 33.4 Subagent output too vague

Add output contract—

```md
| Severity | Finding | Evidence | Recommendation |
```

## 33.5 Subagent lacks needed context

Fix handoff prompt—

```text
Include relevant files, criteria, error/evidence, constraints.
```

## 33.6 Subagent cost too high

Use only when justified—

```text
- read-only exploration
- independent review
- large logs/files
- high-stakes verification
```

For small edits, use main agent + skill.

# 36. Chapter Summary

ဒီအခန်းမှာ **Subagents** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Subagent = specialist worker with its own role, context, tools, and output contract
```

Subagent သုံးရတဲ့အကြောင်း—

```text
Context isolation
Parallel review
Specialized instructions
Tool restrictions
Quality gate separation
```

Claude Code မှာ—

```text
.claude/agents/<agent>.md
~/.claude/agents/<agent>.md
```

နဲ့ Markdown + YAML frontmatter subagents ရေးနိုင်တယ်။ `name` နဲ့ `description` required ဖြစ်ပြီး `tools`, `model`, `permissionMode`, `skills`, `mcpServers`, `hooks`, `memory`, `background`, `isolation` စတာတွေကို optional ထည့်နိုင်တယ်။ ([Claude][1])

Codex မှာ—

```text
.codex/agents/<agent>.toml
~/.codex/agents/<agent>.toml
```

နဲ့ custom agents ရေးနိုင်ပြီး `name`, `description`, `developer_instructions` required ဖြစ်တယ်။ Subagent workflows က parallel agents spawn လုပ်ပြီး results consolidate လုပ်နိုင်ပေမယ့် explicit ask လိုပြီး token/cost ပိုသုံးနိုင်တယ်။ ([OpenAI Developers][3])

အရေးကြီးဆုံး safety rule—

```text
Verifier / Auditor / Reviewer = read-only
Developer = scoped edit
Deployer = planning-only until explicit approval
Parallel = review/exploration
Sequential = implementation
Human = final approval
```

Agentic Coding အတွက် takeaway—

```text
Agent တစ်ယောက်တည်းကို အကုန်မခိုင်းပါနဲ့။
Role ခွဲပါ။
Context ခွဲပါ။
Tools ခွဲပါ။
Output contract ခွဲပါ။
ပြီးမှ Coordinator / Human က result ကိုစုစည်းဆုံးဖြတ်ပါ။
```
