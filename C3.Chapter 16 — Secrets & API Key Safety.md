# Chapter 16 — Secrets & API Key Safety

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့အသုံးချနိုင်ရပါမယ်။

```text
[ ] Secret ဆိုတာဘာလဲ ရှင်းပြနိုင်မယ်
[ ] API key, token, password, database URL, private key တို့ကိုခွဲနိုင်မယ်
[ ] .env နဲ့ .env.example ကွာခြားချက်ကိုနားလည်မယ်
[ ] Secret ကို source code / Git / chat ထဲမထည့်ရကြောင်းနားလည်မယ်
[ ] Frontend/browser/mobile app ထဲ API key မထည့်ရကြောင်းနားလည်မယ်
[ ] GitHub Secrets ကို ဘယ်အချိန်သုံးမလဲ သိမယ်
[ ] DigitalOcean App Platform environment variables concept ကိုနားလည်မယ်
[ ] Agent ကို .env မဖတ်/မပြင်အောင် rule ထားနိုင်မယ်
[ ] Secret accidentally committed ဖြစ်ရင် revoke / rotate / untrack / history cleanup workflow ကိုနားလည်မယ်
[ ] Secret scanning / push protection concept ကိုနားလည်မယ်
[ ] Secret safety checklist နဲ့ project ကိုစစ်နိုင်မယ်
```

ဒီအခန်းက Agentic Coding မှာ အရေးကြီးဆုံး safety chapter တွေထဲကတစ်ခုပါ။ Agent တစ်ယောက်က code ဖတ်နိုင်၊ terminal command run နိုင်၊ Git diff ကြည့်နိုင်လာတဲ့အခါ secret တွေ မပေါက်အောင် အစောပိုင်းကတည်းက workspace rule ထားရမယ်။

## 2. Beginner Explanation

Secret ဆိုတာ **အခြားသူမသိသင့်တဲ့ access information** ပါ။

ဥပမာ—

```text
API key
Database password
Database connection URL
GitHub token
DigitalOcean token
OpenAI API key
Claude / Anthropic API key
JWT secret
Session secret
Private SSH key
OAuth client secret
Payment provider secret key
```

Secret ကို အိမ်သော့နဲ့နှိုင်းယှဉ်ပါ။

```text
အိမ်လိပ်စာ = public info ဖြစ်နိုင်
အိမ်သော့ = secret
```

API endpoint URL တစ်ခုက public ဖြစ်နိုင်တယ်။ ဒါပေမယ့် API key က secret ဖြစ်တယ်။

```text
API_BASE_URL=https://api.example.com
ဒီဟာက public config ဖြစ်နိုင်တယ်။

OPENAI_API_KEY=sk-...
ဒီဟာက secret ဖြစ်တယ်။
```

OpenAI Help Center က API key ကို API ခေါ်ရန်သုံးတဲ့ access code အဖြစ်ဖော်ပြထားပြီး key ပေါက်သွားရင် unauthorized use, unexpected charges, account activity risk ဖြစ်နိုင်ကြောင်းသတိပေးထားပါတယ်။ OpenAI က API key ကို environment variables ထဲသိမ်းရန်၊ client-side environments ဖြစ်တဲ့ browser/mobile app ထဲမထည့်ရန်၊ repository ထဲ commit မလုပ်ရန်လည်းအကြံပြုထားပါတယ်။ ([OpenAI Help Center][1])

# 3. Secret ဆိုတာဘာလဲ

Secret ကို အောက်ပါအတိုင်းခွဲနိုင်တယ်။

## 3.1 API Key

Service တစ်ခုကိုအသုံးပြုခွင့်ပေးတဲ့ key ပါ။

```text
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
STRIPE_SECRET_KEY=
SENDGRID_API_KEY=
```

## 3.2 Access Token

Account / API / GitHub / cloud resource ကို access လုပ်ခွင့်ပေးတဲ့ token ပါ။

```text
GITHUB_TOKEN=
DIGITALOCEAN_TOKEN=
VERCEL_TOKEN=
```

## 3.3 Database URL

Database username, password, host, db name တွေပါနိုင်တဲ့ connection string ပါ။

```text
DATABASE_URL=postgres://user:password@host:5432/dbname
```

ဒီထဲမှာ password ပါနိုင်လို့ secret ဖြစ်တယ်။

## 3.4 Password / Session Secret

```text
ADMIN_PASSWORD=
SESSION_SECRET=
JWT_SECRET=
```

## 3.5 Private Key

```text
id_rsa
private.pem
service-account.json
```

Private key files တွေကို Git ထဲမထည့်ရ။ Agent context ထဲမထည့်ရ။

# 4. Secret မဟုတ်တဲ့ Config နဲ့ Secret Config

Config အားလုံးက secret မဟုတ်ပါဘူး။

## Public / Non-secret Config

```text
APP_NAME=Event Management
PUBLIC_SITE_URL=https://events.example.com
API_BASE_URL=https://api.example.com
NODE_ENV=development
```

## Secret Config

```text
DATABASE_URL=postgres://user:password@host/db
SESSION_SECRET=super-secret
OPENAI_API_KEY=sk-...
DIGITALOCEAN_TOKEN=dop_v1_...
```

Beginner rule—

```text
Value ကို public GitHub repo ထဲမြင်ရင် ပြဿနာဖြစ်နိုင်လား?
ဟုတ်တယ်ဆိုရင် secret ဖြစ်တယ်။
```

# 5. `.env` ဆိုတာဘာလဲ

`.env` file ဆိုတာ local development environment variables တွေသိမ်းတဲ့ file ပါ။

ဥပမာ `.env`—

```text
API_BASE_URL=http://localhost:3000
DATABASE_URL=postgres://local-user:local-password@localhost:5432/eventdb
SESSION_SECRET=local-secret-value
OPENAI_API_KEY=real-key-here
```

`.env` ထဲမှာ actual secret values ပါနိုင်တယ်။ ဒါကြောင့်—

```text
.env ကို Git commit မလုပ်ရ
.env ကို Agent chat ထဲ paste မလုပ်ရ
.env ကို screenshot မရိုက်ရ
.env ကို public issue/PR ထဲမထည့်ရ
```

`.gitignore` ထဲမှာ `.env` ကို ignore ထားရမယ်။

```text
.env
.env.local
.env.*.local
```

# 6. `.env.example` ဆိုတာဘာလဲ

`.env.example` က secret values မပါဘဲ required variable names ကိုပြတဲ့ template file ပါ။

ဥပမာ `.env.example`—

```text
API_BASE_URL=
DATABASE_URL=
SESSION_SECRET=
OPENAI_API_KEY=
DIGITALOCEAN_TOKEN=
```

ဒီ file ကို Git commit လုပ်နိုင်တယ်။ ဘာကြောင့်လဲဆိုရင် actual secret value မပါဘူး။

ကောင်းတဲ့ `.env.example`—

```text
API_BASE_URL=
DATABASE_URL=
SESSION_SECRET=
```

မကောင်းတဲ့ `.env.example`—

```text
DATABASE_URL=postgres://real-user:real-password@real-host/db
OPENAI_API_KEY=sk-real-key
```

Rule—

```text
.env = local/private actual values
.env.example = shared template, no real values
```

# 7. Secret ကို ဘယ်မှာသိမ်းမလဲ

Secret ကို context အလိုက်သိမ်းမယ့်နေရာကွာတယ်။

| Context                   | Secret သိမ်းမယ့်နေရာ                             |
| ------------------------- | ------------------------------------------------ |
| Local development         | `.env` / OS environment variables                |
| GitHub Actions            | GitHub Secrets                                   |
| DigitalOcean App Platform | Environment variables, encrypted values          |
| Production backend server | Cloud provider secret/environment variable store |
| Frontend browser app      | မသိမ်းရ                                          |
| Mobile app binary         | မသိမ်းရ                                          |

GitHub Actions မှာ secrets တွေကို repository, environment, organization level မှာဖန်တီးသိမ်းနိုင်ပြီး workflow က secret ကို explicit include လုပ်မှအသုံးပြုနိုင်ပါတယ်။ Repository secret ထည့်ချင်ရင် repository Settings → Secrets and variables → Actions → New repository secret workflow ကိုသုံးနိုင်သလို GitHub CLI `gh secret set` နဲ့လည်းထည့်နိုင်ပါတယ်။ ([GitHub Docs][2])

DigitalOcean App Platform မှာ environment variables တွေကို app/component level မှာ build time သို့မဟုတ် runtime အတွက်သတ်မှတ်နိုင်ပြီး Encrypt option ကိုရွေးရင် variable values တွေကို build, deployment, application logs ထဲမှာ hide လုပ်နိုင်ကြောင်း DigitalOcean docs ကဖော်ပြထားပါတယ်။ ([DigitalOcean Docs][3])

# 8. Frontend ထဲ Secret မထည့်ရ

React / Vite / browser app ထဲမှာ secret API key မထည့်ရပါ။

မကောင်း—

```js
const OPENAI_API_KEY = "sk-real-key";
```

မကောင်း—

```text
VITE_OPENAI_API_KEY=sk-real-key
```

Vite မှာ `VITE_` နဲ့စတဲ့ environment variable တွေကို frontend bundle ထဲ expose ဖြစ်နိုင်တာကြောင့် **secret value မထည့်ရ**။ Frontend app က browser ထဲမှာ run ဖြစ်ပြီး user က DevTools / bundled JavaScript ထဲက values တွေကိုမြင်နိုင်တယ်။

OpenAI Help Center က OpenAI API key ကို browser/mobile app လို client-side environment ထဲ deploy မလုပ်ရန်၊ requests တွေကို backend server ကနေ route လုပ်ရန်၊ API key ကို environment variables ထဲသိမ်းရန်အကြံပြုထားပါတယ်။ ([OpenAI Help Center][1])

မှန်တဲ့ architecture—

```text
Frontend Browser
   ↓ request without API secret
Your Backend Server
   ↓ uses API key from environment variable
External API
```

မမှန်တဲ့ architecture—

```text
Frontend Browser
   ↓ includes API key
External API
```

Beginner rule—

```text
Secret key လိုတဲ့ API call ကို frontend ကတိုက်ရိုက်မခေါ်ပါနဲ့။
Backend route မှတဆင့်ခေါ်ပါ။
```

# 9. API Key Sharing မလုပ်ရ

Team project မှာ user တစ်ယောက်ရဲ့ API key ကို teammate တွေကိုပေးတာမကောင်းပါဘူး။ OpenAI Help Center က user-based API keys ကို team collaboration အတွက်မမျှဝေရန်၊ project-based API keys ကိုအသုံးပြုရန်၊ environment/staging/production အလိုက် keys ခွဲပြီး permissions, spend controls, usage visibility ထိန်းရန်အကြံပြုထားပါတယ်။ ([OpenAI Help Center][4])

Course rule—

```text
တစ်ယောက် key ကိုအဖွဲ့တစ်ခုလုံးမသုံးပါနဲ့။
Project/environment အလိုက် key ခွဲပါ။
```

ဥပမာ—

```text
OPENAI_API_KEY_LOCAL=
OPENAI_API_KEY_STAGING=
OPENAI_API_KEY_PRODUCTION=
```

Production key ကို local development ထဲမသုံးသင့်ပါ။

# 10. GitHub Secret Scanning နဲ့ Push Protection

Secret accidentally commit မလုပ်မိအောင် GitHub မှာ secret scanning နဲ့ push protection features တွေရှိတယ်။ GitHub docs က secret scanning သည် repository ရဲ့ Git history အားလုံးကို branches အားလုံးပေါ်မှာ scan လုပ်ပြီး API keys, passwords, tokens, known secret types စတဲ့ hardcoded credentials တွေကိုရှာနိုင်ကြောင်းဖော်ပြထားပါတယ်။ Secret leak alert ရရင် affected credential ကိုချက်ချင်း rotate လုပ်ရန်လည်း GitHub docs ကအကြံပြုထားပါတယ်။ ([GitHub Docs][5])

Push protection က push လုပ်ချိန်မှာ hardcoded secret တွေကို scan လုပ်ပြီး credential leaks တွေ history ထဲမဝင်ခင် တားနိုင်တဲ့ preventative defense ဖြစ်ကြောင်း GitHub docs ကဖော်ပြထားပါတယ်။ ([GitHub Docs][6])

Beginner အတွက် သတိ—

```text
Secret scanning ရှိလို့ secret commit လုပ်လို့ရတယ် မဟုတ်ပါ။
Secret scanning = last line of defense
Good habit = .env ignore + diff review + no secrets in code
```

# 11. Secret Accidentally Committed ဖြစ်ရင်

Secret တစ်ခါ Git history ထဲဝင်သွားရင် `.gitignore` ထည့်ရုံနဲ့မလုံလောက်နိုင်ပါ။

GitHub docs က sensitive data ကို repository history ထဲကနေဖယ်နိုင်ပေမယ့် collaborators အားလုံးနဲ့ coordinate လုပ်ရပြီး history rewriting side effects ရှိနိုင်ကြောင်းသတိပေးထားပါတယ်။ Secret ဖြစ်တဲ့ password/token/credential တစ်ခုဖြစ်ရင် ပထမဆုံး revoke သို့မဟုတ် rotate လုပ်ရမယ်၊ revoke/rotate လုပ်ပြီးသွားရင် history rewrite မလိုနိုင်တဲ့အခြေအနေတွေလည်းရှိနိုင်ကြောင်း GitHub docs ကဖော်ပြထားပါတယ်။ ([GitHub Docs][7])

Incident response order—

```text
1. Stop sharing
2. Revoke / rotate secret immediately
3. Remove from working tree
4. Add .env / secret file to .gitignore
5. git rm --cached if tracked
6. Commit cleanup
7. Check GitHub secret scanning alert
8. Decide whether history rewrite needed
9. Notify affected team/users if needed
10. Add prevention rules
```

အရေးကြီးဆုံး—

```text
Secret leaked ဖြစ်ရင် file ဖျက်တာထက် secret rotate/revoke က ပထမ။
```

သတိ—


# 13. Agent Secret Safety Rules

Agent ကို secret မပေါက်အောင် `CLAUDE.md`, `AGENTS.md`, `.claude/settings.json`, prompts, hooks တို့နဲ့ကာကွယ်မယ်။

## 13.1 CLAUDE.md / AGENTS.md Rule

```md
## Secret Safety

- Do not read or edit `.env` files.
- Do not print secret values.
- Do not ask the user to paste API keys into chat.
- Use `.env.example` for variable names only.
- Do not commit secrets.
- Do not log PII or secrets.
```

## 13.2 `.claude/settings.json` Deny Rules

```json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./*.pem)",
      "Read(./*.key)"
    ]
  }
}
```

## 13.3 Agent Prompt Rule

```text
.env files မဖတ်ပါနဲ့။
Secret values မဖော်ပြပါနဲ့။
လိုအပ်တဲ့ variable names ကို .env.example ထဲကနေသာကြည့်ပါ။
```

# 14. Secret Naming Convention

Environment variable names ကို consistent ဖြစ်အောင်ထားပါ။

```text
API_BASE_URL
DATABASE_URL
SESSION_SECRET
JWT_SECRET
OPENAI_API_KEY
ANTHROPIC_API_KEY
DIGITALOCEAN_TOKEN
GITHUB_TOKEN
```

Naming rule—

```text
UPPERCASE
WORDS_SEPARATED_BY_UNDERSCORE
VALUE ထဲ secret ထည့်
NAME ထဲ secret value မထည့်
```

မကောင်း—

```text
SK_REAL_KEY=...
MY_PASSWORD_IS_ABC=...
```

ကောင်း—

```text
OPENAI_API_KEY=
DATABASE_URL=
```

OpenAI docs က environment variable name ကို `OPENAI_API_KEY` အဖြစ် team တစ်လျှောက် consistent ထားရန်အကြံပြုထားပါတယ်။ ([OpenAI Help Center][1])

```bash

```

```powershell
Copy-Item .env.example .env
```


စစ်ရန်—

```bash
git status
```

Expected—

```text
.env should not appear
```

`.env` ပေါ်လာရင် `.gitignore` မမှန်နိုင်တယ်။

# 16. GitHub Actions Secret Workflow

CI/CD မှာ secret ကို workflow YAML ထဲ hard-code မထည့်ရ။

မကောင်း—

```yaml
env:
  OPENAI_API_KEY: sk-real-key
```

ကောင်း—

```yaml
env:
  OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

GitHub Actions secrets ကို repository Settings → Secrets and variables → Actions မှာထည့်နိုင်ပြီး `gh secret set SECRET_NAME` command နဲ့လည်းထည့်နိုင်ပါတယ်။ GitHub docs က workflows triggered from forked repositories, reusable workflows, Dependabot events စသည့်နေရာတချို့မှာ secrets မရနိုင်ကြောင်းလည်းသတိပေးထားပါတယ်။ ([GitHub Docs][2])

Beginner rule—

```text
GitHub Actions workflow ထဲ secret value ကိုတိုက်ရိုက်မရေးပါနဲ့။
${{ secrets.NAME }} ကိုသာသုံးပါ။
```

# 17. DigitalOcean Secret Workflow

# 20. Backend-only Secret Access Pattern

Secret လိုတဲ့ external API ကို backend ကသာခေါ်ရမယ်။

## Bad pattern

```text
Browser React app
  → uses OPENAI_API_KEY directly
  → external API
```

## Good pattern

```text
Browser React app
  → POST /api/ai-helper
Backend API
  → reads process.env.OPENAI_API_KEY
  → external API
```

Backend code concept—

```js
const apiKey = process.env.OPENAI_API_KEY;

if (!apiKey) {
  throw new Error("OPENAI_API_KEY is not configured");
}
```

သတိ—

```text
Error message ထဲ actual key value မထည့်ပါနဲ့။
```

# 21. Secret Rotation

Rotation ဆိုတာ key/token/password အသစ်ထုတ်ပြီး အဟောင်းကို revoke/delete လုပ်တာပါ။

Rotate လုပ်သင့်တဲ့အချိန်—

```text
Secret accidentally committed
Secret pasted into chat/log
Teammate leaves project
Public repository exposure
Unknown usage/spend spike
Provider security alert
Regular maintenance
```

Rotation workflow—

```text
1. New key create
2. Update local .env
3. Update GitHub Secrets / cloud env vars
4. Deploy/restart app if needed
5. Verify app works
6. Revoke old key
7. Monitor usage
8. Document incident
```

OpenAI account security docs မှာ API key leak က unauthorized users ကို API အသုံးပြုခွင့်ပေးနိုင်ပြီး unauthorized charges/violating activity risk ဖြစ်နိုင်ကြောင်းသတိပေးထားပါတယ်။ Reasonable spend limits, third-party tools သုံးရာမှာသတိထားခြင်း, environment variables သုံးခြင်းတို့ကိုလည်းအကြံပြုထားပါတယ်။ ([OpenAI Help Center][8])

# 30. Chapter Summary

ဒီအခန်းမှာ **Secrets & API Key Safety** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Secret = အခြားသူသိရင် access/cost/data risk ဖြစ်နိုင်တဲ့ value
```

အရေးကြီးဆုံး files—

```text
.env
= local actual secret values, Git commit မလုပ်ရ

.env.example
= variable names only, commit လုပ်နိုင်
```

အရေးကြီးဆုံး rules—

```text
Secret ကို source code ထဲမရေး
Secret ကို Git ထဲမ commit
Secret ကို chat/log/screenshot ထဲမထည့်
Frontend/browser/mobile app ထဲ API key မထည့်
Backend/server environment variables သုံး
GitHub Actions မှာ GitHub Secrets သုံး
Deployment မှာ cloud environment variables / encrypted secrets သုံး
Leaked secret ဖြစ်ရင် revoke/rotate first
```

Agentic Coding အတွက် takeaway—

```text
Agent ကို code ရေးခိုင်းတာထက် secret boundary တည်ဆောက်တာက ပိုအရေးကြီးနိုင်တယ်။
.env မဖတ်ပါနဲ့ဆိုတဲ့ instruction တစ်ခုတည်းမလုံလောက်ပါ။
.gitignore, permissions, prompts, hooks, GitHub Secrets, cloud env vars, rotation plan အားလုံးလိုတယ်။
```
