ဥပမာ Event Management Platform ဆောက်မယ်ဆိုပါစို့။ စဉ်းစားစရာတွေ အများကြီးရှိတယ်။


Mobile app ကို PWA လုပ်မလား? React Native လုပ်မလား?
Register ပြီးရင် email ပို့မလား? SMS ပို့မလား?
Agent ကို file တွေကိုယ်တိုင်ပြင်ခွင့်ပေးမလား?
Agent မှားပြင်သွားရင် ဘယ်လိုပြန်ဆယ်မလဲ?
Production deploy ကို Agent ကိုခိုင်းမလား?
MCP tool တွေက data ဖတ်ရုံပဲလား? Data ပြင်ခွင့်ပါပေးမလား?


ဒီဆုံးဖြတ်ချက်တွေကို မရေးထားရင် project ကြီးလာတဲ့အခါ Agent လည်းလမ်းလွဲမယ်၊ learner လည်းရှုပ်မယ်။ ဒါကြောင့် ဒီ course မှာ `COURSE_DECISIONS.md` ဆိုတဲ့ document တစ်ခုထားမယ်။

## 3. Key Concept — Decision Record ဆိုတာဘာလဲ

Decision Record ဆိုတာ project ထဲက အရေးကြီးဆုံးဖြတ်ချက်တွေကို မှတ်ထားတဲ့စာရွက်ပါ။

ဥပမာ—


## Decision
Mobile check-in app ကို PWA-first လုပ်မယ်။

## Reason
Learner တွေက React + Vite ကိုသင်ပြီးသားဖြစ်မယ်။
Native mobile stack အသစ်မတက်ဘဲ offline check-in concept ကိုသင်နိုင်မယ်။

## Consequence
MVP မှာ app store submission မပါ။
Later scope မှာ Expo native app ထည့်နိုင်မယ်။
*

Decision Record က “အမှန်တရားတစ်ခု” မဟုတ်ပါဘူး။ Project con* အတွက် “ယခုအချိန်မှာ သဘောတူထားတဲ့ဆုံးဖြတ်ချက်” ပါ။ နောက်ပိုင်း project ပြောင်းရင် update လုပ်နိုင်တယ်။ ဒါပေမယ့် update လုပ်တဲ့အခါ “ဘာကြောင့်ပြောင်းတာလဲ” ကိုပါရေးရမယ်။

# 4. Course Decisions

## Decision 1 — Mobile Stack ကို PWA-first လုပ်မယ်

ဒီ course ရဲ့ default mobile/check-in app ကို **PWA-first** အဖြစ်သတ်မှတ်မယ်။

**
Default:
React + Vite PWA

Optional Advanced Track:
Expo / React Native
*

PWA ကိုရွေးရတဲ့အကြောင်းရင်းက beginner တွေက Part 6 မှာ React + Vite ကိုသင်ပြီးသားဖြစ်မယ်။ ဒါကြောင့် check-in mobile app အတွက် stack အသစ်တစ်ခုလုံးထပ်မသင်ရဘူး။ Service worker, cache, IndexedDB, offline queue, background sync concept တွေကို web stack နဲ့ပဲသင်နိုင်မယ်။ MDN docs မှာ PWA တွေမှာ manifest, service worker, offline operation, cache, FetchEvent, IndexedDB, background synchronization စတဲ့ web platform APIs တွေသုံးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([MDN Web Docs][1])

Expo / React Native ကို မဖျက်ပါဘူး။ Native app store build, iOS/Android native feature, camera/QR scanner, push notification အဆင့်ကိုသွားချင်သူတွေအတွက် optional advanced track အဖြစ်ထားမယ်။ Expo docs က Android, iOS, web အတွက် universal app တည်ဆောက်နိုင်တဲ့ React Native-based tooling အဖြစ်ဖော်ပြထားပါတယ်။ ([Expo Documentation][2])

ဒီ decision ကြောင့် course ထဲမှာ—

**
MVP:
- apps/checkin-pwa

Later:
- apps/mobile-native
*

ဆိုပြီးခွဲမယ်။

- User က ticket code ကို copy/save လုပ်နိုင်
- Admin dashboard မှာ attendee data ပေါ်

Later:
- Email confirmation
- SMS confirmation
- Viber notification
- Telegram bot notification
- Reminder notification
*

ဒီ decision ရဲ့အကြောင်းရင်းက notification provider တွေထည့်လိုက်တာနဲ့—

**
API key

privacy concern
rate limit

*

တွေပါလာမယ်။ Beginner course MVP အတွက် scope ကြီးသွားမယ်။ ဒါကြောင့် ပထမ version မှာ “ticket code on-screen” ကိုသာလုပ်မယ်။

## Decision 3 — Plan Before Code ကို course rule အဖြစ်သတ်မှတ်မယ်

Agent ကို feature ကြီးတစ်ခုခိုင်းတဲ့အခါ code ချက်ချင်းမရေးခိုင်းပါဘူး။ အရင်ဆုံး plan တောင်းမယ်။

**
Rule:
Feature ကြီးတိုင်း code မရေးခင် plan ပြရမယ်။
*

Claude Code မှာ Plan Mode က Claude ကို source file မပြင်ဘဲ research လုပ်၊ command run လုပ်ပြီး plan တင်ပြစေတဲ့ permission mode ဖြစ်တယ်။ `Shift+Tab`, `/plan`, သို့မဟုတ် `claude --permission-mode plan` နဲ့ Plan Mode ဝင်နိုင်ပြီး plan approve မလုပ်မချင်း source file edit မလုပ်ဘူးလို့ docs မှာဖော်ပြထားပါတယ်။ ([Claude][3])

ဒီ course မှာ feature ကြီးတွေအတွက် default prompt က ဒီလိုဖြစ်မယ်။

**
Code မပြင်သေးပါနဲ့။
ပထမဆုံး plan ပြပါ။
ဘယ် files ဖတ်မလဲ၊ ဘယ် files ပြင်မလဲ၊ ဘာ tests run မလဲ ပြောပါ။
ကျွန်တော် approve လုပ်မှ implement လုပ်ပါ။
*

သေးငယ်တဲ့ typo fix လို task မျိုးမှာ Plan Mode မလိုနိုင်ပေမယ့် beginner course အတွက် အစပိုင်းမှာ plan-first workflow ကိုအမြဲသုံးမယ်။

## Decision 4 — Safety ကို Hooks နဲ့ deterministic enforce လုပ်မယ်

Instruction ပေးရုံနဲ့ မလုံလောက်ပါဘူး။ Agent က instruction ကိုမေ့နိုင်တယ်၊ con* ရှုပ်နိုင်တယ်၊ scope မှားနားလည်နိုင်တယ်။ ဒါကြောင့် safety rule တချို့ကို **Hooks** နဲ့ enforce လုပ်မယ်။

Claude Code hooks docs မှာ hooks တွေကို Claude Code lifecycle point တွေမှာ run သော user-defined shell commands အဖြစ်ဖော်ပြထားပြီး project rules enforce, repetitive tasks automate, external tools integrate လုပ်နိုင်စေတဲ့ deterministic control layer လို့ရှင်းပြထားပါတယ်။ Hooks တွေကို format after edit, block before command execution, notification, session con* injection စတဲ့ workflow တွေအတွက်သုံးနိုင်ပါတယ်။ ([Claude][4])

Course safety hooks examples—

**
.env file edit ကို block
rm -rf ကို approval မရှိဘဲ block
sudo command ကို block
curl | bash command ကို approval တောင်း
package install မလုပ်ခင် reason တောင်း
file edit ပြီးတိုင်း lint/check run
*

ဒီ decision ကြောင့် course မှာ safety ကို ၂ ဆင့်သင်မယ်။

**
Mindset:
Agent ကို မျက်စိမှိတ်မယုံပါနဲ့။

Mechanism:
Hooks နဲ့ rules ကို enforce လုပ်ပါ။
*

## Decision 5 — Recovery Model ကို Checkpoints + Git နှစ်မျိုးသုံးမယ်

Agentic Coding မှာ Agent မှားပြင်သွားနိုင်တယ်။ Beginner အတွက် “ပြန်ဆယ်နည်း” သင်တာ အလွန်အရေးကြီးတယ်။

ဒီ course မှာ recovery layer ကို ၂ ဆင့်ထားမယ်။

**
Fast session-level undo:
Claude Code Checkpoints / Rewind

Durable project history:
Git commits
*

Claude Code checkpointing docs မှာ Claude Code က Claude file edits တွေကို automatically track လုပ်ပြီး prompt တစ်ခုချင်းစီအတွက် checkpoint ဖန်တီးနိုင်ကြောင်း၊ `/rewind` သို့မဟုတ် empty prompt မှာ Esc နှစ်ချက်နဲ့ rewind menu ဖွင့်နိုင်ကြောင်း၊ code + conversation restore, conversation-only restore, code-only restore, summarize options တွေရှိကြောင်းဖော်ပြထားပါတယ်။ အရေးကြီးတဲ့ limitation တစ်ခုက checkpoint က version control replacement မဟုတ်ဘူးဆိုတာပါ။ ([Claude][5])

ဒီ course rule—

**
Feature မစခင် Git commit
Session အတွင်း mistake အသေးစားဆို Rewind
Feature-level rollback ဆို Git
Production release ဆို tag/release note
*

Git ကို save game လိုသုံးမယ်။ Checkpoint ကို session undo လိုသုံးမယ်။

## Decision 6 — Project Memory ကို CLAUDE.md + AGENTS.md နဲ့စတင်မယ်

Agentic Coding မှာ Agent ကို session တိုင်း project con* ပြန်ရှင်းမနေချင်ဘူး။ ဒါကြောင့် project memory files ထားမယ်။

**
CLAUDE.md
AGENTS.md
SPEC.md
*

Claude Code extension guide မှာ project convention သို့မဟုတ် command ကို Claude မှားနေပြီဆိုရင် `CLAUDE.md` ထဲထည့်ဖို့၊ ထပ်ခါတလဲလဲ prompt ကို skill အဖြစ်သိမ်းဖို့၊ side task output ကို subagent ထဲ route လုပ်ဖို့၊ repetitive automation ကို hook အဖြစ်ထားဖို့၊ repository အများကြီးမှာ reuse လုပ်ရင် plugin package လုပ်ဖို့ pattern အဖြစ်ဖော်ပြထားပါတယ်။ ([Claude][6])

Codex official docs မှာလည်း Codex က `AGENTS.md` files တွေကို task မစခင်ဖတ်ပြီး global guidance, project-specific overrides, directory-level instructions တွေကို instruction chain အဖြစ်အသုံးပြုနိုင်တယ်လို့ဖော်ပြထားပါတယ်။ ([OpenAI Developers][7])

ဒီ course မှာ—

**
CLAUDE.md = Claude Code အတွက် project rules
AGENTS.md = Codex / generic coding agents အတွက် project rules
SPEC.md = Product behavior ရဲ့ single source of truth
*

အဖြစ်သုံးမယ်။

## Decision 7 — Skills, Subagents, Plugins ကို တဖြည်းဖြည်းထည့်မယ်

အစပိုင်းမှာ Agent တစ်ယောက်နဲ့စမယ်။ နောက်ပိုင်း project ကြီးလာတဲ့အခါ workflow တွေကို reusable skill, specialist subagent, team-shareable plugin အဖြစ်တိုးချဲ့မယ်။

**
Skill:
Repeated workflow တစ်ခုကို reusable package လုပ်တာ

Subagent:
Specialized role တစ်ခုကို သီးသန့် con*/tool access နဲ့ခွဲတာ

Plugin:
Skills, hooks, subagents, MCP servers တို့ကို shareable package လုပ်တာ
*

Claude Code docs မှာ plugin ကို skills, hooks, subagents, MCP servers တွေကို single installable unit အဖြစ် bundle လုပ်တဲ့ packaging layer အဖြစ်ဖော်ပြထားပါတယ်။ Repo အများကြီးမှာ setup တူတူသုံးချင်တာ သို့မဟုတ် marketplace ကနေ distribute လုပ်ချင်တာမျိုးမှာ plugin သုံးဖို့အကြံပြုထားပါတယ်။ ([Claude][6])

Course progression—

**
Beginner:
Prompt templates

Builder:
Skills

Advanced:
Subagents

Team-ready:
Plugins
*

## Decision 9 — MCP ကို API Direct ပြီးမှသင်မယ်

MCP ကို စောစောတန်းမသင်ပါဘူး။ အရင်ဆုံး API ဆိုတာဘာလဲ, HTTP request/response ဆိုတာဘာလဲ, curl/script နဲ့ API direct usage ကိုနားလည်အောင်လုပ်မယ်။ အဲဒီနောက်မှ MCP ကိုသင်မယ်။

MCP official docs မှာ MCP ကို AI applications တွေက external systems နဲ့ချိတ်ဆက်ဖို့ open-source standard အဖြစ်ဖော်ပြထားပြီး Claude သို့မဟုတ် ChatGPT လို AI apps တွေက local files, databases, tools, workflows တို့နဲ့ချိတ်နိုင်ကြောင်းရှင်းပြထားပါတယ်။ ([Model Con* Protocol][8])

# 6. COURSE_DECISIONS.md Template

ဒီအခန်းရဲ့အရေးကြီးဆုံး artifact က `COURSE_DECISIONS.md` ပါ။

နောက်ပိုင်း project folder ဆောက်တဲ့အခါ ဒီ file ကို root folder ထဲမှာထားမယ်။

*md
# COURSE_DECISIONS.md

## Purpose

This document records the major course and capstone project decisions for Agentic Coding Zero to Hero.

---

## Decision 1 — Mobile Stack

Default mobile/check-in app is PWA-first using React + Vite.

### Reason

Learners already study React + Vite in the frontend foundation.
PWA lets us teach installable/offline-friendly check-in without introducing a new native stack.

### Consequence

MVP uses apps/checkin-pwa.
Expo / React Native is reserved for optional advanced track.

---

## Decision 2 — Notification Scope

MVP does not send email/SMS/Viber/Telegram automatically.

### Reason

Notification providers add API keys, cost, delivery failure, retries, and privacy complexity.

### Consequence

MVP shows ticket code on-screen.
Notification providers are Later Scope.

---

## Decision 3 — Plan Before Code

Non-trivial tasks must start with a plan before implementation.

### Reason

Beginner learners need to review scope, files, risks, and verification before code changes.

### Consequence

Agents must show implementation plan before editing files for major tasks.

---

## Decision 4 — Safety Enforcement

Safety rules are enforced with both instructions and hooks.

### Reason

Instructions alone are not deterministic.
Hooks can block risky tool calls and automate checks.

### Consequence

Course includes .claude/settings.json and hook scripts for dangerous command blocking and post-edit checks.

---

## Decision 5 — Recovery Model

Use checkpoints/rewind for session-level recovery and Git for durable history.

### Reason

Agent edits may need fast undo during a session, while Git commits preserve project history.

### Consequence

Feature work starts from a clean Git state.
Learners use rewind for small session mistakes and Git for durable rollback.

---

## Decision 6 — Project Memory

Use CLAUDE.md, AGENTS.md, and SPEC.md.

### Reason

Agents need persistent project rules, tool-agnostic guidance, and product requirements.

### Consequence

Every major project includes:
- CLAUDE.md
- AGENTS.md
- SPEC.md

---

## Decision 7 — Agentic Workflow

Use SPEC-first and verification-first development.

### Reason

Agent output must be checked against requirements, not vibes.

### Consequence

Features require acceptance criteria, tests or manual verification, and summary before Done.

---

## Decision 8 — MCP Direction

Teach API direct usage before MCP.
MCP Apps are optional advanced enrichment.

### Reason

Learners should understand APIs before wrapping them as MCP tools.

### Consequence

MCP section starts with API direct usage and ends with MCP security and MCP Apps.

---

## Decision 9 — Deployment Provider

Use DigitalOcean App Platform as the default deployment provider.

### Reason

App Platform supports static sites, web services, and managed databases in one deployment model.

### Consequence

Course includes DigitalOcean app spec, staging deploy, smoke test, logs, and rollback.

---

## Decision 10 — Cost Visibility

Deployment and agent usage must include cost awareness.

### Reason

Learners should know what may cost money before deploying or running long agent workflows.

### Consequence

Course includes deployment cost checklist and agent cost/token management.
*

- Run verification commands
- Summarize changed files
- Ask before risky actions

- Known risks are listed
- Changes are ready to commit
