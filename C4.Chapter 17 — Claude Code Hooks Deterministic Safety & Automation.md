# Chapter 17 — Claude Code Hooks: Deterministic Safety & Automation

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့အသုံးချနိုင်ရပါမယ်။

```text
[ ] Hook ဆိုတာဘာလဲ ရှင်းပြနိုင်မယ်
[ ] Hooks က permission rules / CLAUDE.md / prompts ထက် ဘယ်လိုကွာလဲ သိမယ်
[ ] Deterministic safety ဆိုတာဘာလဲ နားလည်မယ်
[ ] PreToolUse, PostToolUse, Notification, SessionStart, Stop hooks တွေရဲ့ role ကိုသိမယ်
[ ] .claude/settings.json ထဲ hooks block ထည့်နိုင်မယ်
[ ] Hook script တစ်ခုက stdin JSON ကိုဖတ်ပြီး decision ပြန်ပေးနိုင်တယ်ဆိုတာနားလည်မယ်
[ ] Dangerous Bash command တွေကို PreToolUse hook နဲ့ block လုပ်နိုင်မယ်
[ ] Protected files တွေကို Edit/Write မလုပ်နိုင်အောင် guard ထားနိုင်မယ်
[ ] PostToolUse hook နဲ့ audit log ရေးနိုင်မယ်
[ ] /hooks menu နဲ့ configured hooks တွေကို verify လုပ်နိုင်မယ်
[ ] Hook failure / JSON syntax / path issue တွေကို debug လုပ်နိုင်မယ်
[ ] Hooks ကို overuse မလုပ်ဘဲ permissions + prompts + hooks ကိုတွဲသုံးနိုင်မယ်
```

ဒီအခန်းက Part 3 ထဲက safety အပိုင်းမှာ အရေးကြီးဆုံးအခန်းတွေထဲကတစ်ခုပါ။ Chapter 15 မှာ permission rules ကိုသင်ခဲ့တယ်။ Chapter 16 မှာ secrets/API key safety ကိုသင်ခဲ့တယ်။ အခု Chapter 17 မှာတော့ “Agent ကို မလုပ်ပါနဲ့” လို့ပြောရုံမဟုတ်ဘဲ **တကယ်လုပ်မရအောင် စနစ်တကျတားခြင်း / အလိုအလျောက်စစ်ခြင်း / audit log သိမ်းခြင်း** ကိုသင်မယ်။

Claude Code docs မှာ hooks တွေကို Claude Code lifecycle point တချို့မှာ user-defined shell commands run လုပ်စေတဲ့ mechanism အဖြစ်ဖော်ပြထားပြီး, model ကိုယုံကြည်ပြီး “လုပ်ပါ” လို့မျှော်လင့်တာထက် certain actions always happen ဖြစ်အောင် deterministic control ပေးတယ်လို့ရှင်းပြထားပါတယ်။ Hooks ကို project rules enforce, repetitive tasks automate, existing tools integrate, formatting, command validation, notifications စတာတွေအတွက်အသုံးပြုနိုင်ပါတယ်။ ([Claude][1])

## 2. Beginner Explanation

Hook ကို အလွယ်ဆုံးနားလည်မယ်ဆိုရင်—

```text
Agent တစ်ခုခုလုပ်မယ့်အချိန်မှာ
အလိုအလျောက်ဝင်စစ်တဲ့ checkpoint
```

ပါ။

ဥပမာ Agent က ဒီ command ကို run ချင်တယ်ဆိုပါစို့။

```bash
rm -rf projects/event-management
```

Prompt ထဲမှာ—

```text
rm -rf မ run ပါနဲ့။
```

လို့ရေးထားရုံနဲ့ မလုံလောက်နိုင်ဘူး။ Agent မေ့နိုင်တယ်။ Context ရှုပ်နိုင်တယ်။ User က approval မှားနှိပ်မိနိုင်တယ်။

Hook နဲ့ဆိုရင်—

```text
Agent က Bash tool နဲ့ command run မလုပ်ခင်
PreToolUse hook က command ကိုဖတ်တယ်
rm -rf ပါလားစစ်တယ်
ပါရင် block လုပ်တယ်
Agent ကို “Destructive command blocked” လို့ပြောတယ်
```

ဒီလိုဖြစ်သွားမယ်။

Mental model—

```text
CLAUDE.md / Prompt = Agent ကို လမ်းညွှန်တာ
Permissions = Tool access gate
Hooks = Event ဖြစ်ချိန်မှာ deterministic script run တာ
```

နောက်ထပ်ဥပမာ—

```text
Agent က app.js ကိုပြင်ပြီးသွားတယ်
PostToolUse hook က edited file path ကို log ထဲသိမ်းတယ်
နောက်ပိုင်း audit လုပ်ရင် “ဘယ် file ကို Agent ပြင်ခဲ့လဲ” သိနိုင်တယ်
```

ဒါကြောင့် Hooks ကို—

```text
Safety guard
Automation checkpoint
Audit logger
Verification trigger
Notification system
```

လိုသုံးနိုင်တယ်။

# 3. Hooks ဘာကြောင့်လိုလဲ

Agentic Coding မှာ Agent ကို “မလုပ်ပါနဲ့” လို့ပြောတာက အမြဲမလုံလောက်ပါဘူး။

ဥပမာ prompt ထဲမှာ—

```text
.env မဖတ်ပါနဲ့။
```

လို့ရေးထားတယ်။ Agent ကများသောအားဖြင့်လိုက်နာနိုင်တယ်။ ဒါပေမယ့်—

```text
Long session ကြာသွား
Context compact ဖြစ်သွား
User က task အသစ်ပေး
Agent က build error ကို debug လုပ်ရင်း .env ဖတ်ချင်လာ
```

လို situation တွေမှာ rule လွတ်နိုင်တယ်။

Hooks က ဒီလိုပြဿနာတွေကိုလျှော့ပေးတယ်။

```text
Instruction မဟုတ်ဘဲ checkpoint ဖြစ်တယ်။
Model ဆုံးဖြတ်ချက်မဟုတ်ဘဲ script ဆုံးဖြတ်ချက်ဖြစ်တယ်။
Tool call မလုပ်ခင် / လုပ်ပြီးချိန်မှာ fixed logic run တယ်။
```

Claude Code hooks guide မှာ hooks တွေက file edit ပြီးတိုင်း formatter run ခြင်း, command မ run ခင် block/validate လုပ်ခြင်း, Claude needs input ဖြစ်တဲ့အခါ notification ပို့ခြင်း, session start မှာ context inject လုပ်ခြင်း စတာတွေကို automate လုပ်နိုင်တယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][1])

# 4. Hook Lifecycle ဆိုတာဘာလဲ

Hook lifecycle ဆိုတာ Claude Code က အလုပ်လုပ်နေတဲ့ flow ထဲမှာ hook run လုပ်နိုင်တဲ့ event points တွေပါ။

Beginner အတွက် အဓိကသိသင့်တဲ့ events—

```text
SessionStart
Claude Code session စချိန်

PreToolUse
Agent tool မသုံးခင်

PostToolUse
Agent tool သုံးပြီးနောက်

Notification
Claude က user input / permission စောင့်နေတဲ့အခါ

Stop
Claude က response ပြီးပြီလို့ရပ်မယ့်အခါ

ConfigChange
settings / skills config ပြောင်းတဲ့အခါ
```

Claude Code hooks reference မှာ hook event, matcher group, hook handler ဆိုပြီး configuration ကို ၃ ဆင့်နဲ့ဖော်ပြထားပါတယ်။ ပထမဆုံး hook event ကိုရွေးရတယ်, ဥပမာ `PreToolUse` သို့မဟုတ် `Stop`; ထို့နောက် matcher group နဲ့ Bash/Edit/Write လို tool-specific filter ထည့်နိုင်တယ်; နောက်ဆုံး hook handler — shell command, HTTP endpoint, MCP tool, prompt, agent — ကို run စေပါတယ်။ ([Claude][2])

# 5. Beginner အတွက် အရေးကြီးဆုံး Hooks

ဒီ course အစပိုင်းမှာ hooks အကုန်မသုံးပါဘူး။ အောက်ပါ ၅ မျိုးကိုပဲအရင်နားလည်ပါ။

## 5.1 PreToolUse

Tool မသုံးခင် run တယ်။

သုံးမယ့်နေရာ—

```text
Dangerous command block
.env edit/read block
package install approval check
git reset / git clean block
deployment command block
```

ဥပမာ—

```text
Agent: rm -rf build ကို run ချင်တယ်
PreToolUse hook: command ထဲ rm -rf ပါလားစစ်
ပါရင် deny
```

`PreToolUse` သည် tool call မလုပ်ခင် fire ဖြစ်ပြီး input JSON ထဲမှာ `tool_name`, `tool_input` ပါနိုင်တယ်။ Official reference example မှာ Bash command ကို hook က stdin JSON မှဖတ်ပြီး destructive command ဖြစ်ရင် JSON decision နဲ့ `permissionDecision: "deny"` ပြန်ပေးကာ Claude Code က tool call ကို block လုပ်တယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][2])

---

## 5.2 PostToolUse

Tool သုံးပြီးနောက် run တယ်။

သုံးမယ့်နေရာ—

```text
Edited file log သိမ်း
Formatter run
Lint run
Audit log append
Tool output redaction / transformation, advanced
```

ဥပမာ—

```text
Agent က app.js ကို edit လုပ်တယ်
PostToolUse hook က edited file path ကို .claude/logs/tool-use.log ထဲရေးတယ်
```

Official hooks guide မှာ `PostToolUse` + `Edit|Write` matcher ကိုသုံးပြီး Claude က file edit လုပ်ပြီးတိုင်း Prettier run စေတဲ့ example ကိုဖော်ပြထားပါတယ်။ ([Claude][1])

## 5.3 Notification

Claude က user input သို့မဟုတ် permission စောင့်နေတဲ့အခါ notification ပို့နိုင်တယ်။

သုံးမယ့်နေရာ—

```text
Agent permission prompt စောင့်နေ
Agent idle ဖြစ်နေ
Long task ပြီးသွား
```

Official guide မှာ `Notification` hook ကို desktop notification ပို့ဖို့သုံးနိုင်ပြီး matcher values အနေနဲ့ `permission_prompt`, `idle_prompt`, `auth_success`, MCP elicitation-related events စတာတွေရှိကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

---

## 5.4 SessionStart

Claude Code session စချိန် run တယ်။

သုံးမယ့်နေရာ—

```text
Current branch context inject
Git status reminder
Course safety reminder
Important sprint/context reminder
```

Official reference မှာ `SessionStart` hook က new session, resume, clear, compact စတဲ့ session start situations တွေမှာ run နိုင်ပြီး `additionalContext`, `sessionTitle`, `watchPaths`, `reloadSkills` စတဲ့ event-specific outputs တွေသုံးနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][2])

## 5.5 Stop

Claude က “ပြီးပြီ” လို့ရပ်မယ့်အချိန် run တယ်။

သုံးမယ့်နေရာ—

```text
Definition of Done မပြည့်ရင် stop မလုပ်စေ
Verification checklist မရှိရင် remind
Task incomplete ဖြစ်ရင် continue
```

Official hooks reference မှာ exit code 2 ကို hook က “stop, don’t do this” signal အဖြစ်သုံးနိုင်ပြီး `Stop` event မှာ exit code 2 က Claude ကို stopping မလုပ်စေဘဲ conversation ဆက်လုပ်စေနိုင်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][2])

# 6. Hook Locations — ဘယ်နေရာမှာထားမလဲ

Hooks ကို ဘယ် settings file ထဲမှာထားလဲဆိုတာအရေးကြီးပါတယ်။

Beginner အတွက် အဓိက locations—

```text
~/.claude/settings.json
= user-wide hooks, project အားလုံးမှာသက်ရောက်

.claude/settings.json
= project-specific hooks, repo ထဲ commit လုပ်နိုင်

.claude/settings.local.json
= project-specific local hooks, personal, gitignored
```

Official reference မှာ hook locations တွေကို scope အလိုက်ခွဲထားပြီး `~/.claude/settings.json` သည် user projects အားလုံး, `.claude/settings.json` သည် single project / shareable, `.claude/settings.local.json` သည် single project / local gitignored, plugin hooks, skill/agent frontmatter hooks စတာတွေရှိကြောင်းဖော်ပြထားပါတယ်။ ([Claude][2])

ဒီ course အတွက် rule—

```text
Shared course/project hooks
→ .claude/settings.json

Personal machine-specific hooks
→ .claude/settings.local.json

Sensitive local paths / machine-specific commands
→ settings.local.json
```

# 7. Hook Configuration Anatomy

Hook config ကို `.claude/settings.json` ထဲမှာ `"hooks"` block အဖြစ်ရေးတယ်။

Basic structure—

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node .claude/hooks/guard-tool-use.mjs"
          }
        ]
      }
    ]
  }
}
```

ခွဲကြည့်ရင်—

```text
PreToolUse
= ဘယ် event မှာ run မလဲ

matcher: "Bash"
= Bash tool ဖြစ်တဲ့အခါပဲ run

type: "command"
= shell command / script run

command
= run မယ့် script
```

Official reference မှာ matcher field က hook ဘယ်အချိန် fire လုပ်မလဲ filter လုပ်ပေးပြီး `PreToolUse`, `PostToolUse`, `PermissionRequest` စတဲ့ tool events တွေမှာ matcher သည် `tool_name` ကို filter လုပ်တယ်လို့ဖော်ပြထားပါတယ်။ `Bash` သည် Bash tool ကို exact match လုပ်နိုင်ပြီး `Edit|Write` သည် Edit သို့မဟုတ် Write tool ကို match လုပ်နိုင်ပါတယ်။ ([Claude][2])

# 8. Hook Input / Output

Command hook တစ်ခုက stdin မှာ JSON input လက်ခံတယ်။

Example input—

```json
{
  "session_id": "abc123",
  "cwd": "/home/user/my-project",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm test"
  }
}
```

Hook script က အဲဒီ JSON ကိုဖတ်ပြီး decision ပြန်ပေးနိုင်တယ်။

## 8.1 Exit code based blocking

`exit 0`

```text
OK / no decision / proceed to normal flow
```

`exit 2`

```text
blocking signal, event အလိုက် tool call / prompt / stop ကို block နိုင်
```

Official reference မှာ command hooks တွေသည် JSON ကို stdin မှလက်ခံပြီး exit codes, stdout, stderr နဲ့ result ပြန်ပေးကြောင်း၊ exit code 2 သည် PreToolUse မှာ tool call ကို block, PermissionRequest မှာ permission ကို deny, Stop မှာ Claude ကို stopping မလုပ်စေကြောင်း event-specific behavior အဖြစ်ဖော်ပြထားပါတယ်။ ([Claude][2])

## 8.2 JSON output based decision

ပိုတိတိကျကျ decision ပေးချင်ရင် stdout မှာ JSON ပြန်ပေးနိုင်တယ်။

Example—

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Destructive command blocked by hook"
  }
}
```

Official reference မှာ JSON output ကိုသုံးရင် hook က `permissionDecision: "deny"` နဲ့ tool call ကို deny လုပ်နိုင်ပြီး, JSON output ကို processed လုပ်ချင်ရင် exit code 0 နဲ့ stdout သည် JSON object တစ်ခုတည်းဖြစ်ရမယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][2])

Beginner rule—

```text
ရိုးရိုး block လုပ်ချင်ရင် exit 2 + stderr message သုံးလို့ရတယ်။
Structured decision လိုချင်ရင် exit 0 + JSON stdout သုံးပါ။
```

# 9. Hooks vs Permissions

ဒီ chapter မှာ အလွန်အရေးကြီးတဲ့ distinction ရှိတယ်။

```text
Permissions
= tool access ကို hard allow/ask/deny လုပ်တဲ့ built-in gate

Hooks
= lifecycle event မှာ script/automation run တဲ့ guardrail
```

Official hooks guide မှာ Bash command matcher/filter သည် best-effort ဖြစ်ပြီး Bash command မ parse လုပ်နိုင်တဲ့အခါ hook filter က fail open ဖြစ်နိုင်ကြောင်းသတိပေးထားပါတယ်။ Hard allow/deny enforce လုပ်ချင်ရင် permission system ကိုသုံးသင့်တယ်လို့ docs ကပြောထားပါတယ်။ ([Claude][1])

ဒါကြောင့် course rule—

```text
Hard block = permissions.deny
Custom validation / logging / automation = hooks
```

# 12. Hook 1 — Dangerous Bash Command Guard

ဒီ hook က Agent က Bash command run မလုပ်ခင် command ကိုစစ်မယ်။

Block လုပ်မယ့် patterns—

```text
rm -rf
git reset --hard
git clean -fd
git clean -fdx
curl ... | bash
curl ... | sh
sudo rm
drop database
```

## `.claude/hooks/guard-bash.mjs`

```js
#!/usr/bin/env node

let input = "";

process.stdin.setEncoding("utf8");

process.stdin.on("data", chunk => {
  input += chunk;
});

process.stdin.on("end", () => {
  let payload;

  try {
    payload = JSON.parse(input || "{}");
  } catch {
    deny("Hook failed: invalid JSON input.");
    return;
  }

  const toolName = payload.tool_name;
  const command = payload.tool_input?.command || "";

  if (toolName !== "Bash") {
    allow();
    return;
  }

  const dangerousPatterns = [
    /rm\s+-rf\s+/i,
    /sudo\s+rm\s+/i,
    /git\s+reset\s+--hard/i,
    /git\s+clean\s+-fdx?/i,
    /curl\s+.*\|\s*(bash|sh)/i,
    /wget\s+.*\|\s*(bash|sh)/i,
    /drop\s+database/i,
    /delete\s+from\s+/i
  ];

  const matched = dangerousPatterns.find(pattern => pattern.test(command));

  if (matched) {
    deny(`Blocked dangerous Bash command: ${command}`);
    return;
  }

  allow();
});

function allow() {
  process.exit(0);
}

function deny(reason) {
  const decision = {
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: reason
    }
  };

  process.stdout.write(JSON.stringify(decision));
  process.exit(0);
}
```

ဒီ script က—

```text
stdin JSON ဖတ်တယ်
tool_name Bash ဖြစ်လားစစ်တယ်
command string ထဲ dangerous pattern ပါလားစစ်တယ်
ပါရင် PreToolUse deny decision ပြန်ပေးတယ်
မပါရင် silent allow
```

## `.claude/hooks/protect-files.mjs`

```js
#!/usr/bin/env node

let input = "";

process.stdin.setEncoding("utf8");

process.stdin.on("data", chunk => {
  input += chunk;
});

process.stdin.on("end", () => {
  let payload;

  try {
    payload = JSON.parse(input || "{}");
  } catch {
    deny("Hook failed: invalid JSON input.");
    return;
  }

  const toolName = payload.tool_name || "";
  const filePath =
    payload.tool_input?.file_path ||
    payload.tool_input?.path ||
    "";

  if (!["Edit", "Write", "MultiEdit"].includes(toolName)) {
    allow();
    return;
  }

  const protectedPatterns = [
    ".env",
    ".env.local",
    "secrets/",
    ".git/",
    "node_modules/",
    "dist/",
    "build/",
    "package-lock.json"
  ];

  const normalized = filePath.replace(/\\/g, "/");

  const matched = protectedPatterns.find(pattern =>
    normalized.includes(pattern)
  );

  if (matched) {
    deny(`Protected file or folder blocked: ${filePath}`);
    return;
  }

  allow();
});

function allow() {
  process.exit(0);
}

function deny(reason) {
  const decision = {
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: reason
    }
  };

  process.stdout.write(JSON.stringify(decision));
  process.exit(0);
}
```

မှတ်ချက်—

```text
package-lock.json ကို beginner stage မှာ protect ထားတာပါ။
နောက်ပိုင်း package install chapter/task မှာ explicit approval နဲ့ temporary allow/ask rule ပြန်ပြင်နိုင်တယ်။
```

# 14. Hook 3 — Tool Use Audit Logger

ဒီ hook က Agent tool သုံးပြီးနောက် audit log ရေးမယ်။

Log file—

```text

```

သိမ်းမယ့်အရာ—

```text

```

Secret မသိမ်းရ။ Full command ထဲ secret ပါနိုင်တဲ့အခါ redaction လုပ်ရမယ်။

## `.claude/hooks/log-tool-use.mjs`

```js
#!/usr/bin/env node

import fs from "node:fs";
import path from "node:path";

let input = "";

process.stdin.setEncoding("utf8");

process.stdin.on("data", chunk => {
  input += chunk;
});

process.stdin.on("end", () => {
  let payload;

  try {
    payload = JSON.parse(input || "{}");
  } catch {
    process.exit(0);
  }

  const cwd = payload.cwd || process.cwd();
  const logDir = path.join(cwd, ".claude", "logs");
  const logFile = path.join(logDir, "tool-use.log");

  const toolName = payload.tool_name || "unknown";
  const toolInput = payload.tool_input || {};

  const filePath = toolInput.file_path || toolInput.path || "";
  const command = toolInput.command ? redact(toolInput.command) : "";

  const record = {
    timestamp: new Date().toISOString(),
    event: payload.hook_event_name || "PostToolUse",
    tool: toolName,
    filePath,
    command
  };

  fs.mkdirSync(logDir, { recursive: true });
  fs.appendFileSync(logFile, JSON.stringify(record) + "\n");

  process.exit(0);
});

function redact(value) {
  return String(value)
    .replace(/sk-[A-Za-z0-9_-]+/g, "sk-REDACTED")
    .replace(/dop_v1_[A-Za-z0-9_-]+/g, "dop_v1_REDACTED")
    .replace(/Bearer\s+[A-Za-z0-9._-]+/g, "Bearer REDACTED")
    .replace(/password=([^@\s]+)/gi, "password=REDACTED");
}
```

ဒီ hook က block မလုပ်ဘူး။ Log ပဲရေးတယ်။

PostToolUse သည် tool သုံးပြီးနောက် run တာဖြစ်လို့ tool call ကိုမတားနိုင်ဘူး။ Official reference မှာ `PostToolUse` event သည် exit code 2 နဲ့ tool call ကို block မလုပ်နိုင်ဘဲ, tool already ran ဖြစ်တာကြောင့် stderr ကို Claude ဆီပြတဲ့ behavior ဖြစ်တယ်လို့ဖော်ပြထားပါတယ်။ ([Claude][2])

## Error 2 — `hooks` block ထည့်ရင်း `permissions` block ဖျက်မိခြင်း

```json
    ...
  },
  "hooks": {
    ...
  }
}
```

---

## Error 3 — Hook script path မမှန်ခြင်း

Command—

```json
"command": "node .claude/hooks/guard-bash.mjs"
```

Project root cwd မမှန်ရင် path issue ဖြစ်နိုင်တယ်။

ပြင်ရန်—

```text
Claude Code ကို project root ထဲ run ပါ။
cwd ကို hook input ထဲကနေစစ်ပါ။
လိုအပ်ရင် absolute path သို့မဟုတ် $CLAUDE_PROJECT_DIR pattern သုံးပါ။
```

Official examples ထဲမှာ project-root hook script ကို `"$CLAUDE_PROJECT_DIR"/.claude/hooks/protect-files.sh` ပုံစံနဲ့ခေါ်ထားတာကိုတွေ့နိုင်ပါတယ်။ ([Claude][1])

---

## Error 4 — stdout JSON မမှန်ခြင်း

Structured JSON output သုံးတဲ့ hook မှာ stdout မှာ JSON object တစ်ခုတည်းသာရှိသင့်တယ်။

မကောင်း—

```text
Checking command...
{"hookSpecificOutput": ...}
```

JSON parse ပျက်နိုင်တယ်။

ကောင်း—

```json
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"Blocked"}}
```

Official reference က JSON output ကို Claude Code process လုပ်ချင်ရင် exit code 0 ဖြစ်ရမယ်, stdout မှာ JSON object တစ်ခုတည်းပါရမယ်, shell profile က startup text print လုပ်ရင် JSON parsing ကိုနှောင့်ယှက်နိုင်တယ်လို့သတိပေးထားပါတယ်။ ([Claude][2])

---

## Error 5 — PostToolUse ကို block hook အဖြစ်သုံးခြင်း

`PostToolUse` က tool သုံးပြီးသွားပြီးမှ run တာဖြစ်လို့ command ကိုမတားနိုင်ဘူး။

```text
PreToolUse
```

သုံးပါ။

## Error 6 — Hook ကို hard security boundary အဖြစ်တစ်ခုတည်းအားကိုးခြင်း

Hooks useful ဖြစ်တယ်။ ဒါပေမယ့် Bash parser/filter best-effort ဖြစ်နိုင်တာကြောင့် hard allow/deny အတွက် permission system သုံးသင့်တယ်လို့ official guide မှာသတိပေးထားပါတယ်။ ([Claude][1])

Course rule—

```text
Sensitive / destructive action:
permissions.deny + PreToolUse hook + human review
```

# 27. Chapter Summary

ဒီအခန်းမှာ **Claude Code Hooks: Deterministic Safety & Automation** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Hook = Agent lifecycle ထဲက fixed checkpoint မှာ အလိုအလျောက် run တဲ့ script/automation
```

Hooks သုံးရတဲ့အကြောင်း—

```text
Prompt instructions ကို model ကလိုက်နာဖို့မျှော်လင့်ရတာထက်
certain safety/automation actions တွေ always happen ဖြစ်အောင်လုပ်ရန်
```

အရေးကြီးဆုံး events—

```text
PreToolUse
= tool မသုံးခင် guard/check/block

PostToolUse
= tool သုံးပြီးနောက် log/format/audit

Notification
= Claude input/permission စောင့်နေတဲ့အခါ notify

SessionStart
= session စချိန် context inject

Stop
= Claude ရပ်မယ့်အခါ done check
```

ဒီ chapter lab မှာတည်ဆောက်ခဲ့တဲ့ hooks—

```text
guard-bash.mjs
= dangerous Bash commands block

protect-files.mjs
= protected files/folders edit block

log-tool-use.mjs
= tool use audit log
```

အရေးကြီးဆုံး safety takeaway—

```text
Hooks သည် permissions ကိုအစားမထိုးပါ။
Permissions သည် hard tool gate ဖြစ်သည်။
Hooks သည် custom guardrail/automation ဖြစ်သည်။
Git diff + human review သည် အမြဲလိုသည်။
```
