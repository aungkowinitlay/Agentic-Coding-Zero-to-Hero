## 2. Beginner Explanation

Frontend Accessibility ဆိုတာ—

```text
လူအများဆုံးက website ကိုမြင်ပြီး mouse နဲ့သုံးနိုင်ရုံတင်မက၊

- keyboard သုံးသူ
- screen reader သုံးသူ
- စာလုံးကြီးချဲ့ဖတ်သူ
- အရောင်ခွဲရခက်သူ
- လက်လှုပ်ရှားမှုအကန့်အသတ်ရှိသူ
- အသံမကြားသူ
- နားလည်မှု/မှတ်ဉာဏ်အခက်အခဲရှိသူ
- mobile device သုံးသူ

တို့ပါ UI ကိုနားလည်၊ လမ်းညွှန်၊ အသုံးပြုနိုင်အောင်တည်ဆောက်ခြင်း
```

ဖြစ်ပါတယ်။

မကောင်းတဲ့ UI—

```text
Mouse နဲ့ပဲ click လို့ရ
Focus ဘယ်မှာရောက်နေလဲမမြင်ရ
Input တွေမှာ placeholder ပဲရှိ
Error ကို အနီရောင်နဲ့ပဲပြ
Button ကို <div> နဲ့ဆောက်
Icon image မှာ alt မရှိ
Status update ဖြစ်တာ screen reader မသိ
Zoom လုပ်ရင် text ပျောက်
```

ကောင်းတဲ့ UI—

```text
Keyboard နဲ့ Tab လုပ်လို့ရ
Focus indicator ရှင်း
Button ကို native <button> သုံး
Input တစ်ခုချင်းစီ visible label ရှိ
Error ကိုစာသားနဲ့ရှင်း
Color အပြင် text/icon/state ပါသုံး
Dynamic result ကို live region နဲ့ကြေညာ
Zoom/mobile မှာ content မပျောက်
```

Accessibility ကို disabled users အတွက် “extra feature” လို့မမြင်သင့်ပါဘူး။

```text
Accessibility
= Frontend quality
= Correct HTML
= Better keyboard support
= Clearer UI
= More robust product
```

# 3. WCAG 2.2 ဆိုတာဘာလဲ

**WCAG — Web Content Accessibility Guidelines** က web content accessibility အတွက် W3C standard ဖြစ်ပါတယ်။ ဒီ course မှာ WCAG 2.2 ကို baseline အဖြစ်သုံးမယ်။ WCAG 2.2 ကို 2023-10-05 မှာ W3C Recommendation အဖြစ်ထုတ်ဝေခဲ့ပြီး 2024-12-12 မှာ update ထုတ်ထားပါတယ်။ W3C က လက်ရှိအချိန်မှာ နောက်ဆုံး WCAG version ကိုအသုံးပြုဖို့ အကြံပြုထားပါတယ်။ ([W3C][1])

WCAG 2.2 မှာ success criteria တွေကို အဓိက level သုံးမျိုးခွဲထားတယ်။

```text
Level A
= အခြေခံမဖြစ်မနေလိုတဲ့ requirements

Level AA
= Production websites အများစုက target လုပ်သင့်တဲ့ standard

Level AAA
= ပိုမြင့်တဲ့ accessibility level
```

ဒီ beginner chapter က full WCAG conformance audit မဟုတ်ပါဘူး။ Accessibility fundamentals ကို frontend စရေးချိန်ကတည်းက ထည့်သွင်းတည်ဆောက်ဖို့ဖြစ်ပါတယ်။ W3C ရဲ့ quick accessibility checks တွေကလည်း preliminary review ပဲဖြစ်ပြီး comprehensive audit ကိုအစားမထိုးနိုင်ကြောင်းရှင်းပြထားပါတယ်။ ([W3C][2])

# 4. POUR Principles

WCAG requirements တွေကို အဓိက principle လေးခုနဲ့မှတ်နိုင်တယ်။

```text
P — Perceivable
O — Operable
U — Understandable
R — Robust
```

W3C WAI က accessibility requirements တွေကို Perceivable, Operable, Understandable, Robust ဆိုတဲ့ principle လေးခုနဲ့စုစည်းထားပါတယ်။ ([W3C][3])

## 4.1 Perceivable — သိရှိခံစားနိုင်ရမယ်

User က content ကို တစ်နည်းမဟုတ်တစ်နည်းနဲ့ သိရှိနိုင်ရမယ်။

```text
Image မှာ alt text ရှိ
Video မှာ captions ရှိ
Text contrast လုံလောက်
Color တစ်ခုတည်းကိုအဓိပ္ပါယ်အဖြစ်မသုံး
Content structure ကို code ထဲမှာပါဖော်ပြ
```

## 4.2 Operable — အသုံးပြုနိုင်ရမယ်

```text
Keyboard တစ်ခုတည်းနဲ့လုပ်နိုင်
Focus indicator မြင်နိုင်
Click target သေးလွန်းမနေ
Keyboard trap မဖြစ်
Animation ကြောင့်အသုံးပြုမရတာမဖြစ်
```

## 4.3 Understandable — နားလည်နိုင်ရမယ်

```text
Labels ရှင်း
Error message ရှင်း
Navigation predictable
Button names တိကျ
Instructions မရှုပ်
```

## 4.4 Robust — Tools မျိုးစုံနဲ့အလုပ်လုပ်ရမယ်

```text
Valid semantic HTML
Native controls
Accessible names
Browser / assistive technology နဲ့ compatible
ARIA ကိုမှန်ကန်စွာသုံး
```

# 5. Accessibility ကို “အစကတည်းက” ထည့်ပါ

Accessibility ကို project နောက်ဆုံးအခန်းရောက်မှ audit လုပ်ပြီး ပြင်တာထက် component တစ်ခုချင်းစီဆောက်ချိန်ကတည်းကထည့်တာပိုကောင်းတယ်။

```text
မကောင်းတဲ့ workflow:

UI ဆောက်
→ Feature အကုန်ပြီး
→ Accessibility audit
→ HTML structure ပြန်ပြင်
→ Components အများကြီးပြန်ပြင်
→ Regression bugs

ကောင်းတဲ့ workflow:

Component plan
→ Semantic HTML
→ Keyboard behavior
→ Focus style
→ Labels / names
→ States / errors
→ Manual keyboard test
→ Accessibility audit
```

Course rule—

```text
Accessibility is part of Definition of Done.
Accessibility is not a final decoration.
```

# 6. Semantic HTML First

Semantic HTML ဆိုတာ element နာမည်က သူ့ရဲ့အဓိပ္ပါယ်နဲ့တာဝန်ကိုပြောနိုင်တဲ့ HTML ဖြစ်တယ်။

မကောင်း—

```jsx
<div className="button" onClick={handleRegister}>
  Register
</div>
```

ကောင်း—

```jsx
<button type="button" onClick={handleRegister}>
  Register လုပ်မည်
</button>
```

အခြား semantic elements—

```html
<header>
<nav>
<main>
<section>
<article>
<aside>
<footer>

<h1> to <h6>

<button>
<a>
<form>
<label>
<input>
<select>
<textarea>

<ul>
<ol>
<li>

<table>
<caption>
<th>
<td>
```

React က browser ရဲ့ native HTML နဲ့ SVG components အားလုံးကို support လုပ်တာကြောင့် accessible HTML elements တွေကို JSX ထဲတိုက်ရိုက်သုံးနိုင်ပါတယ်။ ([React][4])

# 7. Page Landmarks

Landmarks တွေက page ကို အဓိကအပိုင်းတွေအဖြစ်ခွဲပေးတယ်။

```jsx
<>
  <header>
    ...
  </header>

  <nav aria-label="အဓိက navigation">
    ...
  </nav>

  <main>
    ...
  </main>

  <footer>
    ...
  </footer>
</>
```

`header`, `nav`, `main`, `aside`, `footer` စတဲ့ HTML elements တွေက assistive technology တွေအတွက် page regions ကို programmatically ဖော်ပြပေးနိုင်တယ်။ Screen reader users တွေက landmark တစ်ခုကနေတစ်ခုကို လျင်မြန်စွာသွားနိုင်ပါတယ်။ ([W3C][5])

Rule—

```text
Page တစ်ခုမှာ primary <main> တစ်ခုထားပါ။
Navigation တစ်ခုထက်ပိုရင် aria-label နဲ့ခွဲပါ။
Section တစ်ခုမှာ heading ရှိအောင်လုပ်ပါ။
```

# 8. Heading Hierarchy

Headings တွေက စာလုံးကြီးဖို့သုံးတဲ့ style မဟုတ်ပါဘူး။ Page structure ကိုဖော်ပြတာပါ။

ကောင်းတဲ့ hierarchy—

```text
h1 — Frontend Accessibility Basics

  h2 — Event Summary

  h2 — Event Filters

  h2 — Upcoming Events

    h3 — AI Workshop
    h3 — React Meetup

  h2 — Accessibility Checklist
```

ရှောင်ရန်—

```text
h1
  h4
    h2
```

W3C WAI က headings တွေဟာ page content organization ကိုဖော်ပြပြီး browser နဲ့ assistive technology တွေက in-page navigation အတွက်အသုံးပြုနိုင်ကြောင်းရှင်းပြထားပါတယ်။ Heading rank တွေကို logical hierarchy အတိုင်းအသုံးပြုပြီး မလိုအပ်ဘဲ level မကျော်သင့်ပါဘူး။ ([W3C][6])

Rule—

```text
Heading level ကို visual size ကြောင့်မရွေးပါနဲ့။
Content hierarchy ကြောင့်ရွေးပါ။
Visual size ကို CSS နဲ့ပြင်ပါ။
```

# 9. Skip Link

Keyboard user က page header/navigation ကို Tab အများကြီးမနှိပ်ဘဲ main content ဆီတန်းသွားနိုင်ဖို့ skip link သုံးတယ်။

```jsx
<a className="skip-link" href="#main-content">
  အဓိကအကြောင်းအရာသို့ ကျော်ရန်
</a>

<main id="main-content" tabIndex={-1}>
  ...
</main>
```

Skip link ကို ပုံမှန်အချိန်မှာဖျောက်ထားပြီး keyboard focus ရလာမှမြင်အောင် CSS ရေးနိုင်တယ်။

```css
.skip-link {
  position: fixed;
  top: 12px;
  left: 12px;
  transform: translateY(-160%);
}

.skip-link:focus {
  transform: translateY(0);
}
```

Skip links တွေဟာ keyboard users ကို page ရဲ့အရေးကြီးတဲ့ section ဆီတန်းသွားနိုင်အောင်ကူညီပြီး focus ရချိန်မှာမြင်ရအောင်ပြနိုင်ပါတယ်။ ([W3C][7])

# 10. Button vs Link

## Button

Current page ထဲက action တစ်ခုလုပ်တဲ့အခါသုံးပါ။

```jsx
<button type="button" onClick={handleReset}>
  Filters reset
</button>
```

Actions—

```text
Submit
Open modal
Toggle filter
Reset
Delete
Play/Pause
```

## Link

တခြား page/location ကိုသွားတဲ့အခါသုံးပါ။

```jsx
<a href="#events-heading">
  Event များသို့သွားမည်
</a>
```

Navigation—

```text
Page အသစ်သွား
Section တစ်ခုဆီသွား
File download သွား
External website သွား
```

မကောင်း—

```jsx
<div role="button" onClick={handleReset}>
  Reset
</div>
```

ARIA `role="button"` ထည့်လိုက်တာက native button ရဲ့ keyboard behavior နဲ့ browser styling ကို အလိုအလျောက်မပေးပါဘူး။ Custom button ရေးမယ်ဆိုရင် expected keyboard interactions အားလုံးကို developer ကကိုယ်တိုင်တာဝန်ယူရတယ်။ ဒါကြောင့် native `<button>` ကိုဦးစားပေးသင့်ပါတယ်။ ([W3C][8])

# 11. Keyboard Accessibility

UI ကို mouse မသုံးဘဲအောက်ပါ keys တွေနဲ့အသုံးပြုနိုင်ရမယ်။

```text
Tab
= နောက် focusable control သို့သွား

Shift + Tab
= အရင် control သို့ပြန်သွား

Enter
= Link / button အချို့ activate

Space
= Native button activate

Arrow Keys
= Native radio/select နှင့် complex widgets အချို့

Escape
= Dialog / popup ပိတ်ရန်, later
```

WCAG keyboard criterion ရဲ့ရည်ရွယ်ချက်က mouse လို pointer device မသုံးနိုင်သူတွေပါ functionality ကို keyboard interface နဲ့အသုံးပြုနိုင်စေဖို့ဖြစ်ပါတယ်။ ([W3C][9])

Keyboard test—

```text
1. Mouse ကိုမထိပါနဲ့
2. Page ကို refresh လုပ်ပါ
3. Tab နှိပ်ပါ
4. Skip link ပေါ်လားစစ်ပါ
5. Button/link တစ်ခုချင်းစီ focus ရလားစစ်ပါ
6. Focus order logical ဖြစ်လားစစ်ပါ
7. Enter/Space နဲ့ activate လုပ်ပါ
8. Shift+Tab နဲ့ပြန်သွားပါ
9. ဘယ်နေရာမှာမှ stuck မဖြစ်ကြောင်းစစ်ပါ
```

# 12. Focus Order

Keyboard focus order က page ရဲ့ visual/logical order နဲ့ကိုက်သင့်တယ်။

```text
Skip link
→ Header links
→ Filter buttons
→ Action buttons
→ Event registration buttons
→ Footer links
```

WCAG Focus Order criterion က sequential keyboard navigation လုပ်တဲ့အခါ focus order သည် meaning နဲ့ operability ကိုထိန်းထားရမယ်လို့သတ်မှတ်ပါတယ်။ Native DOM order ကိုအခြေခံတာက ပုံမှန်အားဖြင့်အကောင်းဆုံးဖြစ်ပါတယ်။ ([W3C][10])

ရှောင်ရန်—

```jsx
tabIndex={1}
tabIndex={2}
tabIndex={5}
```

Positive `tabIndex` တွေက DOM order နဲ့ keyboard order မကိုက်တာဖြစ်စေနိုင်တယ်။

Course rule—

```text
DOM ကို logical order နဲ့ရေးပါ။
Native focus order ကိုဦးစားပေးပါ။
tabIndex={0} ကိုလည်းလိုအပ်မှသုံးပါ။
```

# 13. Visible Focus

Keyboard user က လက်ရှိ focus ဘယ် element မှာရှိတယ်ဆိုတာမြင်နိုင်ရမယ်။

မလုပ်ရ—

```css
button:focus {
  outline: none;
}
```

ကောင်း—

```css
:where(a, button, input, select, textarea, [tabindex]):focus-visible {
  outline: 3px solid #111827;
  outline-offset: 3px;
}
```

WCAG Focus Visible criterion ရဲ့ရည်ရွယ်ချက်က keyboard user ကို လက်ရှိ interaction point ဘယ်မှာရှိတယ်ဆိုတာမြင်စေဖို့ဖြစ်ပါတယ်။ Focus indicator က focus ရှိနေသရွေ့မြင်နိုင်ရမယ်။ ([W3C][11])

# 15. Color Contrast Basics

WCAG 2.2 Level AA အခြေခံ—

```text
Normal text
= အနည်းဆုံး 4.5 : 1

Large text
= အနည်းဆုံး 3 : 1

Important UI controls / meaningful graphics
= အနည်းဆုံး 3 : 1
```

Text contrast 4.5:1 ratio က normal text ရဲ့ Level AA minimum ဖြစ်ပြီး large text အတွက် 3:1 threshold ရှိပါတယ်။ User interface controls နဲ့ meaning ရှိတဲ့ graphics တွေအတွက်လည်း adjacent colors နဲ့ 3:1 contrast လိုအပ်ပါတယ်။ ([W3C][14])

Rule—

```text
“မျက်လုံးနဲ့ကြည့်ရင်ရပါတယ်” လို့မဆုံးဖြတ်ပါနဲ့။
Contrast checker နဲ့တိုင်းပါ။
```

ဒီ lab မှာ—

```text
Dark text on light background
White text on dark blue button
Text labels in status badges
Strong focus outline
```

တွေကိုသုံးမယ်။

# 16. Form Labels

Input တစ်ခုချင်းစီမှာ visible label ရှိရမယ်။

မကောင်း—

```jsx
<input placeholder="နာမည်" />
```

ကောင်း—

```jsx
<label htmlFor="attendee-name">

  type="text"
```

# 17. Help Text နဲ့ Error Message

Accessible form field example—

```jsx
import { useId } from "react";

function NameField({ name, error, onChange }) {
  const inputId = useId();
  const helpId = useId();
  const errorId = useId();

  const describedBy = error
    ? `${helpId} ${errorId}`
    : helpId;

  return (
    <div>
      <label htmlFor={inputId}>
        နာမည်
      </label>

      <p id={helpId}>
        မှတ်ပုံတင်ရာမှာသုံးမယ့် နာမည်ကိုရေးပါ။
      </p>

      <input
        id={inputId}
        name="name"
        value={name}
        onChange={onChange}
        aria-describedby={describedBy}
        aria-invalid={Boolean(error)}
      />

      {error && (
        <p id={errorId} role="alert">
          {error}
        </p>
      )}
    </div>
  );
}
```

```text

+ Error icon
+ Error text
+ aria-invalid
+ error description association
```

လို အချက်များစွာနဲ့ဖော်ပြတာကောင်းတယ်။

# 19. Native HTML Before ARIA

ARIA က HTML မပေးနိုင်တဲ့ accessibility semantics ကိုဖြည့်ဖို့သုံးတယ်။ Native HTML ရှိရင် native HTML ကိုအရင်သုံးပါ။

မကောင်း—

```jsx
<div role="button" tabIndex={0}>
  Register
</div>
```

ကောင်း—

```jsx
<button type="button">
  Register လုပ်မည်
</button>
```

မကောင်း—

```jsx
<div role="navigation">
  ...
</div>
```

ကောင်း—

```jsx
<nav aria-label="အဓိက navigation">
  ...
</nav>
```

ARIA role တစ်ခုထည့်တာဟာ element က အဲဒီ role ရဲ့ expected behavior အားလုံးကိုလုပ်မယ်ဆိုတဲ့ “promise” လိုဖြစ်တယ်။ ARIA က native semantics ကို override လုပ်နိုင်တာကြောင့် မှားသုံးရင် accessibility ကိုပိုဆိုးစေနိုင်ပါတယ်။ ([W3C][8])

Rule—

```text
Native HTML first.
ARIA only when needed.
ARIA does not repair bad interaction design automatically.
```

# 21. Dynamic Status Messages

React app မှာ state ပြောင်းပေမယ့် keyboard focus ကိုမရွှေ့တဲ့ updates တွေရှိတယ်။

ဥပမာ—

```text

Result count changed

Form validation result
```

Screen reader user ကို update သိစေဖို့ `role="status"` သုံးနိုင်တယ်။

```jsx
<p role="status" aria-atomic="true">
  {message}
</p>
```

```jsx
<section role="alert">
  Event များကိုဖွင့်မရပါ။
```

```text
Normal update → role="status"
Urgent error → role="alert"

Announcement များလွန်းရင် user ကိုအနှောင့်အယှက်ဖြစ်စေနိုင်တယ်။
```

# 22. Touch Target Size

Button/link target တွေသေးလွန်းရင် touch, tremor, motor limitation ရှိသူတွေအတွက်နှိပ်ရခက်တယ်။

WCAG 2.2 Level AA Target Size Minimum မှာ target ကို အခြေခံအားဖြင့် အနည်းဆုံး 24×24 CSS pixels ဖြစ်စေဖို့သတ်မှတ်ထားပြီး spacing နဲ့ဆိုင်တဲ့ exceptions အချို့ရှိပါတယ်။ ([W3C][20])

ဒီ course မှာပိုအဆင်ပြေဖို့—

```css
button,
a {
  min-height: 44px;
}
```

လို generous target size ကိုအသုံးပြုမယ်။

```text
24×24
= WCAG minimum baseline

44×44 ခန့်
= ပိုနှိပ်လွယ်တဲ့ course design practice
```

# 23. Reduced Motion

Animation ကြောင့် လူတချို့မှာ မူးဝေခြင်း၊ ခေါင်းကိုက်ခြင်းနဲ့ distraction ဖြစ်နိုင်တယ်။ Non-essential motion ကိုရှောင်ပြီး user ရဲ့ reduced-motion preference ကိုလေးစားသင့်ပါတယ်။ ([W3C][21])

CSS example—

```css
.loading-dot {
  animation: spin 1s linear infinite;
}

@media (prefers-reduced-motion: reduce) {
  .loading-dot {
    animation: none;
  }
}
```

Flashing effect တွေကိုလည်းမသုံးသင့်ပါ။ WCAG guidance က တစ်စက္ကန့်အတွင်း flash သုံးကြိမ်ထက်ပိုမဖြစ်စေရန်သတိပေးထားပါတယ်။ ([W3C][22])

# 24. Zoom, Text Resize, Reflow

Page ကို zoom လုပ်တဲ့အခါ—

```text
Text မပျောက်ရ
Button label မဖြတ်ရ
Content မထပ်ရ
Functionality မပျောက်ရ
ပုံမှန်စာဖတ်ဖို့ horizontal + vertical နှစ်ဖက်စလုံး scroll မလိုရ
```

WCAG 2.2 Level AA Resize Text criterion မှာ text ကို 200% အထိချဲ့တဲ့အခါ content/functionality မပျောက်ရဘူးလို့သတ်မှတ်ထားပါတယ်။ Reflow criterion က horizontal-language content တွေအတွက် 320 CSS pixel width equivalent မှာ အဓိပ္ပါယ်မပျောက်ဘဲ ပုံမှန်အားဖြင့် two-dimensional scrolling မလိုစေဖို့ရည်ရွယ်ထားပါတယ်။ ([W3C][23])

Manual test—

```text
1. Browser zoom ကို 200% လုပ်ပါ
2. Text clip/overlap ဖြစ်လားစစ်ပါ
3. Button text ဖြတ်သွားလားစစ်ပါ
4. Filter chips အသုံးပြုလို့ရသေးလားစစ်ပါ
5. Event cards single column သို့ reflow ဖြစ်လားစစ်ပါ
6. Functionality မပျောက်ကြောင်းစစ်ပါ
```

```jsx
      selectedEvent.capacity - selectedEvent.registeredCount;

    if (remainingSeats <= 0) {
      setMessage(
        `${selectedEvent.title} သည် seats ပြည့်ပြီးပါပြီ။`
      );
      return;
    }

    const nextRemainingSeats = remainingSeats - 1;

    setEventList((currentEvents) =>
      currentEvents.map((event) => {
        if (event.id !== eventId) {
          return event;
        }

        return {
          ...event,
          registeredCount: event.registeredCount + 1
        };
      })
    );

    setViewStatus(READY);

    setMessage(
      nextRemainingSeats === 0
        ? `${selectedEvent.title} အတွက် registration အောင်မြင်ပြီး ယခု seats ပြည့်သွားပါပြီ။`
        : `${selectedEvent.title} အတွက် registration အောင်မြင်ပါသည်။ ကျန်နေရာ ${nextRemainingSeats} ခု။`
    );
  }

  function handleSelectCategory(category) {
    setSelectedCategory(category);
    setViewStatus(READY);

    setMessage(
      category === ALL_CATEGORIES
        ? "Category filter ကိုဖယ်ရှားပြီး event အားလုံးပြထားသည်။"
        : `${category} category ကိုရွေးထားသည်။`
    );
  }

  function handleToggleOpenOnly() {
    const nextValue = !showOnlyOpen;

    setShowOnlyOpen(nextValue);
    setViewStatus(READY);

    setMessage(
      nextValue
        ? "Open events only filter ကိုဖွင့်ထားသည်။"
        : "Open events only filter ကိုပိတ်ထားသည်။"
    );
  }

  function handleResetFilters() {
    setSelectedCategory(ALL_CATEGORIES);
    setShowOnlyOpen(false);
    setViewStatus(READY);
    setMessage("Filters အားလုံးကို reset လုပ်ပြီးပါပြီ။");
  }

  function handleResetData() {
    setEventList(initialEvents);
    setSelectedCategory(ALL_CATEGORIES);
    setShowOnlyOpen(false);
    setViewStatus(READY);
    setMessage("Event data ကိုမူလအခြေအနေသို့ပြန်ထားပြီးပါပြီ။");
  }

  function handleSimulateLoading() {
    setViewStatus(LOADING);
    setMessage("Loading state ကိုပြထားသည်။");
  }

  function handleSimulateError() {
    setViewStatus(ERROR);
    setMessage("Error state ကိုပြထားသည်။");
  }

  function handleRestoreView() {
    setViewStatus(READY);
    setMessage("Event list ကိုပြန်ဖွင့်ပြီးပါပြီ။");
  }

  return (
    <>
      <a className="skip-link" href="#main-content">
        အဓိကအကြောင်းအရာသို့ ကျော်ရန်
      </a>

      <header className="site-header">
        <div className="site-header-inner">
          <p className="brand-name" lang="en">
            Event Access Lab
          </p>

          <nav aria-label="စာမျက်နှာအတွင်း လင့်များ">
            <a href="#events-heading">Event များ</a>
            <a href="#accessibility-checklist-heading">
              Accessibility checklist
            </a>
          </nav>
        </div>
      </header>

      <main
        id="main-content"
        className="page"
        tabIndex={-1}
      >
        <PageHeader
          eyebrow="Part 6 — Frontend Foundation"
          title="Frontend Accessibility Basics"
          description="Semantic HTML, keyboard operation, visible focus နဲ့ accessible state messages တွေကို React app ထဲတည်ဆောက်ပါမယ်။"
        />

        <StatsSummary
          totalEvents={eventList.length}
          totalCapacity={totalCapacity}
          totalRegistered={totalRegistered}
          openEventCount={openEvents.length}
        />

        <ActionPanel
          categories={categories}
          message={message}
          selectedCategory={selectedCategory}
          showOnlyOpen={showOnlyOpen}
          viewStatus={viewStatus}
          onSelectCategory={handleSelectCategory}
          onToggleOpenOnly={handleToggleOpenOnly}
          onResetFilters={handleResetFilters}
          onResetData={handleResetData}
          onSimulateLoading={handleSimulateLoading}
          onSimulateError={handleSimulateError}
          onRestoreView={handleRestoreView}
        />

        <EventContent
          events={visibleEvents}
          openEventCount={openEvents.length}
          hasActiveFilters={hasActiveFilters}
          viewStatus={viewStatus}
          onRegister={handleRegister}
          onResetFilters={handleResetFilters}
          onRestoreView={handleRestoreView}
        />

        <AccessibilityChecklist />
      </main>

      <footer className="site-footer">
        <p>
          Frontend Accessibility Basics — keyboard, semantics,
          focus, states.
        </p>
      </footer>
    </>
  );
}

function CategoryFilter({
  categories,

  onSelectCategory

function EventContent({
  events,

  onRegister,
  onResetFilters,
  onRestoreView
}) {
  return (
    <section

    >

          onRegister={onRegister}
          onResetFilters={onResetFilters}
        />
      )}

function EventList({
  events,
  hasActiveFilters,
  onRegister,
  onResetFilters
}) {
  if (events.length === 0) {
    return (
      <EmptyState
        title={
          hasActiveFilters
            ? "ကိုက်ညီသော Event မရှိပါ။"
            : "Event မရှိသေးပါ။"
        }
        message={
          hasActiveFilters
            ? "Category filter သို့ open-only filter ကိုပြန်ပြောင်းပါ။"
            : "Event အသစ်ထည့်ပြီးမှ ဒီနေရာမှာပြပါမယ်။"
        }
        actionText={
          hasActiveFilters ? "Filters reset" : undefined
        }
        onAction={
          hasActiveFilters ? onResetFilters : undefined
        }
      />
    );
  }

  return (
    <ul className="event-grid" aria-label="Event list">
      {events.map((event) => (
        <EventCard
          key={event.id}
          event={event}
          onRegister={onRegister}
        />
      ))}
    </ul>
  );
}

function EventCard({ event, onRegister }) {
  const remainingSeats =
    event.capacity - event.registeredCount;

  const isFull = remainingSeats === 0;
  const availabilityId = `${event.id}-availability`;

  const buttonText = isFull
    ? "Seats ပြည့်ပါပြီ"
    : "Register လုပ်မည်";

  return (
    <li className="event-list-item">
      <article
        className="event-card"
        aria-labelledby={`${event.id}-title`}
      >
        <div className="event-card-header">
          <div>
            <p className="event-category">
              {event.category}
            </p>

            <h3 id={`${event.id}-title`}>
              {event.title}
            </h3>
          </div>

          <StatusBadge
            id={availabilityId}
            isFull={isFull}
            remainingSeats={remainingSeats}
          />
        </div>

        <dl className="event-meta">
          <EventMetaItem
            label="ရက်စွဲ"
            value={event.date}
          />

          <EventMetaItem
            label="နေရာ"
            value={event.location}
          />

          <EventMetaItem
            label="ကျန်ရှိသောနေရာ"
            value={remainingSeats}
          />
        </dl>

        {remainingSeats > 0 && remainingSeats <= 10 && (
          <p className="few-seats-note">
            သတိပြုရန် — နေရာအနည်းငယ်သာကျန်ပါသည်။
          </p>
        )}

        <PrimaryButton
          disabled={isFull}
          ariaLabel={`${event.title} အတွက် ${buttonText}`}
          ariaDescribedBy={availabilityId}
          onClick={() => onRegister(event.id)}
        >
          {buttonText}
        </PrimaryButton>
      </article>
    </li>
  );
}

function EventMetaItem({ label, value }) {
  return (
    <div>
      <dt>{label}</dt>
      <dd>{value}</dd>
    </div>
  );
}

function StatusBadge({
  id,
  isFull,
  remainingSeats
}) {
  if (isFull) {
    return (
      <span
        id={id}
        className="status-badge status-badge-full"
      >
        Full — seats ပြည့်
      </span>
    );
  }

  if (remainingSeats <= 10) {
    return (
      <span
        id={id}
        className="status-badge status-badge-warning"
      >
        Few seats — {remainingSeats} ခု
      </span>
    );
  }

  return (
    <span
      id={id}
      className="status-badge status-badge-open"
    >
      Open — {remainingSeats} ခု
    </span>
  );
}

function PrimaryButton({
  disabled = false,
  ariaLabel,
  ariaDescribedBy,
  onClick,
  children
}) {
  return (
    <button
      className="primary-button"
      type="button"
      disabled={disabled}

      aria-describedby={ariaDescribedBy}
      onClick={onClick}
    >
      {children}
    </button>
  );
}
```

# 31. Code ထဲက Accessibility Improvements

## 31.1 Skip link

```jsx
<a className="skip-link" href="#main-content">
  အဓိကအကြောင်းအရာသို့ ကျော်ရန်
</a>
```

Keyboard focus ရချိန်မှာပေါ်တယ်။

## 31.2 Page language

```html
<html lang="my">
```

## 31.3 Landmarks

```jsx
<header>
<nav>
<main>
<footer>
```

## 31.4 Heading hierarchy

```text
h1 — Page title
h2 — Summary / Filters / Events / Checklist
h3 — Event title / Loading / Error / Empty state
```

## 31.5 Filter state

```jsx
aria-pressed={isSelected}
```

Color မဟုတ်ဘဲ visible check mark လည်းပါသေးတယ်။

```jsx
<span aria-hidden="true">
  {isSelected ? "✓ " : ""}
</span>
```

## 31.6 Dynamic status

```jsx
<p role="status" aria-atomic="true">
  {message}
</p>
```

## 31.7 Controls relationship

```jsx
aria-controls="event-results"
```

## 31.8 Loading state

```jsx
aria-busy={viewStatus === LOADING}
```

## 31.9 Semantic event list

```jsx
<ul>
  <li>
    <article>
```

## 31.10 Descriptive registration button

```jsx
aria-label={`${event.title} အတွက် ${buttonText}`}
```

Visible label `"Register လုပ်မည်"` က accessible name ထဲလည်းပါနေတယ်။ Visible label နဲ့ programmatic name ကိုကိုက်ညီစေတာက speech-input users အတွက်အရေးကြီးပါတယ်။ ([W3C][25])

# 46. Common Errors

## Error 1 — `outline: none`

မလုပ်ရ—

```css
button:focus {
  outline: none;
}
```

ပြင်ရန်—

```css
button:focus-visible {
  outline: 3px solid #111827;
  outline-offset: 3px;
}
```

---

## Error 2 — `<div onClick>`

မကောင်း—

```jsx
<div onClick={handleClick}>Reset</div>
```

ကောင်း—

```jsx
<button type="button" onClick={handleClick}>
  Reset
</button>
```

---

## Error 3 — Placeholder ကို Label အဖြစ်သုံးခြင်း

မကောင်း—

```jsx
<input placeholder="နာမည်" />
```

ကောင်း—

```jsx
<label htmlFor="name">နာမည်</label>
<input id="name" />
```

---

## Error 4 — Color-only State

မကောင်း—

```text
Green = Open
Red = Full
```

ကောင်း—

```text
Green + Open
Red + Full
Check mark + aria-pressed
```

---

## Error 5 — Wrong Heading Level

မကောင်း—

```text
h1 → h4 → h2
```

ကောင်း—

```text
h1 → h2 → h3
```

---

## Error 6 — Redundant ARIA

မလိုအပ်—

```jsx
<button role="button">
```

Native button မှာ button semantics ရှိပြီးသားပါ။

ကောင်း—

```jsx
<button type="button">
```

---

## Error 7 — aria-label က Visible Label နဲ့မကိုက်

မကောင်း—

```jsx
<button aria-label="Send">
  Register လုပ်မည်
</button>
```

ကောင်း—

```jsx
<button aria-label="AI Workshop အတွက် Register လုပ်မည်">
  Register လုပ်မည်
</button>
```

Accessible name ထဲမှာ visible label ပါသင့်တယ်။

---

## Error 8 — `aria-live` အရာတိုင်းမှာထည့်ခြင်း

Live announcements များလွန်းရင် screen reader user အတွက် noise ဖြစ်တယ်။

```text
Meaningful status changes → announce
Static descriptions → do not announce
```

---

## Error 9 — Positive Tab Index

မကောင်း—

```jsx
<button tabIndex={5}>...</button>
```

DOM order ကို logical ပြင်ပါ။

---

## Error 10 — Fixed Height ကြောင့် Zoom မှာ Text ဖြတ်

မကောင်း—

```css
.button {
  height: 32px;
  overflow: hidden;
}
```

ပိုကောင်း—

```css
.button {
  min-height: 44px;
  padding: 10px 16px;
}
```

```text
Accessibility
= Correct structure
+ Keyboard operation
+ Visible focus
+ Clear labels
+ Understandable states
+ Flexible visual presentation
+ Assistive technology compatibility
```

WCAG principle လေးခု—

```text
Perceivable
Operable
Understandable
Robust
```

Frontend မှာအရေးကြီးဆုံး rules—

```text
Semantic HTML ကိုအရင်သုံးပါ။

Input တိုင်း visible label ရှိရမယ်။
Dynamic status ကိုလိုအပ်တဲ့အခါ announce လုပ်ပါ။
ARIA ကိုလိုမှသုံးပါ။
```
