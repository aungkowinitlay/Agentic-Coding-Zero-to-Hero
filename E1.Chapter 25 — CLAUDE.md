## 2. Beginner Explanation

`CLAUDE.md` ဆိုတာ Claude Code ကို project တစ်ခုအကြောင်း အမြဲသတိရစေချင်တဲ့ instructions တွေကိုရေးထားတဲ့ Markdown file ပါ။

# 4. CLAUDE.md သည် Memory ဖြစ်ပြီး Enforcement မဟုတ်

`CLAUDE.md` သည် Claude အတွက် context / instruction ဖြစ်ပါတယ်။ Hard security enforcement မဟုတ်ပါဘူး။

ဥပမာ `CLAUDE.md` ထဲမှာ—

```md
- Do not read .env files.
```

ရေးထားရင် Claude ကို “မဖတ်သင့်ဘူး” လို့လမ်းညွှန်ထားတာပါ။ ဒါပေမယ့် tool-level block လုပ်ချင်ရင် `.claude/settings.json` ထဲ permission deny rule သို့မဟုတ် PreToolUse hook လိုတယ်။

Claude Code official docs က `CLAUDE.md` နဲ့ auto memory နှစ်ခုလုံးကို conversation စချိန် context အဖြစ် load လုပ်ပြီး “enforced configuration” မဟုတ်ကြောင်းတိတိကျကျဖော်ပြထားပါတယ်။ Action တစ်ခုကို Claude ဘာဆုံးဖြတ်ဆုံးဖြတ် block လုပ်ချင်ရင် PreToolUse hook လို deterministic mechanism သုံးရန် docs ကအကြံပြုထားပါတယ်။ ([Claude][1])

Mental model—

```text
CLAUDE.md
= soft instruction / memory

permissions
= hard tool gate

hooks
= deterministic automation / guardrail

git diff
= human review evidence

human approval
= final decision
```

# 5. CLAUDE.md ကို ဘယ်နေရာမှာထားမလဲ

Claude Code docs အရ `CLAUDE.md` files တွေကို scope အလိုက်နေရာအမျိုးမျိုးမှာထားနိုင်ပါတယ်။ Project-level instruction အတွက် root `./CLAUDE.md` သို့မဟုတ် `./.claude/CLAUDE.md` ကိုသုံးနိုင်ပြီး personal project-specific preferences အတွက် `./CLAUDE.local.md` ကိုသုံးနိုင်ပါတယ်။ User-wide instruction အတွက် `~/.claude/CLAUDE.md` ကိုသုံးနိုင်ပါတယ်။ ([Claude][1])

```text
agentic-zero-to-hero/

├── AGENTS.md              # generic agent instructions

    ├── settings.json      # permissions/hooks/settings
```

Course rule—

```text
Project rules
→ CLAUDE.md

→ CLAUDE.local.md, gitignored

→ AGENTS.md

Hard safety gates

```

# 6. CLAUDE.md Loading Behavior

Claude Code က current working directory ကနေ directory tree အပေါ်ဘက်သို့တက်ပြီး `CLAUDE.md` နဲ့ `CLAUDE.local.md` files တွေကိုရှာဖတ်ပါတယ်။ ဥပမာ `foo/bar/` ထဲကနေ Claude Code run လုပ်ရင် `foo/bar/CLAUDE.md`, `foo/CLAUDE.md`, အဲဒီ directory levels တွေမှာရှိတဲ့ `CLAUDE.local.md` files တွေကို context ထဲထည့်နိုင်ပါတယ်။ Official docs အရ discovered files တွေကို overwrite မလုပ်ဘဲ concatenate လုပ်ပြီး context ထဲထည့်ပါတယ်။ ([Claude][1])

အရေးကြီးတဲ့ lesson—

```text
Claude Code ကို project root မှာ run လုပ်ပါ။
Wrong folder ထဲမှာ run လုပ်ရင် wrong instructions loaded ဖြစ်နိုင်တယ်။
```

Session စချိန်သုံးရမယ့် safety check—

```bash
pwd
ls
git status
```

ပြီးမှ—

```bash
claude
```

# 9. Import Syntax — `@path`

`CLAUDE.md` ထဲမှာ `@path/to/file` syntax နဲ့ additional files တွေကို import လုပ်နိုင်ပါတယ်။ Official docs အရ relative path တွေဟာ import line ပါတဲ့ file နဲ့ relative ဖြစ်ပြီး, import parsing က Markdown code spans / fenced code blocks ထဲက `@` ကို import မလုပ်ပါ။ Imported files တွေဟာ session startup context ထဲ expand/load ဖြစ်ပါတယ်။ ([Claude][1])

ဥပမာ—

```md
@AGENTS.md

## Additional Project Instructions

See @COURSE_DECISIONS.md for course-wide technical decisions.
See @resources/prompts/safe-agent-startup.md for the safe startup prompt.
```

သတိ—

```text
Import = organization အတွက်ကောင်း
Import = context token အသုံးပြုမှုလျော့တာမဟုတ်
```

Claude docs က imported files တွေသည် context ထဲ expand/load ဖြစ်တာကြောင့် file ခွဲထားတာက organization အတွက်ကောင်းပေမယ့် context usage ကိုလျော့စေခြင်းမဟုတ်ကြောင်းဖော်ပြထားပါတယ်။ ([Claude][1])

# 11. Effective CLAUDE.md Rules

Official docs က `CLAUDE.md` instructions တွေကို specific/concrete/checkable ဖြစ်အောင်ရေးရန်အကြံပြုထားပါတယ်။ ဥပမာ “Format code properly” ထက် “Use 2-space indentation” လို rule ကပိုကောင်းပြီး “Test your changes” ထက် “Run npm test before committing” လို rule ကပိုတိကျပါတယ်။ ([Claude][1])

Bad rule—

```md
- Write clean code.
```

Good rule—

```md
- Prefer small functions with descriptive names.
- If a function has more than one responsibility, propose a refactor before editing.
```

Bad rule—

```md
- Test your work.
```

Good rule—

```md
- For frontend form changes, provide manual tests for empty required fields, invalid input, valid submit, and browser console errors.
```

Bad rule—

```md
- Be safe.
```

Good rule—

```md
- Do not read or edit `.env` files. Use `.env.example` only for variable names. Never print secret values.
```

# 14. Recommended CLAUDE.md for This Course

ဒီ course workspace အတွက် `CLAUDE.md` ကို `AGENTS.md` import pattern နဲ့ရေးမယ်။

```md
# CLAUDE.md

@AGENTS.md

## Claude Code Specific Rules

### Planning

- Use plan mode for non-trivial tasks.
- Do not edit files before presenting a plan when the task touches more than one file.
- Ask for approval before implementing feature-level changes.

### Permissions and Safety

- Use `/permissions` when permission behavior is unclear.
- Do not request or reveal secret values.
- Do not read or edit `.env` files.
- Do not install packages without approval.
- Do not commit, push, or deploy without explicit approval.

### Hooks

- Use `/hooks` to verify configured hooks after changing `.claude/settings.json`.
- If hooks block a task, explain what was blocked and ask for a safer path.

### Reporting

After edits, respond with:

1. Changed files
2. Summary of changes
3. Verification steps
4. Known risks
5. Suggested commit message

### Definition of Done

A task is done only when:

- Scope is respected.
- Acceptance criteria are checked.
- Verification steps are provided.
- Changed files are summarized.
- Known risks or assumptions are listed.
- Human review is still required before commit, push, or deploy.
```

ဒီ template က concise ဖြစ်ပြီး Claude Code-specific rules ကိုသာထည့်ထားတယ်။ Common agent rules တွေကို `AGENTS.md` ထဲထားပြီး `@AGENTS.md` နဲ့ import လုပ်ထားတယ်။

# 15. Recommended AGENTS.md for Import

`CLAUDE.md` က import လုပ်မယ့် `AGENTS.md` ကို ဒီလိုထားနိုင်တယ်။

```md
# AGENTS.md

## Project Overview

This repository is the Agentic Coding Zero to Hero course workspace.

## Repository Layout

- `notes/` — course chapter notes
- `labs/` — lab outputs
- `playground/` — small experiments
- `projects/myaree-store/` — warm-up project
- `projects/event-management/` — final capstone project
- `resources/` — templates, prompts, references
- `.claude/` — Claude Code settings, agents, skills, commands, hooks

## Safety Rules

- Never read or edit `.env` files.
- Never print secrets or tokens.
- Never edit `node_modules/`, `dist/`, or `build/`.
- Ask before installing packages.
- Ask before destructive Git commands.
- Ask before deployment or cloud actions.

## Workflow Rules

- Start with read-only analysis when entering unfamiliar code.
- For non-trivial tasks, plan before editing.
- Before editing, check `git status`.
- If the working tree is not clean, summarize current changes before proceeding.
- Make minimal scoped changes.
- After editing, summarize changed files.
- Before saying Done, provide verification steps.
- Suggest commit messages but do not commit without approval.

## Definition of Done

A task is done only when:

- Scope is respected.
- Acceptance criteria are checked.
- Verification steps are provided.
- Changed files are summarized.
- Known risks or assumptions are listed.
- Human review is still required before commit/deploy.
```

# 16. .claude/rules/ ကို ဘယ်အချိန်သုံးမလဲ

Project ကြီးလာတဲ့အခါ `CLAUDE.md` ထဲမှာ rules အများကြီးထည့်တာမကောင်းဘူး။ `.claude/rules/` ထဲမှာ topic-specific rules ခွဲထားပါ။

Claude docs အရ `.claude/rules/` directory ထဲမှာ Markdown files တွေထားပြီး rules တွေကို modular, maintainable ဖြစ်အောင်ခွဲနိုင်ပါတယ်။ Rules တွေကို path-specific ဖြစ်အောင် YAML frontmatter `paths` field နဲ့ scope ချနိုင်ပြီး matching files တွေနဲ့အလုပ်လုပ်တဲ့အခါ context ထဲ load ဖြစ်နိုင်ပါတယ်။ ([Claude][1])

Example—

```text
.claude/rules/
├── frontend.md
├── backend.md
├── testing.md
└── security.md
```

Path-specific example—

```md
---
paths:
  - "projects/event-management/apps/web/**/*.{js,jsx,ts,tsx}"
---

# Frontend Rules

- Public UI text should be Myanmar unless the task says otherwise.
- Form inputs must have visible labels.
- Do not use placeholder text as the only label.
```

# 17. CLAUDE.md vs .claude/rules/

| Use case                       | Best place                  |
| ------------------------------ | --------------------------- |
| Every-session project overview | `CLAUDE.md`                 |
| Universal safety reminders     | `AGENTS.md` / `CLAUDE.md`   |
| Frontend-only conventions      | `.claude/rules/frontend.md` |
| Backend-only conventions       | `.claude/rules/backend.md`  |
| Test conventions               | `.claude/rules/testing.md`  |
| Long workflow procedures       | skills / commands           |
| Hard safety blocks             | permissions / hooks         |
| Product acceptance criteria    | `SPEC.md`                   |
| One-time task instruction      | current prompt              |

Rule—

```text
CLAUDE.md ကို main index / project contract လိုထားပါ။
အသေးစိတ် rules တွေကို .claude/rules/ ထဲခွဲပါ။
```

```md
# CLAUDE.md

## Claude Code Specific Rules

### Planning

- Use plan mode for non-trivial tasks.
- Do not edit files before presenting a plan when the task touches more than one file.
- Ask for approval before implementing feature-level changes.

### Permissions and Safety

- Use `/permissions` when permission behavior is unclear.
- Do not request or reveal secret values.
- Do not read or edit `.env` files.
- Do not install packages without approval.
- Do not commit, push, or deploy without explicit approval.

### Hooks

- Use `/hooks` to verify configured hooks after changing `.claude/settings.json`.
- If hooks block a task, explain what was blocked and ask for a safer path.

### Reporting

After edits, respond with:

1. Changed files
2. Summary of changes
3. Verification steps
4. Known risks
5. Suggested commit message

### Definition of Done

A task is done only when:

- Scope is respected.
- Acceptance criteria are checked.
- Verification steps are provided.
- Changed files are summarized.
- Known risks or assumptions are listed.
- Human review is still required before commit, push, or deploy.
```

# 25. Common Errors

## Error 1 — CLAUDE.md ထဲ secret ထည့်ခြင်း

မလုပ်ရ—

```md
OPENAI_API_KEY=sk-...
DATABASE_URL=postgres://user:password@host/db
```

`CLAUDE.md` ကို Git commit လုပ်မှာဖြစ်လို့ secret မပါရ။

---

## Error 2 — CLAUDE.md ကို အရမ်းရှည်အောင်ထားခြင်း

Files ရှည်လွန်းရင် context ပိုစားပြီး adherence လျော့နိုင်ကြောင်း official docs ကသတိပေးထားပါတယ်။ 200 lines အောက်ထားရန် target လုပ်ပါ။ ([Claude][1])

ပြင်နည်း—

```text
Core rules only in CLAUDE.md.
Detailed path-specific rules in .claude/rules/.
Long procedures in skills/commands.
```

---

## Error 3 — AGENTS.md ရှိတာနဲ့ Claude Code ဖတ်မယ်လို့ထင်ခြင်း

Claude Code က `CLAUDE.md`, not `AGENTS.md`, ကိုဖတ်တာဖြစ်ပြီး AGENTS.md သုံးချင်ရင် `CLAUDE.md` ထဲ `@AGENTS.md` import လုပ်ရန် official docs ကဖော်ပြထားပါတယ်။ ([Claude][1])

---

## Error 4 — CLAUDE.md ကို enforcement လို့ယူဆခြင်း

`CLAUDE.md` သည် context/instruction ဖြစ်ပြီး strict compliance guarantee မဟုတ်ပါ။ Secret access block, dangerous command block တို့အတွက် permissions/hooks လိုတယ်။ ([Claude][1])

---

## Error 5 — Conflicting rules ထားခြင်း

ဥပမာ—

```md
- Always run npm test automatically.
- Ask before running npm commands.
```

ပြင်—

```md
- After code changes, propose running npm test and ask for approval before running it.
```

---

## Error 6 — CLAUDE.local.md ကို commit မိခြင်း

`CLAUDE.local.md` သည် personal/local preference အတွက်ဖြစ်လို့ `.gitignore` ထဲထည့်ပါ။ ([Claude][1])

---

## Error 7 — One-time prompt ကို CLAUDE.md ထဲထည့်ခြင်း

မကောင်း—

```md
- Today, edit only app.js.
```

ဒါက current task prompt ထဲသင့်တာပါ။ Project-wide memory မဟုတ်ပါ။
