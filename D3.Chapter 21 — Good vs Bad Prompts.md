# Chapter 21 — Good vs Bad Prompts

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့ရေးနိုင်ရပါမယ်။

```text
[ ] Bad prompt နဲ့ Good prompt ကွာခြားချက်ကိုရှင်းပြနိုင်မယ်
[ ] Vague prompt ကို structured prompt အဖြစ်ပြောင်းနိုင်မယ်
[ ] Prompt တစ်ခုမှာ context, goal, scope, non-goals, criteria, verification ပါ/မပါစစ်နိုင်မယ်
[ ] Agent ကို task ကြီးလွန်းတာ မခိုင်းမိအောင် prompt ပြင်နိုင်မယ်
[ ] Agent ကို package install, file edit, commit, deploy မလုပ်ခင် approval တောင်းအောင်ခိုင်းနိုင်မယ်
[ ] Debug prompt, feature prompt, review prompt, verification prompt တွေကိုခွဲနိုင်မယ်
[ ] Bad prompt smells တွေကိုဖမ်းနိုင်မယ်
[ ] Prompt quality scorecard သုံးပြီး prompt တစ်ခုကိုအမှတ်ပေးနိုင်မယ်
[ ] Prompt ကို iteration လုပ်ပြီး ပိုကောင်းအောင်ပြင်နိုင်မယ်
[ ] Agent ကို “ဘာလုပ်ရမလဲ” တင်မက “ဘာမလုပ်ရဘူး” ပါညွှန်နိုင်မယ်
```

Chapter 18 မှာ Prompt Anatomy ကိုသင်ခဲ့တယ်။ Chapter 19 မှာ Plan Before Code ကိုသင်ခဲ့တယ်။ Chapter 20 မှာ Task Decomposition ကိုသင်ခဲ့တယ်။ ဒီ Chapter 21 မှာတော့ အဲဒီသင်ခန်းစာတွေကိုပေါင်းပြီး **Bad Prompt ကို Good Prompt အဖြစ်ပြောင်းနည်း** ကို ဥပမာများများနဲ့လေ့ကျင့်မယ်။

---

## 2. Beginner Explanation

Agentic Coding မှာ prompt က “AI ကိုမေးတဲ့စာ” မဟုတ်ပါဘူး။ Prompt က—

```text
Agent ကိုပေးတဲ့ အလုပ်အမိန့်စာ
```

ပါ။

Bad prompt က အလုပ်အမိန့်စာမရှင်းတဲ့အခါဖြစ်တယ်။

မကောင်းတဲ့ prompt—

```text
ဒီ app ကိုကောင်းအောင်လုပ်ပါ။
```

ဒီ prompt ကို Agent က ဘယ်လိုနားလည်မလဲ မသိနိုင်ဘူး။

Agent တွေးနိုင်တာ—

```text
UI design ပြင်ရမလား?
Bug fix လုပ်ရမလား?
Backend API ထည့်ရမလား?
Database ထည့်ရမလား?
Package install လုပ်ရမလား?
Performance optimize လုပ်ရမလား?
README ပြင်ရမလား?
```

Good prompt ကတော့ Agent ကိုရှင်းရှင်းလင်းလင်းပြောတယ်။

```text
ဒီ app ရဲ့ registration form validation message ကိုမြန်မာလိုပြင်ပါ။
Allowed file တစ်ဖိုင်ပဲပြင်ပါ။
package install မလုပ်ပါနဲ့။
Code မပြင်ခင် plan ပေးပါ။
ပြင်ပြီးရင် manual test cases နဲ့ git diff summary ပေးပါ။
```

Good prompt က Agent ကို လမ်းပြတဲ့မြေပုံလိုပါ။

```text
Bad prompt  = “မြို့ထဲသွားပါ”
Good prompt = “ရန်ကုန်မြို့၊ ဆူးလေဘုရားလမ်း၊ အမှတ် ၁၂၃ ကိုသွားပါ။ လမ်းမှာ ဆေးဆိုင်မဝင်ပါနဲ့။ ရောက်ရင် ဖုန်းဆက်ပါ။”
```

Agentic Coding မှာ prompt ကောင်းရင်—

```text
scope ထိန်းရလွယ်တယ်
Agent လမ်းမလွဲဘူး
Git diff သေးတယ်
Review လုပ်ရလွယ်တယ်
Bug နည်းတယ်
Verification ရှိတယ်
Rollback လုပ်ရလွယ်တယ်
```

Prompt မကောင်းရင်—

```text
Agent က file အများကြီးပြင်နိုင်တယ်
package မလိုဘဲ install လုပ်နိုင်တယ်
security risk ပါလာနိုင်တယ်
test/build မစစ်ဘဲ “done” လို့ပြောနိုင်တယ်
project ပျက်သွားရင် ပြန်ဆယ်ရခက်နိုင်တယ်
```

# 3. Good Prompt ဆိုတာဘာလဲ

Good prompt ဆိုတာ Agent အတွက်—

```text
ဘာလုပ်ရမလဲ
ဘာကြောင့်လုပ်ရမလဲ
ဘယ်နေရာမှာလုပ်ရမလဲ
ဘယ်နေရာမထိရဘူးလဲ
ဘယ်လိုပြီးပြီလို့စစ်မလဲ
ဘယ်အချိန် approval တောင်းရမလဲ
```

ရှင်းရှင်းပြောထားတဲ့ prompt ပါ။

Good prompt တစ်ခုမှာ အောက်ပါအရာတွေထဲက လိုအပ်တဲ့အပိုင်းတွေပါသင့်တယ်။

```text
Role
Mode
Context
Goal
Scope
Non-goals
Allowed files
Do-not-touch list
Constraints
Acceptance Criteria
Workflow
Verification
Output Format
Approval Gates
```

Good prompt example—

```text
Role:
သင်က careful frontend developer ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Context:
ဒီ app က static HTML/CSS/JS lab app ဖြစ်ပါတယ်။ Backend API မရှိပါ။

Goal:
Registration form မှာ phone field empty ဖြစ်ရင် မြန်မာလို error message ပြချင်ပါတယ်။

Scope:
Frontend validation only.

Allowed file:
- playground/chapter-11-coding-agent/app.js

Do not touch:
- package.json
- package-lock.json
- .env
- node_modules/**
- index.html
- style.css

Constraints:
- Package install မလုပ်ပါနဲ့။
- Existing submit flow မပြောင်းပါနဲ့။
- Function name မပြောင်းပါနဲ့။

Acceptance Criteria:
- Phone empty ဖြစ်ရင် “ဖုန်းနံပါတ်ဖြည့်ပါ။” ပြရမယ်။
- Name validation မပျက်ရဘူး။
- Valid form submit မပျက်ရဘူး။

Workflow:
1. app.js ကိုဖတ်ပါ။
2. ဘာပြင်မလဲ plan ပေးပါ။
3. Approval ရမှ edit လုပ်ပါ။
4. Manual test cases ပေးပါ။
5. git diff summary ပေးပါ။

Output Format:
## Plan
## Files to Change
## Risks
## Verification Steps
## Approval Needed
```

ဒီ prompt က Agent ကို လုပ်ရမယ့်အလုပ်ကို သေချာညွှန်ထားတယ်။

# 4. Bad Prompt ဆိုတာဘာလဲ

Bad prompt ဆိုတာ Agent အတွက်မရှင်းတဲ့ prompt ပါ။

Bad prompt ထဲမှာ များသောအားဖြင့်—

```text
Goal မရှင်း
Context မရှိ
Scope မရှိ
Non-goals မရှိ
Allowed files မရှိ
Verification မရှိ
Output format မရှိ
Approval gate မရှိ
“ကောင်းအောင်လုပ်” လို vague စကားများ
```

ပါဝင်တတ်တယ်။

Bad prompt examples—

```text
ဒီ app ကိုပြင်ပေး။
```

```text
Registration လုပ်ပေး။
```

```text
Error fix လုပ်။
```

```text
လိုတာ install လုပ်ပြီး ဆောက်ပေး။
```

```text
Production-ready ဖြစ်အောင်လုပ်။
```

```text
အကုန်ပြီးအောင်လုပ်ပြီး commit/push လုပ်ပေး။
```

ဒီ prompt တွေက Agent ကို အလွန်များတဲ့ freedom ပေးတယ်။ Freedom များတာက productivity မဟုတ်ပါဘူး။ Scope မရှင်းတဲ့ freedom က risk ပါ။

---

# 5. Bad Prompt Smells

Prompt တစ်ခုမကောင်းကြောင်းပြတဲ့ smell တွေကိုသိထားရမယ်။

```text
[ ] “ကောင်းအောင်လုပ်” လို့ပဲရေးထား
[ ] “အကုန်လုပ်” လို့ရေးထား
[ ] “လိုတာ install လုပ်” လို့ရေးထား
[ ] “ပြီးရင် commit/push/deploy လုပ်” လို့ရေးထား
[ ] Code မပြင်ခင် plan မတောင်းထား
[ ] Allowed files မပါ
[ ] Do-not-touch list မပါ
[ ] Acceptance criteria မပါ
[ ] Verification မပါ
[ ] Error details မပါ
[ ] Expected / Actual မပါ
[ ] Non-goals မပါ
[ ] Secret safety rule မပါ
[ ] Package install approval မပါ
```

Prompt smell တွေ့ရင် Agent ကိုမခိုင်းခင် ပြင်ပါ။

# 6. Good Prompt Checklist

Prompt မပေးခင် ဒီ checklist နဲ့စစ်ပါ။

```text
[ ] Role ပါလား?
[ ] Mode ပါလား?
[ ] Context ပါလား?
[ ] Goal ရှင်းလား?
[ ] Scope ရှင်းလား?
[ ] Non-goals ပါလား?
[ ] Allowed files/folders ပါလား?
[ ] Do-not-touch list ပါလား?
[ ] Constraints ပါလား?
[ ] Acceptance criteria စစ်လို့ရလား?
[ ] Workflow ပါလား?
[ ] Verification ပါလား?
[ ] Output format ပါလား?
[ ] Approval gates ပါလား?
```

Task သေးရင် checklist အားလုံးမလိုပါ။ ဒါပေမယ့် feature-level task, bug fix, refactor, package install, deployment, database, auth/security task တွေမှာ များများပါလေကောင်းလေပါ။

---

# 7. Bad → Good Transformation Formula

Bad prompt တစ်ခုကို Good prompt ပြောင်းတဲ့ formula—

```text
1. Vague request ကို Goal အဖြစ်ပြောင်း
2. Project background ကို Context အဖြစ်ထည့်
3. ဘယ်အထိလုပ်မလဲ Scope သတ်မှတ်
4. ဘာမလုပ်ရဘူးလဲ Non-goals ထည့်
5. ဘယ် files/folders ထိလို့ရလဲ Allowed list ထည့်
6. ဘယ် files/folders မထိရဘူးလဲ Do-not-touch ထည့်
7. စစ်လို့ရတဲ့ Acceptance Criteria ထည့်
8. Plan first / approval gate ထည့်
9. Verification ထည့်
10. Output format သတ်မှတ်
```

Example—

Bad—

```text
Filter လုပ်ပေး။
```

Good—

```text
Role:
သင်က careful frontend developer ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Context:
Event list page သည် static frontend lab ဖြစ်ပြီး events data ကို local array ထဲကနေပြနေပါတယ်။

Goal:
User က category dropdown ရွေးပြီး event cards ကို filter လုပ်နိုင်ရမယ်။

Scope:
Frontend only. Existing mock data ကိုသုံးပါ။

Non-goals:
- Backend API မထည့်ပါ
- Database မထည့်ပါ
- Package install မလုပ်ပါ
- Search box မထည့်ပါ

Allowed files:
- playground/event-list-filter/index.html
- playground/event-list-filter/style.css
- playground/event-list-filter/app.js

Do not touch:
- package.json
- .env
- node_modules/**

Acceptance Criteria:
- Category dropdown ပေါ်ရမယ်။
- “All” ရွေးရင် events အားလုံးပြရမယ်။
- Category တစ်ခုရွေးရင် အဲဒီ category events ပဲပြရမယ်။
- Result မရှိရင် empty state ပြရမယ်။
- Browser console error မရှိရဘူး။

Workflow:
Plan → approval → implement → manual tests → git diff summary.

Do not edit before approval.
```

# 8. Example 1 — “Website လုပ်ပေး”

## Bad Prompt

```text
Website တစ်ခုလုပ်ပေး။
```

## ဘာကြောင့်မကောင်းလဲ

```text
ဘာ website လဲမသိ
static website လား web app လားမသိ
HTML/CSS/JS လား React လားမသိ
page ဘယ်နှခုလိုလဲမသိ
design requirement မရှိ
verification မရှိ
```

## Good Prompt

```text
Role:
သင်က beginner-friendly frontend developer ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Context:
ဒီ course ရဲ့ playground ထဲမှာ static HTML/CSS/JS practice website တစ်ခုဆောက်ချင်ပါတယ်။

Goal:
Event workshop landing page တစ်ခုဆောက်ချင်ပါတယ်။

Scope:
- Static frontend only
- HTML, CSS, JavaScript ပဲသုံးပါ
- Backend API မထည့်ပါ
- Package install မလုပ်ပါ

Allowed folder:
- playground/event-landing-page/

Pages/Sections:
- Hero section
- Event details section
- Speaker section
- Registration CTA section

Acceptance Criteria:
- မြန်မာ UI text ပါရမယ်
- Mobile responsive ဖြစ်ရမယ်
- CTA button ပါရမယ်
- Browser console error မရှိရဘူး

Workflow:
1. File structure plan ပေးပါ
2. Approval ရမှ files create/edit လုပ်ပါ
3. Manual test steps ပေးပါ
4. git diff summary ပေးပါ

Do not edit before approval.
```

# 9. Example 2 — “Registration လုပ်ပေး”

## Bad Prompt

```text
Registration လုပ်ပေး။
```

## ပြဿနာ

```text
Frontend form လား?
Backend API လား?
Database လား?
Auth registration လား?
Event registration လား?
Email notification ပါလား?
Ticket code ပါလား?
```

## Good Prompt — Frontend-only

```text
Role:
သင်က careful frontend developer ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Context:
ဒီ app က static HTML/CSS/JS lab app ဖြစ်ပါတယ်။ Backend API မရှိပါ။

Goal:
Event registration form UI ကိုဆောက်ချင်ပါတယ်။

Scope:
Frontend UI + client-side validation only.

Non-goals:
- Backend API မထည့်ပါ
- Database မထည့်ပါ
- Email/SMS notification မထည့်ပါ
- Package install မလုပ်ပါ

Allowed files:
- playground/registration-form/index.html
- playground/registration-form/style.css
- playground/registration-form/app.js

Acceptance Criteria:
- Name input ပါရမယ်
- Phone input ပါရမယ်
- Email input ပါရမယ်
- Submit button ပါရမယ်
- Empty name/phone error မြန်မာလိုပြရမယ်
- Valid submit success message ပြရမယ်
- Mobile layout မပျက်ရဘူး

Verification:
- Manual browser tests
- Browser console error check
- git diff summary

Approval:
Code မပြင်ခင် plan ပေးပါ။
```

## Good Prompt — Backend API

```text
Role:
သင်က backend developer ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Context:
Event Management backend service ထဲမှာ registration API endpoint ထည့်ချင်ပါတယ်။

Goal:
POST /api/events/:id/register endpoint ကို implement လုပ်ရန်။

Scope:
Backend API only.

Non-goals:
- Frontend UI မပြင်ပါ
- Email/SMS notification မထည့်ပါ
- Payment မထည့်ပါ
- Admin dashboard မပြင်ပါ

Requirements:
- Request body: name, phone, email
- Server-side validation
- Duplicate registration prevention
- Ticket code generation
- Consistent error response format

Acceptance Criteria:
- Missing name returns validation error
- Missing phone returns validation error
- Valid request creates attendee
- Duplicate registration returns conflict
- Response includes ticketCode

Verification:
- API integration tests
- npm test
- git diff summary

Approval:
Implementation plan ပေးပြီး approval ရမှ code ပြင်ပါ။
```

ဒီ example က prompt ထဲမှာ scope ကိုရှင်းခြင်းက ဘယ်လောက်အရေးကြီးလဲပြတယ်။

# 10. Example 3 — “Error fix လုပ်”

## Bad Prompt

```text
Error fix လုပ်။
```

## ဘာကြောင့်မကောင်းလဲ

```text
Error ဘာလဲမသိ
ဘယ် command/run မှာ error ဖြစ်လဲမသိ
Expected/actual မရှိ
Relevant files မရှိ
Reproduce steps မရှိ
```

## Good Prompt

```text
Mode:
Debug first. Code မပြင်သေးပါနဲ့။

Context:
playground/chapter-11-coding-agent app ကို browser မှာဖွင့်ပြီး registration form ကိုစမ်းနေပါတယ်။

Bug:
Register button နှိပ်ပေမယ့် success message မပြပါ။

Steps to reproduce:
1. index.html ကို browser မှာဖွင့်ပါ
2. Name field ထဲ “Aung Aung” ရေးပါ
3. Phone field ထဲ “09123456789” ရေးပါ
4. Register button နှိပ်ပါ

Expected:
“Registration အောင်မြင်ပါသည်။” message ပြရမယ်။

Actual:
Message မပြဘဲ browser console မှာ error တက်ပါတယ်။

Error:
Cannot read properties of null (reading 'addEventListener')

Allowed files:
- playground/chapter-11-coding-agent/index.html
- playground/chapter-11-coding-agent/app.js

Do not touch:
- style.css
- package.json
- .env

Task:
1. Root cause candidates ပေးပါ
2. Minimal fix plan ပေးပါ
3. Approval ရမှ code ပြင်ပါ
4. ပြင်ပြီး manual test steps ပေးပါ
```

ဒီ prompt က Agent ကို debug workflow ထဲထည့်ထားတယ်။

# 11. Example 4 — “လိုတာ install လုပ်”

## Bad Prompt

```text
လိုတာ install လုပ်ပြီး feature ဆောက်ပေး။
```

## ဘာကြောင့်အန္တရာယ်ရှိလဲ

```text
Package မလိုဘဲ dependency ထည့်နိုင်
package.json/package-lock.json ပြောင်းနိုင်
security vulnerability ပါလာနိုင်
project size ကြီးနိုင်
version conflict ဖြစ်နိုင်
```

## Good Prompt

```text
Role:
သင်က careful coding agent ဖြစ်တယ်။

Mode:
Plan first. Code မပြင်သေးပါနဲ့။

Task:
Event date formatting ကို user-friendly ဖြစ်အောင်ပြချင်ပါတယ်။

Constraint:
Package install မလုပ်ပါနဲ့။ Built-in JavaScript Intl.DateTimeFormat ကိုအရင်စဉ်းစားပါ။

If you think a package is required:
- Package name ပြောပါ
- Why needed ရှင်းပါ
- Built-in alternative မရကြောင်းရှင်းပါ
- dependencies/devDependencies ဘယ်ထဲထည့်မလဲပြောပါ
- security/maintenance risk ပြောပါ
- approval တောင်းပါ

Approval မရသေးဘဲ package install မလုပ်ပါနဲ့။
```

Good prompt က package install ကို default မခွင့်ပြုဘဲ review gate ထည့်ထားတယ်။

```text
ဒီ app ကို production-ready ဖြစ်အောင်လုပ်ပါ။
```

# 13. Example 6 — “Commit လုပ်ပေး”

## Bad Prompt

```text
ပြီးရင် commit လုပ်ပေး။
```

## ဘာကြောင့်သတိထားရမလဲ

Agent က diff မဖတ်ဘဲ commit လုပ်သွားနိုင်တယ်။ Secret, node_modules, package changes ပါသွားနိုင်တယ်။

## Good Prompt

```text
Code မပြင်ပါနဲ့။
Commit မလုပ်ပါနဲ့။

git status နဲ့ git diff ကို review လုပ်ပါ။

စစ်ရန်:
- changed files list
- allowed files ထဲပါလား
- .env / secret ပါလား
- node_modules / dist ပါလား
- package.json/package-lock.json မလိုဘဲပြောင်းထားလား
- acceptance criteria pass ဖြစ်လား
- verification result ရှိလား

Output:
- Commit-ready ဖြစ်/မဖြစ်
- Required fixes
- Suggested commit messages ၃ ခု

Human approval မရသေးဘဲ git add/commit မလုပ်ပါနဲ့။
```

Commit ကို Agent အလိုအလျောက်မလုပ်စေဘဲ human review gate ထည့်ထားတယ်။

# 16. Prompt Quality Scorecard

Prompt တစ်ခုကို ၁၀ မှတ်နဲ့အမှတ်ပေးနိုင်တယ်။

| Criteria                             | Points |
| ------------------------------------ | -----: |
| Context clear                        |      1 |
| Goal specific                        |      1 |
| Scope defined                        |      1 |
| Non-goals included                   |      1 |
| Allowed files/actions included       |      1 |
| Do-not-touch / safety rules included |      1 |
| Acceptance criteria checkable        |      1 |
| Verification included                |      1 |
| Output format defined                |      1 |
| Approval gates included              |      1 |
| **Total**                            | **10** |

Score interpretation—

```text
0–3  = Dangerous / vague
4–6  = Usable but risky
7–8  = Good
9–10 = Strong, agent-ready
```

Example bad prompt—

```text
Registration history လုပ်ပေး။
```

Score—

```text
Context: 0
Goal: 0.5
Scope: 0
Non-goals: 0
Allowed files: 0
Safety: 0
Criteria: 0
Verification: 0
Output format: 0
Approval: 0

Total: 0.5/10
```

Good prompt က ၉/၁၀ ဖြစ်နိုင်တယ်။

# 17. Prompt Iteration Workflow

Prompt တစ်ခါတည်း perfect ဖြစ်စရာမလိုပါ။ Iterate လုပ်ပါ။

```text
Draft prompt
  ↓
Prompt review
  ↓
Improve prompt
  ↓
Agent plan
  ↓
Plan review
  ↓
Revise plan/prompt
  ↓
Approve implementation
```

Agent ကို prompt review ခိုင်းနိုင်တယ်။

```text
ဒီ prompt ကို Agentic Coding prompt အနေနဲ့ review လုပ်ပါ။
ဘာ missing ဖြစ်နေလဲရှာပါ။
ပိုကောင်းတဲ့ version ရေးပါ။
Code မရေးပါနဲ့။
```

Prompt ကို Agent ကိုပေးခိုင်းတာမဟုတ်ဘဲ Agent ကို **prompt coach** အဖြစ်သုံးတာပါ။

# 23. Common Errors

## Error 1 — Prompt ကိုအရှည်ရေးတာနဲ့ကောင်းပြီလို့ထင်ခြင်း

Prompt အရှည်ကြီးဖြစ်တာက good prompt မဟုတ်ပါ။ Good prompt ဆိုတာ **ရှင်းပြီး စစ်လို့ရတဲ့ prompt** ပါ။

မကောင်း—

```text
စာအများကြီးရေးထားပေမယ့် goal/scope/criteria မရှိ
```

ကောင်း—

```text
တိုသော်လည်း goal, scope, verification ရှင်း
```

---

## Error 2 — “Do not” list မထည့်ခြင်း

Agent ကမလုပ်ရမယ့်အရာမသိရင် scope ကျော်နိုင်တယ်။

ပြင်—

```text
Do not:
- package install
- backend API
- database
- .env
- deployment
```

---

## Error 3 — Verification မပါဘဲ feature ခိုင်းခြင်း

Agent က code ပြီးရင် “done” လို့ပြောနိုင်တယ်။ Verification မပါရင် user မစစ်နိုင်ဘူး။

ပြင်—

```text
Verification:
- npm run build
- manual test cases
- git diff summary
```

---

## Error 4 — Package install ကို wide open ပေးခြင်း

မကောင်း—

```text
လိုတာ install လုပ်ပါ။
```

ကောင်း—

```text
Package install မလုပ်ပါနဲ့။
လိုအပ်သည်ထင်ရင် reason/risk/alternative ပေးပြီး approval တောင်းပါ။
```

---

## Error 5 — Commit / Push / Deploy ကို auto ခိုင်းခြင်း

မကောင်း—

```text
ပြီးရင် commit/push/deploy လုပ်ပေး။
```

ကောင်း—

```text
Commit message suggestion ပေးပါ။
Human approval မရသေးဘဲ commit/push/deploy မလုပ်ပါနဲ့။
```

---

## Error 6 — Error prompt မှာ error details မပါခြင်း

မကောင်း—

```text
မရဘူး။
```

ကောင်း—

```text
Expected, Actual, Steps, Error, Relevant code ပါ။
```

---

## Error 7 — Prompt ထဲမှာ secret paste လုပ်ခြင်း

မလုပ်ပါနဲ့။

ပြင်—

```text
Secret value မပေးပါ။
Environment variable name ကိုသာပြောပါ။
```

---

# 24. Debug / Recovery

## 24.1 Agent က prompt မကောင်းလို့လမ်းလွဲသွားရင်

Prompt—

```text
Stop. Code မပြင်သေးပါနဲ့။

ငါ့ prompt မရှင်းလင်းခဲ့လို့ scope ကျော်သွားနိုင်ပါတယ်။

လုပ်ရန်:
1. မင်းဘာလုပ်ဖို့နားလည်ခဲ့လဲ summarize လုပ်ပါ
2. Actual changes summary ပေးပါ
3. Scope ကျော်တဲ့ changes ရှိလားဖော်ပြပါ
4. git diff review ပေးပါ
5. Prompt ကိုပိုကောင်းအောင်ပြန်ရေးပေးပါ
6. Rollback options ပေးပါ
```

---

## 24.2 Agent က package install လုပ်ချင်လာရင်

Prompt—

```text
Package install မလုပ်သေးပါနဲ့။

အရင် review ပေးပါ:
- Package name
- Why needed
- Built-in alternative
- dependencies/devDependencies
- Security/maintenance risk
- Files changed
- Rollback plan

Approval မရသေးဘဲ install မလုပ်ပါနဲ့။
```

---

## 24.3 Agent က commit/deploy လုပ်ချင်လာရင်

Prompt—

```text
Commit/deploy မလုပ်ပါနဲ့။

အရင်:
- git status
- git diff summary
- verification result
- risks
- suggested commit message
ပေးပါ။

Human approval မရသေးဘဲ commit/push/deploy မလုပ်ပါနဲ့။
```

---

## 24.4 Agent က “done” ပြောပေမယ့် verification မရှိရင်

Prompt—

```text
“Done” လို့မသတ်မှတ်သေးပါ။

Code မပြင်ပါနဲ့။
Verification review လုပ်ပါ။

စစ်ရန်:
- Acceptance criteria တစ်ခုချင်းစီ
- Manual tests
- Build/test command, if available
- Browser console
- git diff summary
- known risks

PASS/FAIL table ပေးပါ။
```

---

## 24.5 Prompt ကအရမ်းရှုပ်သွားရင်

Prompt—

```text
ဒီ prompt ကရှုပ်နေပါတယ်။
Code မရေးပါနဲ့။

လုပ်ရန်:
- Prompt ကို concise version အဖြစ်ပြန်ရေးပါ
- Must-have constraints ကိုသာထည့်ပါ
- Optional items ကိုခွဲပါ
- Output format ကိုရှင်းပါ
```

# 27. Chapter Summary

ဒီအခန်းမှာ **Good vs Bad Prompts** ကိုလေ့လာခဲ့တယ်။

အရေးကြီးဆုံး mental model—

```text
Bad prompt = Agent ကိုမရှင်းတဲ့အလုပ်ပေးခြင်း
Good prompt = Agent ကို scope, criteria, verification, safety boundaries ပါတဲ့ task specification ပေးခြင်း
```

Bad prompt smells—

```text
vague goal
too broad
no scope
no non-goals
no allowed files
no verification
no approval gate
install anything
commit/push/deploy automatically
secret safety missing
```

Good prompt မှာပါသင့်တာ—

```text
Role
Mode
Context
Goal
Scope
Non-goals
Allowed files
Do-not-touch list
Constraints
Acceptance Criteria
Workflow
Verification
Output Format
Approval Gates
```

Agentic Coding အတွက် takeaway—

```text
Prompt ကောင်းတာက Agent ကိုအလုပ်လုပ်စေတဲ့ instruction တင်မက,
project ကိုကာကွယ်တဲ့ safety boundary လည်းဖြစ်တယ်။
```
