# Chapter 6 — Git, GitHub, and .gitignore

## 1. Learning Objectives

ဒီအခန်းပြီးသွားရင် သင်က အောက်ပါအရာတွေကို နားလည်ပြီး လက်တွေ့လုပ်နိုင်ရပါမယ်။

## 2. Beginner Explanation

Git ကို အလွယ်ဆုံးပြောရရင်—

```text
Git = Project အတွက် Save Game System
```

Game ကစားတဲ့အခါ level တစ်ခုအောင်တိုင်း save လုပ်တယ်။ နောက်ပိုင်းမှားသွားရင် save point ကိုပြန်သွားနိုင်တယ်။ Git လည်းအဲဒီလိုပါပဲ။

```text
Project setup ပြီးပြီ       → commit
Product list feature ပြီးပြီ → commit
Cart feature ပြီးပြီ        → commit
Checkout feature ပြီးပြီ    → commit
```

Git official book မှာလည်း project က မှတ်တမ်းတင်ချင်တဲ့ state တစ်ခုရောက်တိုင်း changes တွေကို snapshot အနေနဲ့ commit လုပ်သွားတာကို Git basic workflow အဖြစ်ရှင်းပြထားပါတယ်။ ([Git SCM][1])

Agentic Coding မှာ Git ကို ဒီလိုသုံးမယ်။

```text
Feature မစခင် → git status စစ်
Feature ပြီးရင် → git diff စစ်
အဆင်ပြေရင် → git add + git commit
Agent မှားပြင်ရင် → Git နဲ့ rollback
```

အရေးကြီးဆုံး mental model—

```text
Git = local save history
GitHub = online backup + collaboration place
```

Git က ကိုယ့် computer ထဲမှာ project history သိမ်းတယ်။ GitHub က အဲဒီ project ကို online ပေါ်မှာသိမ်းပြီး share/collaborate လုပ်နိုင်တဲ့ platform ပါ။

# 3. Git ဆိုတာဘာလဲ

Git ဆိုတာ version control system ပါ။ Version control ဆိုတာ project file တွေရဲ့ပြောင်းလဲမှု history ကိုစီမံပေးတဲ့စနစ်ပါ။

Git က သင့် project ထဲမှာ—

```text
ဘယ် file ပြင်ခဲ့လဲ
ဘာ lines ပြောင်းခဲ့လဲ
ဘယ်အချိန် commit လုပ်ခဲ့လဲ
ဘယ်သူ commit လုပ်ခဲ့လဲ
ဘယ် commit ကိုပြန်သွားနိုင်လဲ
```

တွေကိုမှတ်ထားနိုင်တယ်။

ဥပမာ—

```text
Commit 1: initial course folder
Commit 2: add course decisions
Commit 3: add chapter 1 notes
Commit 4: add chapter 2 terminal lab
Commit 5: add JSON/npm practice project
```

ဒီလို commit history ရှိရင် project ပျက်သွားလည်း “Commit 4 အထိပြန်သွားမယ်” လို့ဆုံးဖြတ်နိုင်တယ်။


GitHub docs မှာ repository တွေက project အမျိုးမျိုးကိုသိမ်းနိုင်ပြီး GitHub မှာ repo create လုပ်၊ first change commit လုပ်နိုင်တဲ့ workflow ကို quickstart အဖြစ်ဖော်ပြထားပါတယ်။ GitHub Hello World guide မှာလည်း repository, branch, commit, pull request workflow ကို GitHub essentials အဖြစ်သင်ပေးထားပါတယ်။ ([GitHub Docs][2]) ([GitHub Docs][3])


# 6. Git Workflow အခြေခံ

Git basic workflow ကို ဒီလိုမှတ်ပါ။

```text
Working Directory
   ↓ git add
Staging Area
   ↓ git commit
Repository History
```

မြန်မာလို—

```text
Working Directory = လက်ရှိပြင်နေတဲ့ files
Staging Area      = နောက် commit ထဲထည့်ဖို့ရွေးထားတဲ့ changes
Repository History = commit လုပ်ပြီးသိမ်းထားတဲ့ history
```

Git official `git add` docs မှာ `git add` က new/changed files တွေကို index, staging area ထဲထည့်တာဖြစ်ပြီး `git commit` က staged changes တွေကို commit လုပ်တာဖြစ်ကြောင်းရှင်းပြထားပါတယ်။ `git status` က staged changes summary ကိုကြည့်ဖို့သုံးနိုင်တယ်လို့လည်း docs မှာဖော်ပြထားပါတယ်။ ([Git SCM][4])

အလွယ်ဆုံး flow—

```bash
git status
git add .
git commit -m "add chapter 6 git basics"
```

ဒါက—

```text
git status → ဘာတွေပြောင်းထားလဲကြည့်
git add .  → ပြောင်းထားတာတွေကို staging area ထဲထည့်
git commit → save point တစ်ခုလုပ်
```

ဆိုတဲ့အဓိပ္ပါယ်ပါ။

ဥပမာ—


## 11.3 Already tracked file ကို .gitignore ထဲထည့်လည်း မရပ်နိုင်ခြင်း

အရေးကြီးတဲ့ rule—

```text
Git က track လုပ်ပြီးသား file ကို .gitignore ထဲထည့်ရုံနဲ့ ignore မလုပ်တော့ပါဘူး။
```

GitHub docs မှာ already checked-in file ကို ignore လုပ်ချင်ရင် `.gitignore` rule ထည့်ပြီးနောက် `git rm --cached FILENAME` နဲ့ untrack လုပ်ရန်ဖော်ပြထားပါတယ်။ ([GitHub Docs][6])

ဥပမာ `.env` ကိုမတော်တဆ Git track လုပ်မိပြီးသားဆိုရင်—

```bash
git rm --cached .env
```

ပြီးရင်—

```bash
git status
git commit -m "stop tracking env file"
```

သတိ—

```text
Secret တကယ် commit history ထဲဝင်သွားပြီဆိုရင် .gitignore ထည့်တာနဲ့မလုံလောက်ပါ။
Secret ကို revoke/rotate လုပ်ရမယ်။
```

# 12. git add — Staging Area ထဲထည့်ခြင်း

All changes stage လုပ်ရန်—


```bash

```

```bash

```

Agentic Coding beginner rule—

```text
git add . မလုပ်ခင် git status + git diff ကြည့်ပါ။
```

ဘာကြောင့်လဲဆိုရင် `git add .` က current folder ထဲက changes အားလုံး stage လုပ်နိုင်တယ်။ Agent က မလိုအပ်တဲ့ file ပြင်ထားရင် အဲဒါပါ commit ထဲပါသွားနိုင်တယ်။

# 13. git diff — ဘာပြင်ထားလဲကြည့်ခြင်း

Agentic Coding မှာ အလွန်အရေးကြီးဆုံး command တစ်ခုက—

```bash
git diff
```

ပါ။

```bash

git diff README.md
```

အဓိပ္ပါယ်—

```text

git diff --staged → stage လုပ်ပြီးသား changes ကြည့်
git diff file     → file တစ်ခုရဲ့ changes ကြည့်
```


# 25. Git Recovery Basics

Agent မှားပြင်သွားရင် Git နဲ့ပြန်ဆယ်နိုင်တယ်။


ဥပမာ—

```bash

```

## 25.2 Stage လုပ်ထားတာကို unstage လုပ်ခြင်း

```bash
git restore --staged path/to/file
```

ဥပမာ—

```bash
git restore --staged package.json
```

## 25.3 All uncommitted changes ဖျက်ခြင်း

အန္တရာယ်ရှိတယ်။ Beginner အနေနဲ့ approval မရှိဘဲမသုံးပါနဲ့။

```bash
git restore .
```

## 25.4 Last commit ကိုပြန်ကြည့်ခြင်း

```bash
git log --oneline
```

## 25.5 Commit တစ်ခုရဲ့ details ကြည့်ခြင်း

```bash
git show COMMIT_ID
```

## 25.6 Safe recovery rule

```text
git restore / git reset / git clean command မ run ခင်:
[ ] git status စစ်
[ ] git diff စစ်
[ ] ဘာဆုံးရှုံးမလဲနားလည်
[ ] Agent ကို recovery plan တောင်း

```

# 26. Dangerous Git Commands

အောက်ပါ command တွေကို မျက်စိမှိတ်မ run ပါနဲ့။

```bash
git reset --hard
git clean -fd
git clean -fdx
git push --force
git rm -r
```

အဓိပ္ပါယ်—

```text
git reset --hard → uncommitted changes ဖျက်နိုင်
git clean -fd    → untracked files/folders ဖျက်နိုင်
git clean -fdx   → ignored files ပါဖျက်နိုင်
git push --force → remote history ကို override လုပ်နိုင်
git rm -r        → files/folders ကို Git + working tree ထဲကဖျက်နိုင်
```

Agent prompt—

```text
ဒီ Git command က destructive ဖြစ်နိုင်လားရှင်းပြပါ။
ဘာ data ဆုံးရှုံးနိုင်လဲ?
Safe alternative ရှိလား?
Dry-run ရှိလား?
Run မလုပ်ခင် backup/commit လိုလား?
```

# 32. Common Errors

```bash
pwd

```


```text
Agent ပြင်ထားတဲ့ မလိုအပ်တဲ့ files ပါ commit ထဲဝင်နိုင်တယ်။
```

## Error 3 — .env ကို commit မိခြင်း

အလွန်အရေးကြီးတဲ့ security error ပါ။

ဖြစ်ရင်—

```text
1. .gitignore ထဲ .env ထည့်
2. git rm --cached .env
3. commit
4. Secret ကို revoke/rotate
```

Secret တစ်ခါ commit history ထဲဝင်သွားရင် “file ဖျက်လိုက်ပြီ” နဲ့မလုံလောက်နိုင်ဘူး။

## Error 4 — node_modules ကို commit မိခြင်း

`node_modules` ကအလွန်ကြီးတတ်ပြီး dependency install output ဖြစ်လို့ Git ထဲမထည့်သင့်ပါ။

ဖြေရှင်းရန်—

```text
.gitignore ထဲ node_modules/ ထည့်
already tracked ဖြစ်နေပါက git rm --cached -r node_modules
commit လုပ်
```

သတိ—

```text
git rm --cached -r node_modules က Git tracking ထဲကဖယ်တာဖြစ်ပြီး local file system ထဲမှာကျန်နိုင်တယ်။
သို့သော် command run မလုပ်ခင် Agent ကိုရှင်းပြခိုင်းပါ။
```

Agentic Coding အတွက် takeaway—

```text
Agent ကို code ပြင်ခိုင်းမယ့်အခါ Git checkpoint မရှိဘဲမစပါနဲ့။
Feature မစခင် git status စစ်ပါ။
Feature ပြီးရင် git diff review လုပ်ပါ။
အဆင်ပြေမှ commit လုပ်ပါ။
```

မကောင်းတဲ့ prompt—

```text
ဒီ feature လုပ်ပြီး commit လုပ်ပေး။
```

ကောင်းတဲ့ prompt—

```text
ဒီ feature မစခင် git status စစ်ပါ။
Working tree clean မဟုတ်ရင် code မပြင်ပါနဲ့။
ပြင်ပြီးရင် git diff summary, verification result, commit message suggestion ပေးပါ။
ကျွန်တော် approve လုပ်မှ commit လုပ်ပါ။
```
